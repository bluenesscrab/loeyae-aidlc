# AIDLC 项目规范

> **定位**：本文件是 loeyae-aidlc 仓库自身的设计与治理准则。
> **约束力**：新增或修改流程规则、平台入口、Skill、Agent、Hook、MCP 配置、安装说明或发布清单时，必须符合以下所有条款。
> **优先级**：高于任何单个 steering、skill 或平台入口文件的内部惯例。发生冲突时以本文件为准。

---

## 第一部分：项目定位与职责边界

### 项目定位

本仓库是 Loeyae AI-DLC 的统一源，同时支持 Kiro Power、Claude Code 插件和 OpenCode 插件。三个平台共享 `steering/` 中的流程规则和 `skills/` 中的平台无关薄入口。

项目结构、支持平台和入口总览以 `README.md` 为准。本文件只定义稳定的仓库治理约束，不重复安装命令、版本号、完整工具清单或平台实现细节。

### 事实来源

| 信息类型 | 事实来源 |
|---------|---------|
| 项目定位、支持平台、入口总览 | `README.md` |
| 共享流程规则 | `steering/` |
| 主工作流路由 | `steering/core-workflow.md` |
| 平台无关 Skill 入口 | `skills/` |
| Kiro 接入与能力声明 | `POWER.md`、`mcp.json` |
| Claude Code 接入与能力声明 | `CLAUDE.md`、`.claude-plugin/` |
| OpenCode 接入、安装与发布配置 | `.opencode/`、`package.json` |
| 平台无关子 Agent 指令 | `agents/` |
| 平台 Hook 模板 | `hooks/` 中对应平台目录 |

一个事实只在其事实来源中完整定义。其他文件只能保留当前职责所需的简述和引用，不得复制流程细节、安装步骤、版本号或完整能力矩阵。

## 第二部分：AIDLC 流程目标（流程必须达到的效果）

| # | 原则 | 含义 | 违反示例 |
|---|------|------|----------|
| 1 | **先思考后编码** | 不假设、不隐藏困惑，呈现权衡取舍 | 跳过 Inception 直接写代码；遇到模糊需求不提问直接猜测 |
| 2 | **简洁优先** | 最少代码解决问题，不添加投机性内容 | 添加“以后可能用到”的抽象层；引入需求未要求的功能 |
| 3 | **精准修改** | 只改必须的，只清理自己造成的混乱 | 顺手重构不相关代码；修复 bug 时改动命名风格 |
| 4 | **目标驱动执行** | 定义成功标准，循环直到验证通过 | 代码生成后不做审查就标记完成；测试失败但继续下一步 |

## 第三部分：AIDLC 自身设计规范（流程本身必须符合的特征）

| # | 规范 | 含义 | 检查方法 |
|---|------|------|----------|
| 1 | **渐进式披露** | 每个流程文件只包含自身职责范围内的核心逻辑；涉及其他文件已定义的规则时用引用或加载指令代替内联重复；禁止大段复制另一文件的内容 | 一个文件中出现另一文件已定义规则的实质性重复（≥3 行相同语义）→ 违反 |
| 2 | **简洁高效** | 每个文件只做一件事，无冗余重复；token 消耗最小化 | 同一规则是否在多个文件中重复定义 → 违反 |
| 3 | **防幻觉** | 不知道就问；禁止用假设代替澄清；所有决策必须有依据 | 流程中是否有“默认假设 X”而非“询问用户 X”→ 违反 |
| 4 | **上下文即代码** | 所有规则存于可版本控制的 `.md` 文件中；无口头约定 | 是否有只存在于对话记忆中的规则 → 违反 |
| 5 | **自适应流程** | 简单任务跳过阶段；流程复杂度匹配任务复杂度 | 是否有“无论什么情况都必须走完全流程”的硬编码 → 违反 |
| 6 | **会话延续** | 通过 `state.md` 等文件维持状态，支持跨会话恢复 | 新增流程分支是否能通过 `state.md` 恢复 → 未实现则违反 |

## 第四部分：多平台一致性

修改共享流程或平台入口时，必须遵循：

1. **共享优先**：可跨平台复用的流程规则只在 `steering/` 中维护，平台入口只负责加载、路由和适配。
2. **影响识别**：修改前识别受影响的平台和事实来源；修改后检查相关入口、清单与说明是否仍然一致。
3. **平台隔离**：平台专属能力和实现只写入对应平台入口，不得污染共享流程。
4. **能力有据**：不得声明未经仓库实现或实际验证的平台能力；无法验证时必须明确标注并询问用户。
5. **精准同步**：仅同步实际受影响的平台；单平台修改不得无依据扩散到其他平台。
6. **避免漂移**：入口路径、安装方式、MCP、Agent、Hook 或发布配置变化时，必须核对 `README.md` 和对应平台文档。

## 检查清单（每次 PR / 修改前自查）

按本次修改范围逐条确认；不适用项应明确跳过原因：

### 仓库治理

- [ ] **职责边界**：修改是否落在该类信息的事实来源中？
- [ ] **平台影响**：是否识别本次修改影响 Kiro、Claude Code、OpenCode 中的哪些入口？
- [ ] **共享优先**：可共享的流程规则是否只维护在 `steering/`？
- [ ] **平台隔离**：平台专属实现是否只位于对应平台入口？
- [ ] **描述一致**：项目总览、入口路径、安装文档和发布清单是否仍然一致？
- [ ] **能力有据**：新增或修改的平台能力声明是否有实现或验证依据？

### 流程设计

- [ ] **渐进式披露**：`core-workflow.md` 中新增内容是否只有路由表和不超过 5 行的简述？详情是否在独立文件中？
- [ ] **简洁高效**：新增内容是否与已有文件重复？是否能合并到现有文件？
- [ ] **防幻觉**：流程中遇到不确定情况是否有明确的“询问用户”指令？
- [ ] **上下文即代码**：所有新规则是否都落入可版本控制的 `.md` 文件？
- [ ] **自适应流程**：新增步骤是否有明确的执行条件和跳过条件？
- [ ] **会话延续**：新增流程分支的状态是否能写入 `state.md` 并支持恢复？
- [ ] **先思考后编码**：流程是否保证在编码前完成必要的思考和确认？
- [ ] **简洁优先**：新增规则是否是最小必要的？是否添加了投机性内容？
- [ ] **精准修改**：本次修改是否只涉及需要改动的文件？是否有不必要的连带改动？
- [ ] **目标驱动执行**：新增流程步骤是否有明确的完成标准和验证方式？

## 术语

| 术语 | 定义 |
|------|------|
| 事实来源 | 某类信息允许被完整定义和维护的权威位置 |
| 平台入口 | Kiro、Claude Code 或 OpenCode 用于加载本项目能力的专属文件与目录 |
| 共享规则 | 三个平台共同使用、仅在 `steering/` 中维护的流程定义 |
| 路由表 | `core-workflow.md` 中的步骤表格，只含步骤名、条件、审批级别、加载文件 |
| 步骤详情 | 独立 steering 文件中的具体执行步骤、模板、示例 |
| 投机性内容 | 当前需求未要求但“可能以后有用”的代码或规则 |
| 渐进式披露 | 用户或 AI 只在需要时加载详细内容，不预加载全部 |
