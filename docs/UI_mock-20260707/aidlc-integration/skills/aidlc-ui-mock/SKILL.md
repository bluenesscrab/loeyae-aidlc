# UI Mock 标准体系

**开始时宣布：** "加载 aidlc-ui-mock 执行 UI Mock 阶段"

**目的**：在 Inception 阶段生成标准化、高质量的 UI Mock 产出物。

> **薄入口说明**：本 skill 是 UI Mock 的规则汇总入口，供无 Hook 机制的平台（如 OpenCode）使用。Kiro 和 Claude Code 平台通过 steering + hooks 实现等效保障，无需依赖本 skill。

---

## 核心原则

### 三层分离

| 层次 | 包含什么 | 面向谁 |
|------|----------|--------|
| 第一层：UI 内容本体 | 列标题、按钮文案、页面结构 | 最终用户 |
| 第二层：条件表内容 | 状态判断条件、按钮组合 | 开发人员 |
| 第三层：底部说明 | 推导说明 + 业务规则 | 需求验证 |

### 两阶段工作流

1. **阶段1：页面编排 + 骨架** → 产出 page-specs.md + 主文件骨架 + 子页面骨架 → 用户审流程
2. **阶段2：内容填充** → 逐页填充子页面 UI 内容 + 主文件条件表/说明 → 用户审效果

### iframe 拆分架构

- 每个页面为独立 HTML 文件（子页面）
- 主文件通过 iframe 引入子页面
- 子页面只包含 UI 本体，不包含条件表/说明

## 强制规则（替代 Hook 的软约束）

### 规则1：逐页确认（替代 one-page Hook）

在阶段2填充时：
1. **一次只写一个子页面** HTML 文件的完整内容
2. 写完后**必须停下来**向用户展示结果
3. 等待用户明确确认后才继续下一个页面
4. **禁止**在一个回合中连续写入多个子页面

**例外**：骨架文件（<50行）、page-specs.md、主文件条件表更新不受此限制。

### 规则2：分层推导自检（替代 principles-check Hook）

每次创建或编辑子页面 HTML 后，**必须立即执行以下自检**：

**第一层自检（面向用户）**：
- [ ] 页面头部声明了用户角色和核心任务
- [ ] 列标题/按钮文案无系统内部术语泄漏
- [ ] 条件表 summary 和表头使用用户语言
- [ ] 列表信息不完整时所有状态都有 [查看]
- [ ] mock-box 按用户任务流组织

**第二层自检（面向开发）**：
- [ ] 条件表单元格包含精确状态判断条件
- [ ] 操作列在每种状态下列出完整按钮组合
- [ ] 状态列标题标注了系统枚举值
- [ ] 没有状态差异遗漏在条件表之外

**第三层自检（面向验证）**：
- [ ] 底部说明包含「推导说明」区块
- [ ] 底部说明包含「业务规则」区块
- [ ] 业务规则覆盖计算/校验/边界/连锁影响

### 规则3：需求禁止反转

AI 不能推翻产品已明确回答的决定。可以在空白地带合理细化，但绝不能反转已确认的需求。

## 设计规范速查

### 布局选择

- 无状态差异 → 布局A（APP: 420px 固定宽 / PC: 半宽并排）
- ≥2种状态 → 布局B（全宽左右并排，左 iframe + 右条件表）

### 子页面规范

- PC 端：`width: 1920px; min-height: 1080px`
- APP 端：`width: 375px; min-height: 812px`
- 样式全部内联，不依赖外部 CSS
- 只画 UI 本体，不包含 mock-box-title/条件表/推导说明

### 局部改动页面

1. 先读取现有代码，完整还原现有页面
2. 用橙色边框 + "新增" 标签标注增量改动
3. 去掉标记后应与现有系统高度一致
