# 需求细化禁止反转规则

## 核心规则

AI 可以在产品没有明确表态的「空白地带」进行合理细化，**但绝对不能推翻产品已经明确回答的决定**。

## 判定标准

以下来源的内容视为「产品已明确回答」：
- 需求评审问答中的 `[回答]:` 内容
- Won't Have / 本期不做 列表中的条目
- 需求澄清文件中用户填写的选项和说明
- UI Mock 交叉审查中产品确认的交互方案

## 禁止行为

- ❌ 产品回答"不限购"，AI 在需求文档中写"每笔订单只允许购买1张"
- ❌ 产品基准 UI Mock 中包含某组件，AI 在需求文档中写"不支持该功能"
- ❌ 产品回答"本期不做X"，AI 将 X 作为 Must Have 写入
- ❌ 将上述反转标记为"合理细化"或"合理推导"

## 允许行为

- ✅ 产品没有提到某个边界条件（空白地带），AI 补充明确的处理规则
- ✅ 产品给出方向性回答但未说明具体实现，AI 细化为具体实现方案
- ✅ 产品的两个回答之间存在逻辑矛盾，AI 标记为矛盾并提问澄清（而非自行选择一方）

## 审核阶段强制检查

在基准审核中，对每个标记为"合理细化"的差异项，**必须执行反向验证**：

1. 在产品评审问答中搜索：是否有明确否定这条规则的回答？
2. 在 Won't Have 列表中搜索：是否有把这个功能列为"不做"的条目？
3. 在 UI Mock/交叉审查记录中搜索：是否有与此规则相反的设计决定？

如果找到任何一条相反证据 → 该差异项**不是**"合理细化"，而是**非法反转**，必须移除。
