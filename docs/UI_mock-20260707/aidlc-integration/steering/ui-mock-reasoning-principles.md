# UI Mock 推导原则（分层推导版）

本文件规定 AI 在生成 UI Mock 时必须遵循的**内容决策原则**。UI Mock 中不同层次的内容面向不同受众，需用不同的推导思路。

---

## 前置规则：局部改动必须还原现有页面（优先级最高）

**适用条件**：当 mock-box 标记为「局部改动」类型时，以下规则强制执行。

### 强制流程

1. **读取现有代码**：找到改造基础对应的代码文件，读取其完整结构
2. **还原现有页面**：在 mock-box 中**先画出现有页面的完整布局**
3. **标注增量改动**：用橙色边框 + "新增" 标签标注本次增量修改的部分
4. **非增量部分保持原样**：已有的元素必须保留、位置不变、样式不改

### 判定标准

去掉所有橙色「新增」标记后的内容，应该和现有系统的页面**高度一致**。

---

## 核心架构：三层分离

| 层次 | 包含什么 | 面向谁 | 推导方向 |
|------|----------|--------|----------|
| **第一层：UI 内容本体** | 列标题、按钮文案、条件表 summary/表头、页面结构 | 最终用户 | 用户正向推导 |
| **第二层：条件表内容** | 条件表单元格中的状态描述和差异说明 | 开发人员 + 需求验证 | 表头用用户语言，单元格用系统精确描述 |
| **第三层：底部说明** | mock-box-desc 区域 | 需求验证 + 开发人员 | 推导说明 + 业务规则补充 |

**禁止混淆层次**：不要在 UI 本体中使用系统术语，也不要在条件表单元格中使用模糊的自然语言。

---

## 第一层：UI 内容本体（面向用户）

### 推导步骤

1. **确定用户角色**：这个页面的用户是谁？
2. **确定核心任务**：用户打开这个页面想完成什么？
3. **确定信息需求**：用户需要看到什么才能完成任务/做出判断？
4. **确定操作需求**：用户在什么情境下需要执行什么动作？
5. **自检**：这个元素去掉后，用户的任务是否受阻？如果不受阻，它不该存在。

### 规则 1.1：页面头部必须声明用户和任务

```html
<!-- ✅ 正确 -->
<p>用户：平台运营人员 | 核心任务：创建和管理规则、监控订单情况</p>

<!-- ❌ 错误 -->
<p>涉及页面：规则列表、规则新增/编辑弹窗...共 6 个 mock-box</p>
```

### 规则 1.2：列标题只表达用户认知

去掉某个标注/后缀后，用户仍能正确理解和操作 → 它是内部逻辑泄漏，应去掉。

### 规则 1.3：条件表的 summary 和表头用用户语言

```html
<!-- ✅ 用户视角 -->
<summary>📋 操作可用性（取决于规则是否已有商家在用）</summary>
<th>我想做的事</th><th>规则没人用过</th><th>规则已有商家在用</th>

<!-- ❌ 系统视角 -->
<summary>📋 状态条件表（基准状态：启用+未使用）</summary>
<th>区域/组件</th><th>启用 + 未使用（基准）</th><th>启用 + 使用中</th>
```

### 规则 1.4：列表信息不完整时，所有状态都需要 [查看]

列表展示字段 < 实体完整字段 → 所有状态都需要 [查看]。

### 规则 1.5：mock-box 按用户任务流组织

以用户故事为驱动单元组织 mock-box，而非按系统功能模块罗列。

### 规则 1.6：UI 元素最小化原则

- 「约束存在」≠「约束需要被可视化」
- 「不可操作」最简表达 = 不出现操作控件
- 「不可选」最简表达 = 不出现在可选列表中
- 选择器/添加弹窗中，已选/已添加的项不出现在待选列表中
- 校验反馈默认位置 = 提交时
- 每多一个 UI 元素 = 多一份认知负担

## 第二层：条件表内容（面向开发 + 验证）

### 规则 2.1：单元格内容必须包含系统精确描述

```html
<!-- ✅ 正确 -->
<td>[查看] [停用]<br><span>编辑和删除 disabled（条件：存在关联商品 OR 已售订单且 status ≠ cancelled）</span></td>

<!-- ❌ 不够精确 -->
<td>❌ 不可以（避免影响已售出的联票）</td>
```

### 规则 2.2：状态列标题用用户语言 + 括号标注系统枚举

```html
<th>没人用过（inUse=false）</th>
<th>已有商家在用（inUse=true）</th>
```

### 规则 2.3：多状态页面必须生成条件表

同一 UI 位置 × ≥2 种状态导致 UI 差异 → 必须生成条件表。没有例外。

---

## 第三层：底部说明（面向验证 + 开发）

### 区块 A：推导说明

解释 UI 设计决策的推导逻辑。必须包含：列/字段的选择逻辑、关键设计决策、用户心智模型假设。

### 区块 B：业务规则补充

补充 mock 中无法可视化展示的业务规则。必须包含：具体计算规则、边界判断条件、校验逻辑和错误处理、状态变更后的连锁影响。

---

## 自检清单

### 第一层自检
- [ ] 页面头部声明了用户角色和核心任务
- [ ] 所有列标题/按钮文案无系统内部术语泄漏
- [ ] 条件表 summary 和表头使用用户语言
- [ ] 列表信息不完整时，所有状态都有 [查看]
- [ ] mock-box 按用户任务流组织
- [ ] 没有自行添加需求未要求的 UI 元素

### 第二层自检
- [ ] 条件表单元格包含精确的状态判断条件
- [ ] 条件表覆盖了操作列在每种状态下的完整按钮组合
- [ ] 状态列标题标注了系统枚举值
- [ ] 没有状态差异被遗漏在条件表之外

### 第三层自检
- [ ] 底部说明包含「推导说明」区块
- [ ] 底部说明包含「业务规则」区块
- [ ] 业务规则覆盖了具体计算/校验/边界/连锁影响
- [ ] 推导说明解释了关键设计决策的"为什么"

---

## 完整性自检（生成后必须执行）

### 检查1：操作路径闭环
每一个可点击的按钮/入口，追问：点击后展示什么？操作完成后结果在哪里看？操作失败时展示什么？

### 检查2：确认类操作的信息量
不可逆操作的确认弹窗，信息量必须足以让用户不翻看其他页面就能做出判断。

### 检查3：表单联动完整性
所有 radio/select/switch 组件，每一个选项对应的子组件/字段变化都必须画出。
