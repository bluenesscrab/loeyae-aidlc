# Spring Cloud + Nacos 存量商城定制开发外部工程保障指南

**版本**：1.0.0  
**日期**：2026-07-16  
**适用范围**：15 个以上 Spring Boot/Spring Cloud/Nacos 微服务组成的存量云商城定制开发，覆盖需求进入开发、编码、测试、制品生成和实际部署验证，不覆盖部署后的生产运维。

## 1. 文档定位

本文定义 AI-DLC 流程之外必须由项目工程体系提供的保障。AI-DLC 负责需求、设计、计划、实现、审查、验证编排和证据记录；本文所述平台与机制负责提供可查询事实、并发协调、机器校验、环境隔离、制品管理和部署执行能力。

本文不是 AI-DLC steering，不定义 AI-DLC 内部阶段路由，也不代表当前 AI-DLC 已经实现以下平台能力。

## 2. 目标与非目标

### 2.1 目标

1. 确保定制开发建立在已验证的存量系统事实之上。
2. 确保跨服务接口、事件和配置变更可识别、可兼容、可回滚。
3. 确保订单、库存、支付、履约、ERP 等链路的一致性和补偿方案可验证。
4. 确保搜索、推荐、直播、ERP 等外部系统故障不会无控制地扩散。
5. 确保多人并行开发时状态、契约和认领信息不会相互覆盖。
6. 确保构建、测试、制品和部署结果具有机器证据。
7. 降低定制代码对商城上游版本升级的阻碍。

### 2.2 非目标

- 不重新设计已有云商城的全部微服务边界。
- 不替代代码托管、CI/CD、Nacos、制品仓库、容器平台或发布平台。
- 不规定唯一的厂商产品或工具。
- 不覆盖部署后的告警、值班、事故响应、容量运营和系统退役。

## 3. AI-DLC 与外部工程体系边界

| 能力 | AI-DLC 职责 | 外部工程体系职责 |
|------|-------------|------------------|
| 需求与变更 | 需求澄清、CR 影响评估、产物回写 | 需求/任务系统中的负责人、优先级和审批状态 |
| 系统事实 | 读取并引用事实，生成分析摘要 | 服务目录、仓库、运行时依赖、环境和 Owner 的权威数据 |
| API/Event 契约 | 驱动契约设计、检查实现一致性 | 契约仓库、Schema Registry、兼容性检查和版本发布 |
| Nacos 配置 | 识别影响、生成变更计划和验证项 | 配置版本、权限、环境隔离、发布、审计和回滚 |
| 开发实现 | 计划、TDD、代码审查、精准修改 | 分支保护、CODEOWNERS、静态检查和合并策略 |
| 测试 | 派生用例、选择范围、记录证据 | 测试环境、数据、服务虚拟化、执行器和报告存储 |
| 制品 | 验证构建和记录制品信息 | 制品仓库、镜像签名、漏洞扫描和不可变标识 |
| 部署 | 生成/校验配置并驱动已批准步骤 | 部署平台、权限、环境锁、执行、回滚和部署记录 |

任何外部能力缺失时，AI-DLC 只能标记为“未验证”或“阻塞”，不得用 Markdown 说明代替真实平台执行结果。

## 4. 角色与责任

| 角色 | 最低责任 |
|------|----------|
| 产品负责人 | 确认定制目标、业务边界、验收标准和需求优先级 |
| 架构师 | 审批跨服务影响、契约、一致性、公共组件和高风险配置变更 |
| 服务 Owner | 维护服务事实、代码、契约、数据库和服务级验证结果 |
| 集成 Owner | 维护外部系统协议、环境、凭据、限额、故障策略和联调计划 |
| QA/测试负责人 | 审批跨服务测试范围、测试数据和业务链路回归结果 |
| 平台/DevOps | 提供 CI/CD、Nacos、制品、测试环境和部署平台保障 |
| 安全负责人 | 审批鉴权、权限、敏感数据、Secret 和高风险依赖变更 |
| 发布负责人 | 确认制品、迁移、发布顺序、smoke test 和回滚条件 |

每个服务、契约、Nacos 共享配置、外部系统和数据库必须至少有一个明确 Owner。

## 5. 变更风险分级

| 级别 | 典型变更 | 最低保障 |
|------|----------|----------|
| L1 服务内 | 单服务内部逻辑，不改变接口、配置和数据语义 | 服务内 TDD、静态检查、模块回归 |
| L2 契约级 | Controller、Feign DTO、事件字段、错误语义变化 | 契约兼容检查、Provider/Consumer 测试、Owner 审批 |
| L3 业务链路级 | 订单、库存、支付、优惠、履约、ERP 等多服务流程 | 一致性设计、故障用例、跨服务集成和业务链路回归 |
| L4 平台级 | Gateway、Nacos 共享配置、认证、公共库、共享中间件 | 架构/安全审批、扩大影响域、隔离环境验证、专项回滚 |
| L5 数据/发布级 | 破坏性 DDL、数据修复、服务名/Namespace 变化、发布顺序变化 | 迁移演练、兼容窗口、部署编排、回滚或前滚修复演练 |

风险级别只允许升级，不允许因修改文件少而降级。一个共享 Nacos 配置或 Feign DTO 的单行变更可以是 L4。

## 6. 存量系统事实基线

### 6.1 服务目录

项目必须维护可查询的服务目录，至少包括：

| 字段 | 要求 |
|------|------|
| 服务 ID | 全局唯一且稳定 |
| `spring.application.name` | 与 Nacos 注册名一致 |
| 业务职责 | 明确边界和非职责 |
| Owner | 团队和负责人 |
| 仓库/目录 | 可定位源代码 |
| 构建入口 | 可执行命令或流水线 |
| 数据所有权 | 数据库、Schema、核心表 |
| 同步依赖 | Feign/RPC/HTTP 提供方和消费者 |
| 异步依赖 | Topic、生产者和消费者 |
| 外部依赖 | 搜索、推荐、直播、ERP 等 |
| Nacos 配置 | Data ID、Group、Namespace |
| 部署单元 | 应用、镜像和部署目标 |
| 关键级别 | 核心、重要、普通 |

### 6.2 运行时依赖图

运行时依赖图必须覆盖：

- Gateway 到服务的路由。
- Feign/RPC/HTTP 调用。
- MQ Topic 的生产者和消费者。
- 服务到数据库、Redis、对象存储和搜索引擎的依赖。
- 服务到外部系统的调用和回调。
- Nacos 共享配置到使用服务的反向关系。
- 定时任务、数据同步任务和批处理任务。

仅分析 Maven/Gradle 依赖不能作为跨服务影响分析的充分证据。

### 6.3 基线更新门禁

发生以下变化时必须同步更新服务目录和依赖图：

- 新增、拆分、合并或下线服务。
- 修改服务名、Gateway 路由或 Nacos Namespace/Group。
- 新增 Feign Client、Topic、消费者或外部系统。
- 修改数据所有权或跨服务数据访问方式。
- 引入新的共享配置、公共库或中间件。

未完成基线更新的变更不得进入部署阶段。

## 7. Nacos 配置保障

### 7.1 配置目录

必须维护 Nacos 配置目录，至少包含：

| 字段 | 要求 |
|------|------|
| Data ID | 精确名称 |
| Group | 精确名称 |
| Namespace/环境 | dev、test、staging、prod 等 |
| 使用服务 | 完整消费者列表 |
| Owner | 负责团队 |
| 动态刷新 | 是否使用动态刷新及风险 |
| 敏感性 | 普通、敏感、Secret 引用 |
| 默认值 | 缺失时行为 |
| 重启要求 | 是否需要重启 |
| 回滚方式 | 版本或快照 |

### 7.2 配置事实与权限

- 配置字段定义、模板和变更记录必须版本化。
- Nacos 实际值不能成为唯一事实来源。
- Secret 不得写入普通配置仓库、日志或 AI-DLC 文档。
- dev/test/staging/prod 使用独立 Namespace 或等价隔离机制。
- 生产配置发布采用最小权限和审批。
- 共享配置变更必须列出全部消费者。

### 7.3 配置变更门禁

每次配置变更必须记录：

1. 配置项及旧值语义。
2. 新值及新语义。
3. 影响服务和环境。
4. 是否动态刷新。
5. 是否需要重启。
6. 验证步骤。
7. 回滚值或配置版本。
8. 审批人与发布时间窗口。

以下配置必须进行行为验证，不能只做 YAML/Properties 语法检查：

- 数据源、Redis、MQ 和搜索连接。
- Gateway Route、白名单和权限。
- Feign 超时、重试和负载均衡。
- 线程池、限流、熔断和分布式锁。
- 定时任务和功能开关。
- 外部系统地址、签名和凭据引用。

### 7.4 隔离环境验证

Nacos 变更先在隔离 Namespace 验证：

- 服务能启动并加载目标配置。
- 配置动态刷新行为符合预期。
- 共享配置不会影响未计划服务。
- 配置缺失、类型错误和旧版本字段有明确失败行为。
- 回滚到前一版本后服务恢复正常。

## 8. API、Feign 与事件契约保障

### 8.1 契约事实来源

- HTTP API 使用 OpenAPI 或等价机器可读 Schema。
- 异步事件使用 AsyncAPI、JSON Schema、Avro、Protobuf 或等价 Schema。
- Feign Client 必须与 Provider 契约关联，不能作为独立且可能漂移的第二份事实来源。
- Markdown 契约只保存索引、决策、Owner 和变更摘要。

### 8.2 同步接口最低字段

- 路径、方法、协议和版本。
- 完整请求/响应 Schema。
- 字段约束、必填性、枚举和默认值。
- 错误码、错误响应和错误语义。
- 鉴权和权限要求。
- 超时、重试和幂等要求。
- Provider 和 Consumer 清单。
- 弃用计划和兼容窗口。

### 8.3 事件契约最低字段

- Topic、Event Type 和版本。
- Producer 和 Consumer 清单。
- 完整 Payload Schema。
- 消息键、分区和顺序要求。
- 投递语义、去重和重放策略。
- 字段新增、删除和类型变化的兼容规则。
- 失败队列或人工处理入口。

### 8.4 兼容性门禁

PR 中必须自动执行：

- Schema 语法检查。
- 当前版本与基线版本 diff。
- Breaking change 检查。
- Provider 契约测试。
- Consumer 契约测试。
- Feign Client 与 Provider 契约一致性检查。
- 契约版本和消费者适配状态检查。

破坏性变更必须由 Provider Owner、受影响 Consumer Owner 和架构师审批。

### 8.5 逐消费者适配状态

一条变更影响多个消费者时，必须逐消费者记录：

| 契约版本 | 消费者 | 适配状态 | CI/测试证据 | Owner |
|----------|--------|----------|-------------|-------|
| 示例 v2 | consumer-a | 已完成 | 构建/流水线标识 | 团队 A |
| 示例 v2 | consumer-b | 待处理 | - | 团队 B |

不得使用一个全局“已同步”状态代替逐消费者确认。

## 9. 数据库与数据迁移保障

### 9.1 数据所有权

- 每张核心业务表必须有唯一服务 Owner。
- 服务不得绕过 API/Event 直接修改其他服务拥有的数据。
- 只读报表、同步库和数据仓库必须明确复制延迟及数据用途。
- 共享数据库现状无法立即拆除时，必须登记跨服务访问并制定收敛计划。

### 9.2 兼容迁移策略

优先采用扩展再收缩：

1. 新增兼容字段、表或索引。
2. 发布兼容新旧结构的代码。
3. 执行数据迁移或双写切换。
4. 验证旧版本不再依赖旧结构。
5. 在后续独立版本清理旧字段和逻辑。

### 9.3 数据变更门禁

- 迁移脚本受版本控制。
- 脚本能识别是否已执行，避免重复破坏。
- 有回滚方案或明确的前滚修复方案。
- 对大表 DDL 评估锁表、耗时和存储增长。
- 数据修复脚本执行前后校验影响行数和业务总量。
- 枚举、状态和金额语义变更需要产品及数据 Owner 审批。
- staging 或等价环境完成迁移演练。
- 应用和数据库版本不兼容时，不得进行无序滚动发布。

## 10. 分布式一致性保障

### 10.1 适用范围

涉及两个及以上服务的写操作必须登记为分布式业务流程，例如：

- 下单与库存预占。
- 支付确认与订单状态。
- 退款与优惠/积分返还。
- 履约与库存扣减。
- 商城单据与 ERP 同步。

### 10.2 强制设计内容

每条流程必须明确：

- 发起方和参与服务。
- 每个服务的本地事务边界。
- 同步调用、事件和状态转换。
- 幂等键及有效期。
- 消息发送和消费可靠性。
- 重复、乱序、延迟和丢失的处理。
- 超时和重试上限。
- 补偿动作和补偿失败处理。
- 对账周期、差异判定和人工修复入口。
- 最终一致性允许时限。
- 全链路关联 ID。

### 10.3 允许的保障模式

根据现有系统选择并保持一致：

- Saga 编排或协同。
- Transactional Outbox。
- Inbox/消费去重。
- 事务消息。
- 显式业务状态机。
- 定时对账与补偿任务。

不得用“通过 MQ 最终一致”代替具体设计。

### 10.4 一致性测试门禁

至少覆盖：

- 消息重复、延迟和乱序。
- Producer 本地事务成功但消息发送失败。
- Consumer 处理成功但确认失败。
- 服务重启后的恢复。
- 下游持续不可用。
- 补偿失败和重试耗尽。
- 外部系统重复回调。
- 对账发现差异及人工修复。

## 11. 外部系统集成保障

### 11.1 集成目录

搜索、推荐、直播、ERP 等每个外部系统必须登记：

| 维度 | 必填内容 |
|------|----------|
| Owner | 内部负责人和供应商联系人 |
| 用途 | 业务场景和核心程度 |
| 协议 | HTTP、MQ、文件、SDK 等 |
| 环境 | sandbox、test、staging、prod |
| 鉴权 | Token、签名、证书及轮换 |
| 容量 | QPS、配额、批量限制 |
| SLA | 可用性、延迟和支持窗口 |
| 韧性 | 超时、重试、熔断、隔离和降级 |
| 数据 | 字段映射、主数据归属、时区和编码 |
| 一致性 | 回调、补偿、对账和人工修复 |
| 测试 | Mock、Sandbox 和故障场景 |
| 版本 | 当前版本、兼容周期和升级通知 |

### 11.2 适配层

业务代码不得直接依赖供应商 SDK、DTO 和异常类型。外部系统通过稳定的内部 Port/Adapter 隔离，以支持：

- 供应商替换。
- 统一超时、重试和日志。
- Mock/Stub 测试。
- 字段映射和协议升级。
- 业务降级和故障隔离。

### 11.3 外部系统测试门禁

至少验证：

- 正常响应。
- 连接失败和超时。
- 401/403 和凭据失效。
- 429 和配额耗尽。
- 5xx、非法响应、空响应和字段缺失。
- 重复回调和回调乱序。
- 重试不会造成重复订单、库存或 ERP 单据。
- 降级路径符合业务验收标准。

核心流程依赖的外部系统没有 sandbox 或可控虚拟服务时，对应测试必须标记为阻塞或未验证。

## 12. 安全与 Secret 保障

### 12.1 最低要求

- 用户身份、服务身份和外部系统身份分离。
- 服务间调用具有可验证的认证与授权机制。
- Gateway 白名单和内部接口访问范围受审查。
- Secret 通过 Secret 管理系统或受控环境变量提供。
- 凭据具有 Owner、用途、有效期和轮换机制。
- 敏感数据按分类执行加密、脱敏和最小化返回。
- 日志和 AI-DLC 证据不得包含 Token、密码、私钥和个人敏感数据。

### 12.2 CI 安全门禁

按项目现有工具执行：

- 依赖漏洞扫描。
- Secret 泄漏检查。
- 静态安全分析。
- 容器镜像漏洞扫描。
- 许可证或 SBOM 检查（项目要求时）。

安全检查未配置时应记录缺口，不得将“未配置”等同于“通过”。

## 13. 代码仓库与并行协作保障

### 13.1 分支与审批

- 主分支启用保护，禁止未审查直接合并。
- 服务目录、契约、Nacos 共享配置和数据库迁移设置 CODEOWNERS 或等价审批规则。
- 契约变更优先于 Provider/Consumer 实现合并。
- 高风险配置和数据库变更使用独立 PR。
- PR 必须关联需求/CR、受影响服务和验证证据。

### 13.2 工作认领

单元认领使用 Issue、任务系统、PR 标签或具有原子更新能力的机制，最低要求：

- 单元 ID 唯一。
- 同一单元只有一个活跃 Owner。
- 接管记录原 Owner、原因和时间。
- CI 或任务系统阻止重复认领。
- Markdown 状态只做摘要，不作为高并发锁。

### 13.3 AI-DLC 状态分片接口

多仓库场景建议每个服务仓库维护自身 `docs/aidlc/state.md`；产品治理仓库维护全局 CR、服务目录、依赖图和契约索引。单仓库场景也应尽量使用服务级状态文件，产品级状态只保存汇总和交接索引。

## 14. CI 质量门禁

### 14.1 每次提交或 PR

- 编译。
- 单元测试和组件测试。
- 格式、lint、类型和静态分析。
- API/Event 契约兼容检查。
- Provider/Consumer 契约测试。
- 数据库迁移静态检查。
- 依赖和 Secret 检查。
- 受影响服务识别。

### 14.2 合并前

- 受影响服务及直接运行时消费者测试。
- Gateway/Feign/Event 集成验证。
- Nacos 隔离环境配置验证。
- 数据迁移演练（适用时）。
- 外部系统虚拟化或 sandbox 测试。
- P0/P1 UC-D 覆盖和通过率检查。

### 14.3 部署前

- 目标制品重新确认，不在部署阶段重新编译未审查代码。
- 镜像和依赖扫描。
- 跨服务业务链路回归。
- 配置、迁移、发布顺序和回滚检查。
- staging smoke test。

流水线结果必须提供不可变运行标识、Commit、制品版本、时间和结果，供 AI-DLC 记录为外部证据。

## 15. 测试环境与数据保障

### 15.1 环境隔离

- 测试和 staging 使用独立 Nacos Namespace 或等价隔离。
- 数据库、Redis、MQ Topic、搜索索引和对象存储避免与生产混用。
- 外部系统使用 sandbox、Mock 或服务虚拟化。
- 环境配置可以从版本化模板重建。

### 15.2 测试数据

- 提供可重复初始化和清理的数据脚本。
- 不从生产环境复制未经脱敏的个人或交易数据。
- 订单、库存、支付和 ERP 测试数据具有唯一标识，避免与真实单据混淆。
- 并发、边界和失败场景数据可按需构造。

### 15.3 测试层次

| 层次 | 目标 |
|------|------|
| 单元测试 | 服务内部业务规则 |
| 组件测试 | 单服务与数据库、缓存、MQ 的协作 |
| 契约测试 | Provider、Consumer、事件 Schema 一致性 |
| 服务集成测试 | 2-3 个真实服务协作 |
| 外部系统测试 | 协议、故障、限流和降级 |
| 业务链路 E2E | 关键商城流程 |
| 一致性测试 | 重复、乱序、补偿和对账 |
| 性能测试 | 明确指标下的容量和延迟 |
| 安全测试 | 认证、授权、敏感数据和接口暴露 |

## 16. 定制代码与上游升级保障

### 16.1 定制台账

必须维护定制台账，至少包括：

| 字段 | 要求 |
|------|------|
| 定制编号 | 与需求/CR 关联 |
| 上游基线版本 | 开始定制时的商城版本 |
| 受影响服务 | 完整列表 |
| 修改原文件 | 路径和原因 |
| 扩展方式 | 配置、插件、Adapter、继承、直接修改 |
| 契约/配置/数据变化 | 对应版本和迁移 |
| 测试证据 | 流水线和报告 |
| 升级风险 | 低、中、高 |
| Owner | 维护责任人 |

### 16.2 扩展优先级

优先顺序：

1. 配置和特性开关。
2. 现有扩展点或策略接口。
3. 独立 Adapter、Decorator 或插件。
4. 组合或继承。
5. 最后才直接修改商城核心代码。

不得为了避免直接修改而引入不必要的复杂抽象；选择必须有当前需求依据。

### 16.3 上游升级门禁

- 对比新旧上游版本和定制台账。
- 标识所有被上游修改的定制文件。
- 重新执行契约、数据库和 Nacos 差异检查。
- 重跑受影响服务和业务链路测试。
- 完成升级演练后再进入正式部署。

## 17. 制品与部署保障

### 17.1 制品

- 制品与 Commit、构建流水线和依赖锁定信息关联。
- 镜像使用不可变摘要或唯一版本，不使用可覆盖的模糊标签作为部署证据。
- 制品进入仓库后不得人工修改。
- 记录镜像扫描、签名或等价可信验证结果。

### 17.2 部署前门禁

- 部署服务和版本清单已确认。
- 服务发布顺序和兼容窗口已确认。
- Nacos 配置版本及发布顺序已确认。
- 数据库迁移和回滚/前滚方案已确认。
- 外部系统环境和凭据已确认。
- 目标环境资源和依赖可用。
- smoke test 和回滚触发条件明确。

### 17.3 实际部署证据

部署完成至少记录：

- 部署平台运行标识。
- 环境、服务、镜像摘要和配置版本。
- 数据库迁移结果。
- 服务注册与健康检查结果。
- Gateway 路由和关键依赖连通性。
- smoke test 命令/用例及结果。
- 回滚是否执行；未执行时记录已验证的回滚方案。

部署失败时不得只回滚应用而忽略数据库和配置兼容性。

## 18. 与 AI-DLC 的证据接口

外部平台向 AI-DLC 提供以下最小证据：

| 外部能力 | 最小证据 |
|----------|----------|
| 服务目录 | 版本或更新时间、服务清单、Owner |
| 契约平台 | 契约版本、兼容检查结果、消费者状态 |
| Nacos | Data ID/Group/Namespace、配置版本、发布和回滚记录 |
| CI | 运行标识、Commit、范围、命令/任务、通过/失败 |
| 测试平台 | 环境、用例范围、通过/失败、报告地址 |
| 制品仓库 | 制品版本/摘要、构建来源、扫描结果 |
| 数据库变更 | 迁移版本、演练结果、执行结果 |
| 部署平台 | 环境、服务版本、配置版本、结果和 smoke test |

AI-DLC 只在证据可核验且与当前 Commit/制品一致时标记相应门禁通过。

## 19. 分阶段落地建议

### 阶段 A：事实基线

完成：

- 服务目录和 Owner。
- 运行时依赖图。
- Nacos 配置目录。
- 外部系统目录。
- 数据所有权清单。
- 上游版本与定制台账初始基线。

完成标准：任一服务、接口、Topic、共享配置和外部系统均可定位 Owner 与消费者。

### 阶段 B：机器门禁

完成：

- API/Event Schema 事实来源。
- 契约兼容检查。
- 分支保护和 CODEOWNERS。
- CI 构建、测试、静态和安全检查。
- 数据库迁移检查。
- Nacos 隔离环境验证。

完成标准：破坏性契约、重复认领、未审查高风险配置和未通过测试的代码不能合并。

### 阶段 C：系统级验证

完成：

- 跨服务业务链路测试。
- 分布式一致性故障测试。
- 外部系统 Mock/Sandbox。
- staging 环境和可重复测试数据。
- 部署与回滚演练。

完成标准：核心定制链路在正常、超时、重复、乱序、下游失败和补偿场景下都有证据。

### 阶段 D：规模化使用

完成：

- 每服务 AI-DLC 状态与产品级汇总。
- 外部证据自动回写或可查询链接。
- 定制台账进入上游升级流程。
- 门禁指标持续统计。

完成标准：扩大到更多服务不会显著增加状态冲突、契约漏检和人工重复核对。

## 21. Go/No-Go 标准

### Go

- 服务、配置、契约、数据和外部系统均有 Owner。
- 破坏性契约变更能被机器检查发现。
- 受影响 Provider/Consumer 测试全部通过。
- 跨服务写链路有一致性、补偿和对账设计。
- Nacos 变更在隔离环境验证并有回滚版本。
- 外部系统异常和降级用例有证据。
- 数据迁移已演练。
- 部署制品、配置、迁移和 smoke test 可追溯。

### No-Go

出现任一项即阻断合并或部署：

- 服务、配置、契约或外部系统无 Owner。
- Breaking change 未识别或消费者未确认。
- 跨服务写链路没有幂等、补偿或对账方案。
- 共享 Nacos 配置未识别全部消费者。
- 外部调用无超时或存在无界重试。
- 数据迁移未验证且无前滚/回滚方案。
- 必需测试失败、跳过或没有可核验证据。
- 部署制品与已审查 Commit 不一致。
- Secret 出现在代码、普通配置、日志或 AI-DLC 文档中。

## 22. 建议度量指标

| 指标 | 建议目标 |
|------|----------|
| 服务/契约/配置/外部系统 Owner 覆盖率 | 100% |
| P0/P1 需求到测试证据追溯率 | 100% |
| Breaking change 漏检数 | 0 |
| 重复认领或状态覆盖次数 | 0 |
| 共享 Nacos 配置消费者识别率 | 100% |
| 数据迁移演练覆盖率 | 100% |
| 外部系统异常场景覆盖率 | 核心集成 100% |
| staging 部署与 smoke test 成功率 | 100% 后方可进入目标环境 |
| 硬编码 Secret | 0 |
| 定制项上游版本可追溯率 | 100% |

指标用于发现保障失效，不应通过删除用例、降低范围或伪造不适用来满足。

## 23. 最终原则

大型存量商城定制开发的外部工程保障应达到：

- 系统事实可查询。
- 服务和数据有 Owner。
- 契约可执行、变更可检测。
- Nacos 配置可版本化、可隔离、可回滚。
- 一致性和外部故障可模拟。
- 并行协作不会覆盖状态。
- 测试、制品和部署证据可复现。
- 定制代码可以随上游版本持续演进。

AI-DLC 可以驱动和消费这些保障，但不能替代负责执行和强制这些保障的工程平台。
