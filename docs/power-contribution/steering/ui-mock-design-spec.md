# UI Mock 设计规范

生成或编辑 HTML 格式的 UI Mock 时，必须遵循以下规范。

---

## 1. UI Mock 生成流程（必须按顺序执行）

生成每个页面的 UI Mock 时，必须按以下步骤执行：

### 步骤 1：状态分析（先于画页面）

在开始画任何页面 mock 之前，先扫描该页面涉及的所有用户故事和验收条件，识别出：
- 该页面是否存在多种状态（如订单状态、审核状态、权限状态等）
- 每种状态下哪些 UI 元素会发生变化（显示/隐藏/文案变化/样式变化）

**判断规则**：如果一个页面存在 2 种及以上状态导致 UI 呈现不同，则必须先生成条件表。

### 步骤 2：生成条件表（内嵌在 HTML mock 顶部）

对于多状态页面，在 HTML mock 文件的顶部，使用可折叠面板放置条件表：

```html
<details open class="state-condition-panel">
  <summary>📋 状态条件表（本页面涉及 N 种状态，基准状态：XXX）</summary>
  <table class="condition-table">
    <thead>
      <tr>
        <th>区域/组件</th>
        <th>状态A（基准）</th>
        <th>状态B</th>
        <th>状态C</th>
      </tr>
    </thead>
    <tbody>
      <!-- 逐行列出每个会变化的 UI 区域 -->
    </tbody>
  </table>
</details>
<hr class="mock-divider">
```

条件表中需要标注：
- 哪个是"基准状态"（页面 mock 画的就是这个状态）
- 每个状态下各区域的变化（显示/隐藏/替换文案/变更样式）
- 操作按钮的变化

### 条件表横向滚动（强制）

条件表经常有 3-5 列甚至更多，在 mock-box 的有限宽度内容易溢出。**必须**为条件表的外层容器设置 `overflow-x: auto`，确保用户可以横向滚动查看完整内容。

**强制 CSS 规则**（在 HTML `<style>` 中声明）：

```css
.condition-panel { overflow-x: auto; }
.condition-panel table { min-width: 500px; /* 防止表格被挤压换行 */ }
```

**禁止行为**：
- ❌ 条件表容器没有 `overflow-x: auto`，导致内容被截断不可见
- ❌ 表格列宽被强制压缩到无法阅读（使用 `min-width` 保证最小可读宽度）
- ✅ 条件表可横向滚动，所有列完整可见

### 步骤 3：生成页面 Mock

在条件表下方，画出"基准状态"的完整 UI Mock。基准状态选择信息量最丰富的那个状态（通常是"正常使用中"而非空状态或终态）。

### 无状态差异的页面

如果页面不涉及多状态（如纯设置页、纯列表页无状态筛选），则跳过步骤1-2，直接画 mock。

### 步骤 4：局部改动页面的特殊处理

当 mock-box 标注为「局部改动」时，表示该页面是对现有系统页面的增量修改，**不是重新设计**。

**核心原则**：局部改动 = 现有页面 + 增量变更。mock 必须忠实还原现有页面的全部已有元素，只在改动点处做增量修改。

**必须保持不变的元素**（从逆向工程产物或现有系统截图中获取）：
- Tab 栏的全部选项（不能减少、不能改名、不能重排）
- 搜索栏的已有字段（不能删减、不能改 label）
- 表格的已有列（不能删减、不能改顺序）
- 状态概览卡片（如果现有页面有就保留）
- 操作列的已有按钮（不能删减）
- 分页组件的样式

**允许做的增量变更**：
- 在搜索栏中**新增**筛选字段（用黄色边框+「新增」标注）
- 在表格中**新增**列（用黄色边框+「新增」标注）
- 修改某列的展示逻辑（标注变更原因）
- 新增操作按钮

**执行流程**：
1. 先确认这是「局部改动」还是「新增页面」
2. 如果是局部改动 → 从逆向工程产物中读取现有页面结构
3. 在 mock 中完整还原现有页面
4. 仅在改动点处标注变更（黄色边框 + 「新增」/「变更」标记）
5. 底部说明中列出「改动清单」（新增了什么、改了什么）

**禁止行为**：
- ❌ 简化现有页面的 Tab 项（如把7个 tab 缩减为4个）
- ❌ 删减现有搜索栏字段（如去掉"店铺名称""用户名"等）
- ❌ 重新设计表格列结构（如把现有列替换为全新列）
- ❌ 去掉现有页面的状态概览卡片区域
- ✅ 保持现有全部元素，仅在其中插入新增的改动点

## 2. 表单条件联动规范

### 规则：每个选项驱动的子组件都必须展示

当表单中存在 radio/select/toggle 控件，且不同选项会导致**不同的子组件/子字段**出现时，**必须在 mock 中展示每种选项对应的子组件**。

**判定标准**：表单字段的选项数 > 1 且各选项后续展示不同的输入组件 → 每种选项的展开态都必须画出来。

### 展示方式

在 mock 中使用「条件分支展示」结构，用虚线边框和标注区分不同选项的展开态：

```html
<!-- 选项A的展开态（基准） -->
<div class="mock-form-row">
  <label>有效期 *</label>
  <input type="radio" checked> 购买后 <input value="30" style="width:60px"> 天
</div>

<!-- 选项B的展开态（标注"选中固定日期范围时展示"） -->
<div style="border:1px dashed var(--color-info);border-radius:4px;padding:8px 12px;margin-top:8px;position:relative;">
  <span style="position:absolute;top:-9px;left:8px;background:var(--color-info);color:#fff;font-size:9px;padding:0 3px;border-radius:2px;">选中「固定日期范围」时</span>
  <label>有效期 *</label>
  <input type="radio" checked> 固定日期范围
  <input type="date" value="2026-07-01"> 至 <input type="date" value="2026-12-31">
</div>
```

### 适用场景

- Radio 切换后出现不同输入框（如：时长输入 vs 日期范围选择器）
- Select 选择后联动显示子表单区块
- Toggle 开关后展开/收起额外配置
- 级联选择（如省市区）

### 禁止行为

- ❌ 只画基准选项的展开态，其他选项的子组件省略不画
- ❌ 只在底部业务说明里文字描述「选中 B 时显示日期选择器」
- ✅ 每种选项都画出对应的子组件，用标注区分

## 3. 交互模式映射表

| 按钮动作 | 交互方式 | 原型要求 |
|----------|----------|----------|
| 查看详情、查看信息、查看记录 | **详情弹框** | 信息展示区 + 关闭按钮 |
| 新增、添加、绑定、创建 | **表单弹框** | 表单字段 + 提交 + 取消 |
| 编辑、修改、配置 | **表单弹框**（预填） | 预填表单 + 保存 + 取消 |
| 删除、解绑、作废、禁用 | **确认弹框** | 警告图标 + 文案 + 确认 + 取消 |
| 上传、导入 | **上传弹框** | 文件选择 + 预览 + 提交 |
| 审核、验收 | **操作弹框** | 意见输入 + 确认 + 取消 |
| 导出、下载 | **直接触发 + Toast** | 点击后 Toast 提示 |
| 刷新、同步 | **Loading + Toast** | 按钮加载态 + 完成提示 |

---

## 3. 反馈规范

| 操作结果 | 反馈方式 |
|----------|----------|
| 操作成功 | Toast 绿色提示（3s 自动消失），弹框关闭，列表刷新 |
| 操作失败 | Toast 红色提示（手动关闭），弹框保持，显示错误 |
| 网络异常 | Toast "网络异常，请稍后重试" |
| 表单校验失败 | 字段下方红色文字，输入框红色边框 |
| 删除确认 | 二次确认弹框"确定要删除吗？此操作不可恢复" |

---

## 4. 空状态规范

| 场景 | 表现 |
|------|------|
| 列表无数据 | 空状态图标 + "暂无数据" + 操作引导（如"点击新增"） |
| 搜索无结果 | "未找到匹配结果，请调整搜索条件" |
| 暂无图片/视频 | 灰色渐变占位 + 图标 |
| 网络加载中 | 骨架屏（Skeleton）占位 |
| 页面加载中 | 顶部进度条 + 内容区骨架屏 |

---

## 5. 响应式布局规范

| 断点 | 宽度范围 | 侧边栏 | 内容区 |
|------|----------|--------|--------|
| **超宽屏** | ≥1920px | 240px 固定 | 最大1440px居中 |
| **宽屏** | 1440-1919px | 220px 固定 | 流式 |
| **标准** | 1280-1439px | 200px 固定 | 流式 |
| **窄屏** | 1024-1279px | 64px 折叠 | 流式 |
| **移动端** | <1024px | 隐藏 | 全宽 |

---

## 6. 移动端/小程序特殊规范

| 规范项 | 说明 |
|--------|------|
| 弹框替代 | 底部弹出面板（ActionSheet）替代居中弹框 |
| 确认操作 | 居中确认弹框（宽度85%） |
| 表单编辑 | 优先跳转新页面而非弹框 |
| Toast | 轻提示居中，2s自动消失 |
| 手势交互 | 支持下拉刷新、左滑删除 |
| 安全区域 | 底部留出 safe-area-inset-bottom |

---

## 7. 图标规范

**图标来源**：使用统一图标集
- Web端：Element Plus Icons / Lucide Icons
- 移动端：UniApp uni-icons
- 小程序：微信原生图标

**图标尺寸**：
- 导航图标：20px
- 操作图标：16px
- 状态图标：14px
- 大图标：24px（统计卡片）

---

## 8. 生成前自检清单

UI Mock 生成前必须逐项检查：

- [ ] 所有 CSS 变量使用统一命名
- [ ] 按钮尺寸符合规范（36px基准）
- [ ] 卡片圆角使用 var(--radius-card)
- [ ] 阴影层次正确（card用sm，弹框用xl）
- [ ] 文字颜色使用语义化变量
- [ ] 每个按钮都有对应的交互原型
- [ ] 表格行hover有视觉反馈
- [ ] 弹框有入场动画
- [ ] 空状态有占位设计
- [ ] 多状态页面已生成条件表
- [ ] 条件表标注了基准状态
- [ ] 表单中所有 radio/select 联动的子组件，每种选项的展开态都已画出
- [ ] 标注为「局部改动」的页面，已完整还原现有页面全部元素（Tab、搜索栏、表格列、状态卡片），仅在改动点做增量标注
- [ ] 如存在 page-specs.md，确认当前 HTML 中的 mock-box 数量和内容与 page-specs.md 清单一致（无遗漏、无多余）
