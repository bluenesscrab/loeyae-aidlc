# 产品需求概览（AI-DLC 侧）

## 基线状态

- **变更来源**：`CR-I5-SCOPE-001`
- **批准依据**：SSOT 权威序列 `PM-PRODUCT-BOUNDARY-019` 至 `CR3-PLAN-APPROVAL-040`
- **当前批次**：CR4 B1 双仓产品权威基线
- **风险级别**：L4

## 产品定位

AI-DLC 是软件开发生命周期的角色化资料消费和正式研发文档产出层。它根据当前项目、用户角色、目标文档和提示词，通过 SSOT 自动检索可多轮演进的沟通/过程资料，构建带来源版本/片段的上下文，并在项目工作区生成正式研发文档。

本地 `docs/aidlc/state.md` 保持 v2，并继续作为流程恢复唯一来源。正式研发文档正文和批准版本由项目工作区/Git或既有文档库权威管理；SSOT 权威管理沟通/过程资料及逆向工程派生说明。Kiro、Claude Code 和 OpenCode 仅是共享流程语义的薄适配，不成为业务项目属性。

## 目标角色与核心输出

| 角色 | 资料消费目标 | 首期核心正式文档 |
|------|--------------|------------------|
| 产品经理 | 汇总调研、会议、业务沟通和显式结论 | 调研/会议整理、PRD、产品/业务蓝图 |
| 架构师 | 获取技术约束、系统边界和对接资料 | 技术架构、架构图说明、技术决策与设计文档 |
| 项目经理 | 获取范围、结论、风险和未决项，辅助项目规划 | 项目计划、范围/风险/沟通说明；实际任务与工时仍归外部系统 |
| 开发人员 | 获取需求、接口、数据、规范和逆向说明 | 开发设计、接口/对接说明、数据库设计、代码规范与实现输入 |
| 测试人员 | 获取验收、异常、边界、历史变化和来源 | 测试计划、测试用例、验收与缺陷分析文档 |

独立角色模板和最终有限目录将在 CR4 B4 冻结；B1 不宣称模板已实现或验证。

## 首期交付范围

1. 从本地 state v2 恢复当前阶段、步骤、模块和活跃 CR，不执行 state v3 迁移。
2. 解析当前业务工作区的 SSOT 项目绑定，通过 Provider 契约访问资料；未配置 SSOT 时保持既有本地流程。
3. 根据角色、目标文档和提示词自动检索当前项目资料，默认使用最新修订。
4. 支持用户显式包含、排除资料，或指定旧版；不得因自动选择隐藏实际使用范围。
5. 消费元数据、全文、向量 RAG 和单项目 GraphRAG 结果，并显示索引失败、低置信和降级状态。
6. 构建包含资料 ID、版本、片段、来源、新鲜度和索引状态的上下文包。
7. 在项目工作区生成正式文档，列出具体资料版本/片段引用，并写回资料到文档章节的血缘。
8. 将逆向工程说明上传 SSOT 并关联仓库路径、Git commit 与内容哈希；源代码事实仍以 Git 为准。
9. Kiro、Claude Code 和 OpenCode 使用同一共享流程与数据语义，只适配调用、呈现和凭据接入。

## 明确非范围与历史候选

- 不实现 SSOT 服务端、解析 Worker、数据库、Portal 或索引算法。
- 不把完整 Federated Manifest、统一事件、远端 CR、checkpoint 或 Trace 发布作为首期强制流程。
- 不迁移 state v2 到 v3，不把 SSOT checkpoint 变成本地恢复权威。
- 不支持跨项目检索、跨项目共享目录或企业完整授权治理。
- 不替代 Git、任务/人力、CI/测试执行、制品或部署平台。
- 不自动批准需求、架构、安全、正式文档或发布决定。
- 不在 B1 修改 `src/`、插件、Hook、MCP 配置或发布包。

## 端到端闭环

1. AI-DLC 从当前业务工作区的 state v2 恢复流程；平台切换不改变项目状态或产物位置。
2. 根据当前项目、角色和目标文档生成资料意图，并通过 `ssot-api` 解析项目。
3. 自动检索当前项目资料，展示范围；用户可包含、排除或指定旧版。
4. 消费带资料版本/片段、来源、新鲜度和索引状态的上下文包；索引异常时按 Provider 语义降级。
5. 在项目工作区生成正式文档草案，区分已确认资料、未确认讨论和模型推断。
6. 输出具体来源引用，并写回资料版本/片段到正式文档章节的血缘记录。
7. 正式文档由项目现有审批和 Git 流程管理；AI-DLC 不把生成草案自动标记为批准版本。
8. 逆向说明可上传 SSOT，但代码事实、仓库状态和提交仍以 Git 为准。

## 全局约束

- 业务项目只维护自身工作区的一套 AI-DLC state 和过程产物，不维护 SSOT/AI-DLC 双仓或双流程。
- 本仓 `docs/aidlc/` 只用于开发 AI-DLC 自身，必须排除在三平台运行时分发物之外。
- `state.md` 保持 v2；未配置 SSOT 的存量项目不得产生额外远程调用或被阻断。
- AI-DLC 只通过 SSOT API/MCP 观察资料、索引状态和降级，不直接访问 Worker 或数据库。
- 默认使用最新修订；指定旧版、固定版本和片段引用必须可重现。
- 自动检索必须展示实际使用资料；显式包含/排除和用户修订优先于自动选择。
- 输出必须区分资料事实、未确认讨论、显式结论和模型推断，不得把推断伪装成来源事实。
- context handle、令牌和 Secret 只存在于内存或安全凭据存储，不写入 Git、提示词、文档或审计。
- 三平台共享业务语义，平台层不得复制 SSOT 权限、版本、检索或血缘规则。
- B1 只回写产品级边界；字段契约、缓存、错误/降级和平台适配细节留待 I12。

## 非功能需求概要

- **平台无关**：相同 state、项目、角色、目标文档和资料版本在三平台产生一致的选择与引用语义。
- **安全**：项目隔离、最小权限和敏感信息边界须通过负向测试；不得跨项目缓存或泄露隐藏图关系。
- **可追溯**：每个生成章节可定位到资料版本/片段；逆向说明可定位到 commit/内容哈希。
- **兼容**：未配置 SSOT 的 Legacy 流程保持可用；state v2 不迁移。
- **可恢复**：SSOT 或索引不可用时保持本地流程位置，明确标记未验证/降级，不伪造远端事实。
- **可验证**：契约、上下文、引用、血缘、权限、Legacy 和三平台结论必须有可复现命令与报告。

## 验收方向与证据缺口

- 文档级：双仓定位、五类角色、业务项目单工作区、Git 权威、Provider-first 和外部事实边界一致。
- Consumer：自动检索、包含/排除、最新/旧版、上下文包、引用、章节血缘和逆向上传均需夹具与测试。
- 平台：Kiro、Claude Code、OpenCode 必须通过共享语义 conformance，但业务项目不绑定具体平台。
- E2E：至少一个脱敏真实项目以三类以上资料完成“检索→上下文→正式文档→引用/血缘”闭环。
- 当前 npm 1.20.0 和现有文件只证明存量基线存在；没有 build/test script、Provider Schema、模板验收或运行时报告，所有新能力保持“未验证”。
