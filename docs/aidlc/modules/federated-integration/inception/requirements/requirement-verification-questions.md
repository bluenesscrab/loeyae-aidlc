## 产品视角澄清

### Q-PROD-001：Federated 人工审批与身份模型

**为什么必须现在决定**：审批入口决定 AI-DLC 在 `change.submit` 后是返回 Portal 链接、提供 IDE 人工审批界面，还是只轮询状态；同时决定三平台凭据、重认证、审计和职责分离范围。

| 选项 | 方案 | AI-DLC 行为 |
|------|------|-------------|
| A（推荐） | 企业 OIDC；SSOT Portal 是唯一人工批准入口 | Agent 只提案/提交；展示审批链接；轮询或通过事件接收结果；永不持有审批能力 |
| B | 企业 OIDC；Portal 与 IDE 都可作为人工批准界面，统一调用 SSOT 审批 API | 三平台必须展示同一差异、执行强认证并记录人类确认；实现与测试范围更大 |
| C | 首期使用 SSOT 本地账号并仅在 Portal 审批，后续迁移企业 OIDC | 客户端先实现本地登录/令牌适配，后续承担身份迁移与审计连续性改造 |

**状态**：已批准 A（2026-07-17T06:40:50Z）。企业 OIDC 是身份源；SSOT Portal 是唯一人工批准入口；AI-DLC 只提案/提交、展示审批链接并轮询或消费事件，不持有批准能力。

## 用户视角澄清

### Q-USER-001：等待审批期间的工作阻断范围

**为什么必须现在决定**：用户从任一平台提交变更后需要离开 IDE 到 Portal 审批。AI-DLC 必须明确保存什么状态、哪些工作可以继续，以及批准/拒绝后如何恢复。

| 选项 | AI-DLC 行为 | 风险与约束 |
|------|-------------|------------|
| A | 提交后暂停整个项目，直到 Portal 批准或拒绝 | 实现简单，但无关模块也被阻断 |
| B（推荐） | 只阻断影响分析确定的模块、服务、单元和下游产物；不受影响的已批准范围可继续。state 保存 SSOT CR、影响范围和审批链接，通过事件或恢复轮询解除门禁 | 兼顾权威与并行；需要精确影响域和跨平台一致恢复 |
| C | 允许基于未批准候选继续生成下游草案，统一标记未批准，Construction 前阻断 | 易产生失效草案和权威混淆，后续清理成本高 |

**状态**：已批准 B（2026-07-17T07:08:15Z）。仅阻断影响分析确定的模块、服务、单元和下游产物；其他已批准且不受影响的工作继续。state 保存 SSOT CR、影响范围与审批链接，通过事件或恢复轮询解除门禁。

## 架构视角澄清

### Q-ARCH-001：业务工作区与 SSOT 项目的稳定绑定来源

**数据源溯源**：三平台必须在首次 I1 和恢复时得到同一项目候选，且不能根据仓库名称猜测。绑定来源必须可审计、无 Secret、支持多根工作区和项目切换。

| 选项 | AI-DLC 行为 | 风险与约束 |
|------|-------------|------------|
| A（推荐） | 从业务仓库版本化的非敏感 `docs/aidlc/ssot-binding.json` 读取 Schema 版本、组织/项目稳定 ID、默认发布和服务别名；调用 `project.resolve` 结合 OIDC 身份、服务端仓库绑定和权限复核，冲突时要求用户选择 | 三平台和离线识别一致；需保护绑定文件变更并验证服务端事实 |
| B | 不保存本地绑定；每次由 SSOT 根据 Git 远程 URL/仓库指纹解析 | 本地简单，但离线、多根和镜像仓库场景不稳定 |
| C | 只依赖 `state.md` 中的组织/项目/发布字段 | 首次启动和 state 重建无来源，人工修改风险高，职责混淆 |

**状态**：已批准 A（2026-07-17T08:26:38Z）。业务仓库版本化保存非敏感 `docs/aidlc/ssot-binding.json`；`project.resolve` 结合企业 OIDC、服务端仓库绑定和权限复核；歧义或冲突必须由用户显式选择。

## 开发视角澄清

### Q-DEV-001：三平台共享 SSOT 集成语义的实现方式

**为什么必须现在决定**：三个入口的插件与 Hook 能力不同，但绑定、MCP、state、错误、幂等和门禁语义不得分叉。

| 选项 | AI-DLC 实现 | 风险与约束 |
|------|-------------|------------|
| A（推荐） | 建立平台无关的共享协议核心与一致性测试包，统一契约模型、状态机、错误映射、幂等和门禁；Kiro、Claude Code、OpenCode 只实现凭据、MCP 传输与 Hook 触发等薄适配。无法直接复用代码的平台也必须通过同一 conformance suite | 共享语义且保留平台适配能力；需要清晰的核心/适配器边界 |
| B | 三平台统一调用一个 Node CLI/Sidecar，由单一进程完成全部 SSOT 交互 | 语义集中，但引入额外运行时、安装和进程故障边界 |
| C | 三个平台各自完整实现 SSOT 集成 | 容易形成三套行为，升级和故障语义难以一致 |

**状态**：已批准 A（2026-07-17T08:37:22Z）。建立平台无关共享协议核心与 conformance suite；三平台只实现薄适配，不能直接复用代码时也必须通过同一套一致性测试。

## 测试视角澄清

### Q-TEST-001：平台无关的项目流程不变量（已纠正并确认）

**最终结论**：平台不是项目属性，也不是项目门禁维度。`ssot-binding.json`、Manifest、state 和 SSOT 产品事实均不得记录“项目绑定到哪个 AI 平台”。Kiro、Claude Code、OpenCode 只是可替换的执行客户端。

项目级门禁只验证平台无关结果：批准基线、影响域、契约版本、执行状态、测试证据和审计。任何受支持客户端都必须能读取同一工作区状态并继续流程；平台差异由共享协议核心与薄适配器消化。

三平台 conformance、适配器兼容矩阵和跨平台接力测试属于本仓产品发布责任，不进入业务项目范围或门禁。客户端版本不兼容时，只阻断该客户端并提示升级或切换，不得修改或废止项目状态。
