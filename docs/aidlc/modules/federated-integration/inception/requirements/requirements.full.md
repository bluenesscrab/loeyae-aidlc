## 1. 意图分析

- **请求类型**：L4 产品定位纠偏后的 AI-DLC Consumer 需求基线回写。
- **目标**：使产品经理、架构师、项目经理、开发人员和测试人员在业务项目工作区中，按角色和目标文档消费 SSOT 沟通/过程资料，生成可引用、可追溯的正式研发文档。
- **需求深度**：全面。
- **权威输入**：SSOT 冻结澄清 019—039、已批准 CR3 计划、已批准 B1 产品基线和 041。
- **首期原则**：state v2 不迁移、业务项目单工作区、Provider-first、默认最新修订、显式资料选择、固定引用、三平台薄适配、Legacy 零远程调用。

## 2. 产品边界

### 2.1 首期范围

1. 从当前业务工作区 `docs/aidlc/state.md` v2 恢复流程，不读取插件源码仓的自举状态。
2. 根据产品经理、架构师、项目经理、开发人员、测试人员五类角色和目标文档形成资料意图。
3. 仅通过 SSOT Provider API/MCP 解析已配置项目、查询资料和版本、检索项目资料、构建上下文包。
4. 默认使用最新资料修订，支持显式包含、排除和指定旧版本；范围过大、低置信或冲突时请求确认。
5. 在项目工作区生成正式研发文档，区分已确认资料、未确认讨论、显式结论和模型推断。
6. 输出实际使用的资料、修订和片段引用，并回写“资料版本/片段 → 正式文档章节”血缘。
7. 可选上传逆向工程说明，并携带仓库路径、Git commit 和内容哈希。
8. Kiro、Claude Code、OpenCode 共享同一业务语义，只处理调用、凭据和交互差异。
9. Provider 或索引不可用时保留本地流程位置，明确失败/降级，不伪造远端事实。

### 2.2 非范围

- state v2→v3 迁移、Manifest、新鲜度事件游标、远端 CR、影响分析控制面和 Portal 审批交接。
- AI-DLC 直接访问 `ssot-worker`、数据库、对象存储或复制 SSOT 资料治理规则。
- 把客户端平台写入业务项目属性、state 门禁或正式文档事实。
- 在未配置 SSOT 的 Legacy 项目中产生额外远程调用或阻断。
- 管理任务、人力、工时、实际进度、CI/测试执行、制品或部署事实。
- 把模型推断、未确认讨论或 GraphRAG 关系伪装成已确认资料。

## 3. 角色与目标文档

| 角色 | 核心目标文档 | 资料意图重点 | 责任边界 |
|------|--------------|--------------|----------|
| 产品经理 | 调研/会议、PRD、产品/业务蓝图 | 业务目标、用户旅程、范围、显式结论 | 产品/业务蓝图负责，技术内容由架构师条件协作 |
| 架构师 | 技术架构、关键技术说明 | 约束、集成、技术资料、关系与来源 | 技术架构蓝图独立负责 |
| 项目经理 | 评估、计划、排期、风险建议 | 会议结论、范围、依赖和风险 | 不替代任务、人力和实际进度系统 |
| 开发人员 | 开发设计、对接、逆向说明 | 需求、接口沟通、代码事实引用 | 源代码/Git 是代码事实权威 |
| 测试人员 | 测试计划、用例、报告框架 | 需求、设计、风险和确认结论 | 不伪造测试/CI 执行结果 |

### 3.1 端到端旅程

```text
state v2 恢复 → RoleIntent/TargetDocument → Provider 项目解析
→ 自动检索 + 包含/排除/旧版 → ContextBundle → 正式文档生成
→ FragmentCitation → DocumentSection/LineageRecord → 可选逆向说明上传
```

## 4. 功能需求与验收标准

### ADLC-FR-001 state v2 与业务工作区唯一恢复源（Must）

AI-DLC 必须先从当前业务工作区恢复 state v2；不得读取插件安装根目录的 `docs/aidlc/` 作为业务状态，也不得执行 state v3 迁移。

```gherkin
Given 插件源码仓和业务项目都存在 docs/aidlc/state.md
When 任一受支持客户端恢复业务项目
Then 只使用业务项目 state v2，流程位置不因客户端或插件自举状态改变
```

### ADLC-FR-002 五角色意图与目标文档（Must）

AI-DLC 必须形成 `RoleIntent` 和 `TargetDocument`，覆盖五类角色及其有限核心文档目录；角色、目标文档和提示词共同决定资料意图。

```gherkin
Given 用户声明当前角色为测试人员且目标为测试计划
When AI-DLC 形成资料意图
Then 意图包含测试计划类型、当前项目和相关需求/设计资料条件，不加载项目管理执行数据作为权威事实
```

### ADLC-FR-003 Provider-first 项目解析（Must）

已配置 SSOT 的项目必须通过 Provider API/MCP 解析项目与访问上下文；AI-DLC 不得按仓库名猜测项目，也不得直接依赖 Worker 或数据库。

```gherkin
Given 本地配置与 Provider 返回的项目身份冲突
When AI-DLC 解析项目
Then 流程停止资料读取并要求修复或显式选择，不静默使用本地猜测
```

### ADLC-FR-004 自动检索与显式资料选择（Must）

`MaterialSelection` 必须按角色、目标文档和提示词自动检索当前项目，默认选择最新修订；用户可显式包含、排除或指定旧版。范围过大、低置信或冲突时必须请求确认。

```gherkin
Given 自动检索选中 M1/R2 和 M2/R1，用户排除 M2 并指定 M3/R4
When AI-DLC 确认资料选择
Then 最终选择只包含 M1/R2 与 M3/R4，并记录每项选择、排除和旧版理由
```

### ADLC-FR-005 上下文包固定与降级透明（Must）

AI-DLC 必须消费 Provider 构建的 `ContextBundle`，固定资料 ID、修订、片段、来源、讨论/结论状态、索引状态、实际检索路径和预算。索引降级必须对生成流程可见。

```gherkin
Given GraphRAG 不可用而 Provider 降级到向量与全文
When AI-DLC 接收上下文包
Then 记录实际降级路径并只引用返回的具体修订/片段，不把图推断缺失解释为资料不存在
```

### ADLC-FR-006 正式文档生成与事实分层（Must）

AI-DLC 必须在项目工作区生成正式研发文档，并明确区分已确认资料、未确认讨论、显式结论和模型推断；正文与批准版本由 Git或既有文档库权威管理。

```gherkin
Given 上下文包含一项已确认结论、一条未确认评论和一项模型推断
When 生成正式文档
Then 三者在输出中具有不同来源状态，未确认评论和推断均不得写成已批准事实
```

### ADLC-FR-007 片段引用与章节血缘（Must）

每个有资料支撑的生成章节必须包含 `FragmentCitation`，并将 `DocumentSection` 与 `LineageRecord` 回写 Provider；固定引用必须持续指向使用时的具体修订和片段。

```gherkin
Given 生成文档后源资料从 R1 更新为 R2
When 用户检查原文档章节来源
Then 章节引用仍解析到 R1 的原片段，默认新生成任务才使用 R2
```

### ADLC-FR-008 逆向工程说明上传（Must）

AI-DLC 可选生成并上传 `ReverseDocUpload`，至少携带说明内容、仓库路径、Git commit 和内容哈希。说明至少覆盖项目最底层目录结构、技术栈、模块详细功能与核心逻辑/流程图、数据库规范与详细设计、代码规范；不得反向覆盖代码事实。

```gherkin
Given 当前仓库 commit 为 C1 且逆向说明内容哈希为 H1
When AI-DLC 上传说明
Then Provider 返回固定修订引用 C1/H1，源代码和 Git 历史不被修改
```

### ADLC-FR-009 三平台共享语义与薄适配（Must）

Kiro、Claude Code、OpenCode 必须共享 RoleIntent、TargetDocument、MaterialSelection、ContextBundle、引用和血缘语义；适配层只能处理调用、凭据、Hook 和呈现差异。

```gherkin
Given 三个平台读取同一 state v2、角色、目标文档和 Provider 响应
When 执行资料选择与引用生成
Then 选择的资料修订、片段引用和下一流程状态语义一致，平台名称不写入项目事实
```

### ADLC-FR-010 Legacy 零远程调用（Must）

未配置 SSOT 的现有业务项目必须保持原有 AI-DLC 行为，不进行 SSOT 网络调用，不要求任何远端字段参与门禁。

```gherkin
Given 业务项目未配置 SSOT
When 使用升级后的 AI-DLC 恢复并执行既有流程
Then SSOT 远程调用数为 0，state v2 和本地流程行为不因本模块受阻
```

### ADLC-FR-011 Provider/索引失败与本地恢复（Must）

Provider 不可用、权限拒绝、解析/索引失败或返回冲突时，AI-DLC 必须保留本地流程位置，显示稳定原因、影响和恢复动作；不得伪造远端读取、引用或同步成功。

```gherkin
Given Provider 在上下文构建时不可用
When AI-DLC 处理失败
Then state v2 的当前步骤保持不变，输出明确未取得远端资料，不生成虚假引用或“已同步”状态
```

### ADLC-FR-012 外部事实边界（Must）

AI-DLC 可利用资料辅助生成项目管理、测试和交付文档，但任务、人力、工时、实际进度、CI/测试执行、制品和部署结果必须来自各自权威平台；缺稳定证据时标记未验证。

```gherkin
Given 沟通资料声称测试已通过但没有测试平台运行标识
When AI-DLC 生成测试报告框架
Then 将该声明标为未验证资料，不生成“测试实际通过”的结论
```

### ADLC-FR-013 真实项目端到端验收（Must）

首期必须与 Provider 使用一个真实项目、至少三类代表性沟通资料和多轮修订完成角色化正式文档闭环；必须验证自动选择、显式包含/排除/旧版、降级、人工修订、片段引用、章节血缘、逆向说明 Git 关联及三平台业务语义一致。在取得稳定运行标识前，该能力保持未验证。

```gherkin
Given 一个真实项目包含至少三类资料、两个以上修订并已配置 SSOT
When 分别通过受支持客户端执行资料选择、上下文、正式文档生成和血缘回写
Then 实际选择和固定引用可复现，平台差异不改变业务语义；缺少运行标识或 Provider 证据时不得标记端到端通过
```

## 7. 优先级

### 7.1 MoSCoW

- **Must**：ADLC-FR-001—013、ADLC-NFR-001—008。
- **Should**：资料选择预览、冲突对比、引用完整性报告。
- **Could**：上下文质量可视化、跨文档血缘浏览、角色资料推荐。
- **Won't（首期）**：state v3、Manifest/事件/远端 CR、跨项目共享、AI-DLC 直连 Worker/数据库、平台绑定项目、离线伪造远端事实。

### 7.2 RICE 排序约束

当前无经批准的人月和用户量估计，不伪造数值 RICE 分数。实现顺序按依赖确定：state v2/Legacy → 角色/目标 → Provider 项目解析 → 资料选择 → 上下文 → 正式文档 → 引用/血缘 → 逆向上传 → 三平台 conformance。I11/I14 在取得工作量证据后补充量化 RICE。

## 9. 风险与约束

- SSOT Provider 尚无实现和机器 Schema，AI-DLC 当前不能取得真实资料上下文。
- `package.json` 当前没有 build/test script；Construction 前必须建立可复现测试入口。
- 三平台凭据与 Hook 能力不同，但差异不得渗入项目状态和业务语义。
- 上下文超限、错误资料选择、引用漂移、逆向说明与 Git commit 错配是首期高风险项。
- 字段级契约、稳定错误码、重试、幂等和质量阈值留待 I12/I13，B2 不提前实施。

## 11. 决策记录

1. AI-DLC 是角色化资料消费和正式研发文档产出出口，不是 SSOT 事实副本。
2. state 保持 v2；业务项目只维护自身工作区的一套 AI-DLC 状态和过程产物。
3. Consumer 只通过 Provider 契约访问资料，不直接依赖 Worker 或数据库。
4. 默认使用最新修订，支持包含、排除和指定旧版；低置信/冲突时请求确认。
5. 正式文档生成到项目工作区并保留片段引用和章节血缘。
6. Kiro、Claude Code、OpenCode 只做薄适配，不成为项目属性。
