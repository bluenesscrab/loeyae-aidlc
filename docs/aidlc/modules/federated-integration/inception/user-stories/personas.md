## 共同原则

1. 五类角色在同一业务项目工作区使用 `docs/aidlc/state.md` v2；客户端平台不是项目属性。
2. AI-DLC 只通过八项已批准文档契约消费 Provider，不复制字段 Schema、资料治理规则或 Provider 内部实现。
3. 默认使用最新修订，支持显式包含、排除和指定旧版；低置信、冲突或范围过大时必须请求确认。
4. 正式研发文档正文归项目工作区/Git；源代码事实归 Git；外部执行事实归各专业平台。
5. 未配置 SSOT 的 Legacy 项目保持零远程调用；Provider 失败不得推进错误流程状态或伪造远端成功。

## ADLC-PER-001 产品经理

- **核心目标**：按业务目标和目标文档检索产品资料，生成有固定来源的需求、用户故事和产品/业务蓝图。
- **核心正式文档位置**：`inception/requirements/`、`inception/user-stories/`。
- **资料意图**：调研、会议、PRD、用户旅程、范围、优先级和显式业务结论。
- **主要痛点**：自动选择混入无关资料；未确认评论被写成已批准事实；源资料更新后引用漂移。
- **权限与边界**：负责产品/业务蓝图，技术内容由架构师条件协作；不得把模型推断或外部执行声明提升为事实。
- **核心场景**：选择角色和目标文档、确认资料选择、生成事实分层文档、核对并回写章节血缘。
- **关联故事**：`ADLC-US-001`—`ADLC-US-007`、`ADLC-US-009`—`ADLC-US-013`。

## ADLC-PER-002 架构师

- **核心目标**：消费技术资料和关系上下文，生成可追踪的应用设计与关键技术说明。
- **核心正式文档位置**：`inception/application-design/`。
- **资料意图**：技术约束、集成、架构决策、GraphRAG 关系、精确片段和逆向说明。
- **主要痛点**：Provider 降级被隐藏；图推断被误作事实；逆向说明与 Git commit 不一致。
- **权限与边界**：负责技术架构蓝图；不复制 Provider Schema，不以逆向说明覆盖代码事实。
- **核心场景**：形成架构资料意图、检查实际检索路径、生成设计文档、回写血缘和上传逆向说明。
- **关联故事**：`ADLC-US-001`—`ADLC-US-009`、`ADLC-US-011`、`ADLC-US-013`。

## ADLC-PER-003 项目经理

- **核心目标**：基于已确认范围、依赖和风险资料生成评估、计划、排期与风险建议。
- **核心正式文档位置**：`inception/plans/`、`construction/plans/`。
- **资料意图**：会议结论、范围、依赖、风险、计划假设和外部平台证据引用。
- **主要痛点**：口头进度被当作权威事实；冲突资料被静默合并；Provider 失败导致本地步骤误推进。
- **权限与边界**：不替代任务、人力、工时和实际进度系统；无稳定标识的外部信息必须标记未验证。
- **核心场景**：恢复本地计划步骤、确认资料范围、生成计划建议、在失败时保持原步骤、引用外部稳定证据。
- **关联故事**：`ADLC-US-001`—`ADLC-US-007`、`ADLC-US-009`—`ADLC-US-013`。

## ADLC-PER-004 开发人员

- **核心目标**：基于需求、接口和技术资料生成开发设计、对接说明和与 Git 事实绑定的逆向说明。
- **核心正式文档位置**：`inception/reverse-engineering/`、`inception/application-design/`、`construction/<unit>/`。
- **资料意图**：需求、接口沟通、技术资料、指定旧版、片段引用、仓库路径、commit 和内容哈希。
- **主要痛点**：历史资料被自动提升到最新版；逆向说明上传重试产生重复；客户端差异改变业务语义。
- **权限与边界**：源代码/Git 是代码事实权威；不直连 Worker、数据库或索引存储。
- **核心场景**：显式选择旧版、生成固定引用设计、上传逆向说明、从同一 state v2 在任一客户端恢复。
- **关联故事**：`ADLC-US-001`—`ADLC-US-011`、`ADLC-US-013`。

## ADLC-PER-005 测试人员

- **核心目标**：从需求、设计、风险和确认结论生成可追踪的测试计划、用例及报告框架。
- **核心正式文档位置**：`inception/application-design/test-cases/`、`construction/build-and-test/`。
- **资料意图**：需求 GWT、设计约束、风险、失败/降级、引用血缘和测试/CI 稳定运行标识。
- **主要痛点**：资料中的“测试通过”缺少运行证据；排除项重新进入上下文；无来源推断未被标识。
- **权限与边界**：不伪造测试/CI 执行结果；实际执行事实由测试/CI 平台权威管理。
- **核心场景**：形成测试资料意图、核对上下文最小化、生成用例、检查引用完整性并标记未验证外部信息。
- **关联故事**：`ADLC-US-001`—`ADLC-US-007`、`ADLC-US-009`—`ADLC-US-013`。
