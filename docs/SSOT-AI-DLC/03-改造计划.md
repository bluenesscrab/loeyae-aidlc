# AI-DLC 引入多项目 SSOT 改造实施计划

## 1. 计划目标

以向后兼容方式将多项目SSOT接入现有AI-DLC，先实现只读上下文和基线验证，再实现受控需求变更、CR1—CR5、证据回链和多项目协作。整个改造不得破坏Legacy项目和三平台恢复能力。

关联文档：

- [01-改造需求.md](./01-改造需求.md)
- [02-改造设计.md](./02-改造设计.md)
- [SSOT实施计划](../01-SSOT系统/03-实施计划.md)

## 2. 实施原则

1. 先协议后代码；
2. 先只读后写入；
3. 先提示门禁后强制门禁；
4. state本地恢复优先；
5. 所有改造通过Feature Flag条件启用；
6. Kiro、Claude Code、OpenCode共享语义，平台差异封装在适配层；
7. 每阶段都验证Legacy回归；
8. SSOT与AI-DLC仓库可独立发布和回滚。

## 3. 改造范围

### 3.1 AI-DLC核心

- 核心路由；
- state模式；
- 会话恢复；
- 步骤完成协议；
- 质量门禁；
- 审计和交接；
- Inception、Construction、Operations；
- CR1—CR5；
- 团队和多模块协作。

### 3.2 平台适配

- MCP客户端封装；
- Kiro Hooks；
- Claude Code Hooks/命令；
- OpenCode插件适配；
- 生成文件保护；
- Feature Flags和配置。

### 3.3 不在本改造中实现

- SSOT后端和Portal本身；
- 图算法和Embedding模型；
- 企业身份平台；
- CI、任务系统和部署平台；
- 生产数据迁移。

这些由SSOT或外部系统工作流负责，本项目只实现契约和集成。

## 4. 阶段规划

### A0：兼容性基线和协议冻结

**目标**：在修改AI-DLC前建立可回归基线和跨仓库契约。

**任务**：

- 固化当前Kiro、Claude Code、OpenCode关键场景；
- 建立state v2恢复黄金样本；
- 建立单人、团队、多模块和活跃CR样本；
- 与SSOT团队冻结MCP、Manifest、事件和错误码；
- 明确Legacy/Mirror/Federated模式；
- 明确远端不可用和Schema升级策略；
- 评审安全与权限传递。

**门禁**：

- 契约具有版本和Owner；
- Legacy行为有自动或可重复验证；
- 两个产品线里程碑和依赖一致；
- 未决关键问题进入阻塞清单。

### A1：state v3和条件加载

**目标**：引入SSOT字段但不改变默认流程。

**任务**：

- 新增`common-ssot-integration.md`；
- 扩展`core-workflow.md`按需加载入口；
- 实现state v3模板；
- 实现v2到v3无损迁移；
- 增加Legacy/Mirror/Federated模式；
- 扩展审计和交接格式；
- 增加Feature Flags。

**验收**：

- v2 Legacy项目可恢复；
- v3 Legacy项目行为不变；
- 迁移不改变当前步骤和活跃CR；
- SSOT字段不适用时无额外MCP调用。

### A2：只读MCP和上下文包

**目标**：三平台可从SSOT读取相同的可信项目上下文。

**任务**：

- 实现平台无关`SsotContextProvider`；
- 接入`project.resolve`；
- 接入需求、决策、问题和`context.build`；
- 读取和校验Manifest；
- 生成只读上下文缓存；
- 扩展I1、I2、I5、I7、I12、I13、I14上下文加载；
- 增加引用和版本显示。

**验收**：

- 三平台同一请求得到同一实体版本；
- 项目权限和上下文句柄正确；
- token预算生效；
- 生成投影不可被误认为权威源；
- MCP不可用时Legacy正常、Mirror/Federated按规则降级。

### A3：会话恢复和新鲜度门禁

**目标**：跨会话识别SSOT变化，防止基于旧需求继续实现。

**任务**：

- 改造`common-session-continuity.md`；
- 接入`changes.since`；
- 接入`baseline.verify`；
- 实现VALID、STALE_UNRELATED、STALE_RELEVANT等状态处理；
- 扩展Construction入场、C5和C8检查；
- 实现项目切换流程；
- 实现远端离线恢复。

**验收**：

- 相关变化阻断并路由CR；
- 无关变化可刷新游标且有证据；
- 离线时可从state恢复但不能伪造远端新鲜；
- 活跃CR恢复能对齐远端CR ID；
- 项目切换不混合上下文。

### A4：受控写入和CR1—CR5

**目标**：产品可在Kiro中完成受控变更，AI-DLC保持完整审批和影响流程。

**任务**：

- 接入`change.propose/submit`；
- 接入`impact.analyze`；
- 改造`change-request-process.md`；
- I5对已有需求输出结构化Patch；
- I6增加来源、版本和非法反转审查；
- CR3记录双端CR；
- CR4区分SSOT产品事实和Git机器事实；
- CR5接入一致性检查；
- 实现严格模式审批交互。

**验收**：

- F008示例端到端通过；
- Agent不能自动批准；
- SSOT提案失败不生成本地批准版本；
- L3—L5不得走快速通道；
- 必需项未验证时CR保持活跃；
- 跨项目共享升级形成项目独立CR。

### A5：步骤镜像、追踪与证据

**目标**：将AI-DLC派生产物和真实执行证据回链SSOT。

**任务**：

- 接入`trace.publish`；
- 接入`evidence.publish`；
- 接入`checkpoint.publish`和`sync.status`；
- 扩展步骤完成协议的远端镜像说明；
- 扩展用户故事、设计、测试、单元和实施报告的Trace字段；
- 实现待同步队列和幂等重试；
- 接入C8和Operations证据索引。

**验收**：

- 本地四步协议顺序不变；
- 镜像失败不重复完成本地步骤；
- 证据包含需求版本、提交、运行ID和制品摘要；
- 只有真实执行结果可标记通过；
- Federated CR在证据未同步时按规则保持活跃。

### A6：Hooks和投影保护

**目标**：提升日常使用体验，但不让Hook成为唯一门禁。

**任务**：

- 提供Kiro Hook模板；
- 提供Claude Code/OpenCode等价适配；
- SessionStart增量检查；
- Pre/PostTask门禁与镜像；
- Stop一致性检查；
- 生成投影文件写保护；
- 权限拒绝和循环Hook测试。

**验收**：

- Hook缺失时显式步骤检查仍有效；
- Hook不会自动批准或发布；
- 权限拒绝后不绕过重试；
- 不产生循环调用；
- 直接编辑投影会获得明确处理路径。

### A7：多项目、团队和发布验证

**目标**：验证真实团队协作和多个项目环境。

**任务**：

- 云Mall项目试点；
- 第二差异项目试点；
- 跨项目相同本地ID测试；
- 团队严格模式、接力和认领测试；
- 多模块切换测试；
- 共享能力升级和项目CR测试；
- 三平台兼容测试；
- 文档、迁移和发布准备。

**验收**：

- 两项目状态、上下文和证据完全隔离；
- state恢复和SSOT绑定一致；
- 团队认领仍由Git/任务系统决定；
- 共享升级不自动改项目；
- Legacy、Mirror、Federated三种模式均通过发布门禁。

## 7. 两仓协作与契约治理

SSOT与AI-DLC建议独立仓库：

```text
SSOT Repo
- MCP Server实现
- OpenAPI/MCP/事件Schema权威源
- Manifest生成

AI-DLC Repo
- MCP客户端适配
- steering和hooks
- Manifest消费
- AI-DLC行为测试
```

契约变更流程：

```text
提出契约变更
→ 兼容性分析
→ 发布候选Schema
→ 双方契约测试
→ 先发布兼容Provider
→ 发布Consumer
→ 观察
→ 弃用旧版本
```

Breaking change必须进入双方CR，记录版本并提供迁移窗口。

## 8. 测试与验证矩阵

### 8.1 模式矩阵

| 维度 | 取值 |
|---|---|
| 平台 | Kiro / Claude Code / OpenCode |
| 项目类型 | 新项目 / 存量项目 |
| 协作 | 单人 / 团队 |
| 架构 | 单模块 / 多模块 |
| SSOT模式 | Legacy / Mirror / Federated |
| 连接 | 在线 / 超时 / 离线 / 恢复 |
| 基线 | 有效 / 无关过期 / 相关过期 / Schema不兼容 |
| CR | 无 / 活跃L1 / 活跃L3 / 活跃L5 |

不要求对笛卡尔积所有组合做完整E2E，但高风险组合必须覆盖，其他通过分层测试降低数量。

### 8.2 必测高风险场景

1. v2团队项目带活跃CR升级到v3；
2. Federated项目在C5前发生相关需求变化；
3. SSOT在步骤完成后、checkpoint前故障；
4. SSOT在产品版本发布事务中失败；
5. 两项目都存在F008，Agent切换项目；
6. 用户无跨项目权限但GraphRAG存在关系；
7. Hook触发MCP并再次触发Hook的循环防护；
8. AI推导与产品Won't Have冲突；
9. CI证据无稳定运行ID；
10. Manifest被人工修改；
11. 共享定义升级但项目未接受；
12. Claude Code接手Kiro中断的活跃单元。

### 8.3 门禁

- 单元测试和Schema验证；
- steering内容校验；
- 状态迁移测试；
- 三平台恢复测试；
- MCP契约测试；
- 权限与负向测试；
- Hook循环和拒绝语义；
- Legacy回归；
- F008端到端验收。

## 9. 发布策略

### 9.1 版本阶段

```text
alpha：开发环境，Legacy默认
beta-1：Mirror只读，单项目
beta-2：Mirror多项目，提示门禁
rc-1：Federated受控写入，强制CR
rc-2：证据闭环与三平台验证
stable：第二项目、迁移和回滚验证通过
```

### 9.2 开启顺序

1. 发布支持SSOT但默认关闭的AI-DLC版本；
2. 云Mall项目启用Mirror只读；
3. 开启新鲜度提示；
4. 开启Construction强制门禁；
5. 开启Federated受控写入；
6. 开启证据和checkpoint；
7. 第二项目验证；
8. 再考虑默认启用新项目模板。

## 10. 回滚策略

- 关闭`ssot.integration.enabled`可回到Legacy，但Federated项目不得把未同步需求视为本地批准；
- 保留state v3，旧版本应忽略未知SSOT字段或通过兼容读取；
- 不删除Manifest和本地审计；
- Hook可单独禁用；
- 写入MCP可关闭而保留只读；
- 回滚前确认没有活跃Federated CR依赖新版本；
- Breaking Schema不得通过简单回滚解决，必须执行兼容迁移。

## 11. 风险与应对

| 风险 | 应对 |
|---|---|
| AI-DLC规则被SSOT逻辑污染 | 新增共享steering，条件加载，平台无关Adapter |
| state和SSOT产生双主 | state只管执行，SSOT只管产品事实 |
| 离线导致无法工作 | state优先恢复、冻结快照只读降级 |
| Agent绕过CR直接编辑投影 | 文件标记、Hook保护、质量门禁 |
| 多平台行为不一致 | 共享语义测试和平台适配层 |
| 同步重试产生重复记录 | 幂等键、事件ID、状态查询 |
| 远端checkpoint阻塞本地流程 | checkpoint非权威，失败进入待同步 |
| 远端产品发布失败被当成完成 | Federated发布是阻断门禁 |
| GraphRAG过度自信 | inferred标识、证据缺口、不得下调风险 |
| state迁移破坏恢复位置 | 原文件摘要、无损迁移、失败回退 |
| Hook循环 | 调用来源标记、最大深度和循环测试 |

## 13. 粗略周期

在SSOT MCP只读接口按期可用的前提下：

- A0—A2：约3—5周；
- A3—A4：约3—4周；
- A5—A6：约2—3周；
- A7和稳定发布：约2—3周。

合计约8—12周，部分工作可并行。该估计必须在读取AI-DLC实际仓库、完成影响分析和单元生成后修订。

## 14. 完成定义

AI-DLC SSOT改造稳定版完成必须满足：

1. 需求和设计中的必需项均有实现或批准的不适用依据；
2. Legacy模式完整回归通过；
3. state v2/v3恢复和迁移通过；
4. 三平台至少完成核心恢复、CR和Construction场景；
5. 云Mall项目完成Kiro产品变更到Claude Code开发验证闭环；
6. 第二项目完成隔离和项目切换验证；
7. SSOT离线、超时、权限拒绝和Schema不兼容场景通过；
8. 所有真实测试、CI和制品证据可复现；
9. 迁移、回滚、运维和用户文档完成；
10. 未执行项明确为未验证，不以文档审阅代替运行结果。

## 15. 建议首个验收脚本

以`WEHK:F008`为端到端样例：

```text
1. Kiro以产品角色读取F008@v13
2. 提出“进入拣货后不可取消并提示联系客服”
3. AI-DLC识别已有需求变化，进入CR1
4. CR2返回影响范围和L3风险，严格模式等待确认
5. CR3生成计划并绑定SSOT CR
6. 产品批准，SSOT发布F008@v14
7. Claude Code恢复UNIT-ORDER-03
8. baseline.verify返回STALE_RELEVANT并阻断C5
9. 按CR4更新故事、设计、契约索引、代码和测试
10. C8执行实际构建和测试并发布证据
11. CR5检查需求—设计—实现—证据一致
12. 清除活跃CR，更新state、审计和交接
13. 在SSOT离线条件下验证恢复和阻断行为
```

该脚本通过后，再扩展到共享能力升级和跨项目场景。
