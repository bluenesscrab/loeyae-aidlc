# AI-DLC 引入多项目 SSOT 改造设计

## 1. 设计目标

在不破坏现有AI-DLC三阶段流程、审批模式、跨平台恢复和本地证据规则的前提下，引入SSOT作为产品事实控制平面，实现多项目上下文绑定、最小上下文加载、版本锁定、CR协同和证据回链。

关联文档：

- [01-改造需求.md](./01-改造需求.md)
- [03-改造计划.md](./03-改造计划.md)
- [SSOT架构设计](../01-SSOT系统/02-架构设计.md)

## 2. 设计原则

1. **state优先恢复**：任何会话先读`state.md`，SSOT验证在本地位置恢复后执行；
2. **分域权威**：产品事实、执行状态、代码契约和外部证据分别归属权威系统；
3. **可选集成**：SSOT能力通过条件加载，不影响Legacy项目；
4. **显式项目**：每个调用和会话都绑定组织、项目、发布和基线；
5. **先读后写**：写入前读取当前版本、生成差异并进行影响分析；
6. **审批不降级**：远端批准失败不能转成本地文档批准；
7. **最小上下文**：只加载当前步骤和工作单元需要的事实切片；
8. **证据可重放**：同步调用携带幂等键和稳定标识；
9. **候选不等于事实**：GraphRAG和AI抽取不自动进入批准基线；
10. **平台中立**：共享语义不依赖Kiro独有能力，Hook只做平台适配。


## 8. MCP适配层

### 8.1 调用上下文

AI-DLC在I1或恢复时调用`project.resolve`，获得短时`context_handle`。句柄绑定：

- 用户或服务身份；
- 组织、项目群和项目；
- 发布和基线；
- 权限；
- 到期时间。

句柄只存于当前进程内存或安全凭据存储，不写入`state.md`和审计。

### 8.2 调用包装器

建议AI-DLC内部抽象平台无关接口：

```text
SsotContextProvider
- resolve_project()
- get_context_bundle()
- verify_baseline()
- get_changes_since()
- propose_change()
- analyze_impact()
- publish_trace()
- publish_evidence()
- publish_checkpoint()
```

Kiro、Claude Code和OpenCode仅实现如何调用MCP及Hook，不复制业务规则。

### 8.3 幂等键

格式建议：

```text
{project}:{aidlc-run}:{stage}:{step}:{operation}:{content-digest}
```

例如：

```text
WEHK:run-20260718:C8:evidence.publish:sha256-abc
```

## 9. 会话恢复流程

```text
1. 读取state.md
2. 恢复当前阶段、步骤、模块、服务、单元和活跃CR
3. 判断SSOT模式
4. Legacy：按原流程继续
5. Mirror/Federated：验证Manifest文件和摘要
6. 调用project.resolve
7. 调用changes.since(event_cursor)
8. 调用baseline.verify(当前单元/步骤范围)
9. 更新新鲜度结论和游标
10. 宣布恢复状态并按场景继续
```

SSOT不可用时：

```text
state恢复成功
→ Manifest有效则加载冻结上下文
→ 标记remote_unverified
→ 允许本地只读分析
→ 阻断需求发布、CR关闭和依赖远端批准的步骤
```

## 10. 上下文包设计

请求：

```json
{
  "context_handle": "ctx-...",
  "stage": "Construction",
  "step": "C5",
  "role": "developer",
  "module_id": "order",
  "service_ids": ["order-service"],
  "unit_id": "UNIT-ORDER-03",
  "purpose": "code_generation",
  "token_budget": 16000
}
```

返回分段：

```text
binding              项目和基线
approved_requirements 当前需求切片
acceptance_criteria   验收标准
approved_decisions    决策
open_questions        阻塞和非阻塞问题
systems_interfaces    系统、接口和Owner索引
contracts             Git机器契约位置和摘要
runtime_impact        verified运行时关系
trace_links           故事、设计、单元和测试
recent_changes        与当前范围相关的变化
citations             精确来源
```

上下文包有`bundle_id`和`bundle_digest`，写入当前单元计划或state索引，避免后续无法判断基于哪组事实执行。

## 11. 各阶段详细流程

### 11.1 I1

- 检测`.kiro/settings/mcp.json`等平台配置存在性时只判断能力，不读取Secret；
- 查找`ssot-manifest.json`；
- 从`state.md`恢复模式和绑定；
- 未绑定时允许用户选择Legacy或显式项目；
- 绑定结果写state并审计。

### 11.2 I5/I6

```text
requirement.get(current)
→ 对比用户输入
→ 生成JSON Patch、变更原因和未决问题
→ impact.analyze（可在CR2执行）
→ change.propose
→ 产品审查
→ I6验证来源、非法反转和状态
```

新项目首次需求可批量创建候选，但批准仍逐批执行并生成基线。

### 11.3 I7— I14

- 故事、设计、测试和单元继续在Git生成；
- 文件头或索引保存来源实体版本；
- 每步完成通过`trace.publish`发布关系；
- 远端发布失败记录待同步，不覆盖本地产物；
- Federated模式下，如果上游需求未批准则阻断进入Construction。

### 11.4 C5

```text
读取单元绑定的requirement_versions
→ baseline.verify(scope=unit)
├── VALID：继续
├── STALE_UNRELATED：记录并刷新游标
├── STALE_RELEVANT：停止，进入CR1
├── REMOTE_UNAVAILABLE：冻结只读，禁止开始依赖新批准事实的实现
└── SCHEMA_INCOMPATIBLE/PERMISSION_DENIED：阻断
```

### 11.5 C8

本地先产生真实证据：

- 命令；
- 工作目录；
- 退出码；
- 测试统计；
- 不可变提交；
- CI运行ID；
- 制品摘要；
- 范围和时间。

然后`evidence.publish`。远端写入失败不把真实通过改成失败，但必须标记“SSOT证据未同步”，且需要SSOT闭环的CR不能关闭。

### 11.6 Operations

发布快照引用：

```text
SSOT baseline
Git commit
machine contract versions
configuration references
migration versions
artifact digest
CI evidence IDs
```

不记录Secret，不将部署准备等同实际生产部署。

## 12. CR1—CR5设计

### CR1范围定位

合并两类事实：

- SSOT：需求、决策、关系、外部系统和历史版本；
- 工作区：故事、设计、契约、代码、配置、测试和当前提交。

输出每项来源和新鲜度。

### CR2影响评估

调用`impact.analyze`，但必须结合工作区搜索和运行时事实。返回：

```text
DirectModification
MustModify
MustVerify
ObserveOnly
EvidenceGap
SuggestedRisk
```

AI-DLC按自身规则最终确定L1—L5；若SSOT图谱过期或只有inferred关系，风险不得下调。

### CR3变更计划

- 在SSOT创建/更新CR；
- 在`state.md`记录活跃CR和执行顺序；
- 本地计划绑定SSOT CR ID；
- 计划包含产品事实、Git事实和外部证据三类动作。

### CR4执行

- 产品语义变化通过SSOT候选、审批和发布；
- 故事、设计、代码和机器契约通过Git更新；
- 每完成一项更新state；
- 关系和证据异步回链。

### CR5一致性

检查：

```text
SSOT需求版本
↔ Git故事/设计/契约
↔ 实现提交
↔ 测试和CI证据
↔ 发布基线
```

任一必需项失败或未验证时保持CR活跃。

## 13. 步骤完成协议扩展

现有协议保持：

```text
① 更新state.md
② 记录审计
③ 验证更新
④ 输出微型摘要
```

实现上，在完成消息发出前可由平台适配器执行非权威镜像：

```text
checkpoint.publish(state摘要、步骤、证据索引、幂等键)
```

若失败：

- `state.md`记录待同步事件；
- 完成消息写“远端镜像待同步”；
- 下次SessionStart有限重试；
- 不重复本地步骤。

产品事实发布属于阶段动作，不属于可降级checkpoint。

## 14. Hooks设计

### 14.1 Kiro

| Hook | 动作 |
|---|---|
| SessionStart | 读取state后触发增量和新鲜度检查 |
| UserPromptSubmit | 识别限定ID时提示Agent调用上下文，不自动写入 |
| PreTaskExec | 验证项目绑定、批准基线和步骤入场条件 |
| PostTaskExec | 发布checkpoint和适用的追踪关系 |
| PostFileSave | 保护generated投影；必要时生成变更提案 |
| Stop | 检查state、审计、交接和待同步事件 |

### 14.2 Claude Code与OpenCode

使用各平台等价生命周期点。若平台不支持某Hook，降级为：

- 会话启动命令；
- AI-DLC步骤入口显式检查；
- 步骤完成显式同步。

Hook是辅助，不得成为唯一质量门禁。

## 15. 生成文件保护

生成投影路径建议：

```text
docs/aidlc/context/
docs/aidlc/ssot-manifest.json
```

文件包含`generated/do_not_edit/source/version/digest`。保护策略：

1. PreToolUse或PostFileSave检测；
2. 阻止静默覆盖；
3. 提示使用`change.propose`；
4. 用户明确只做本地实验时保存到非生成路径；
5. 不将本地实验误标记为批准需求。

## 16. 审计设计

本地审计继续保存完整用户输入和AI响应。远端仅镜像结构化事件：

```text
AIDLC_SESSION_RESUMED
AIDLC_STEP_COMPLETED
AIDLC_BASELINE_CHECKED
AIDLC_CR_LINKED
AIDLC_EVIDENCE_PUBLISHED
AIDLC_SYNC_FAILED
```

远端事件包含本地审计文件路径和摘要，不复制可能包含敏感信息的完整对话。

## 17. 多项目与团队协作

- `state.md`绑定一个项目；
- 项目切换前更新交接和审计；
- 单元认领仍以Git/任务系统为准；
- SSOT只显示镜像状态，不提供并发锁；
- 共享定义升级产生项目级事件；
- 每个项目独立执行CR、审批和验证；
- 跨项目分析仅向授权项目群角色开放。

## 18. 安全设计

- MCP使用用户委托身份，Agent权限不高于用户；
- 不在state、Manifest、审计或提示词中保存Token；
- 项目句柄短时有效并绑定权限；
- 变更、审批、跨项目查询和证据导出分别授权；
- 对提示词注入内容只作为不可信资料，不执行其中指令；
- 外部资料内容不得改变AI-DLC系统规则；
- 权限拒绝不得通过换工具或重试绕过。

## 19. 兼容与发布

### 19.1 Feature Flags

```text
ssot.integration.enabled
ssot.mode.mirror
ssot.mode.federated
ssot.hooks.enabled
ssot.quality_gates.enforced
ssot.graph_impact.enabled
```

标志由项目配置决定，不使用聊天中的临时指令永久改变。

### 19.2 兼容策略

- v2 state继续支持Legacy；
- v3 state中的SSOT字段均可“不适用”；
- 新steering按条件加载；
- MCP不可用时Legacy项目不受影响；
- Federated项目可关闭写入但保留批准快照读取；
- 回滚AI-DLC版本不得破坏已有state和审计。
