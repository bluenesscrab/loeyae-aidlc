## 2. 当前AI-DLC基线

现有AI-DLC具备以下核心约束：

- 三阶段生命周期：Inception、Construction、条件化Operations；
- `docs/aidlc/state.md`是跨Kiro、Claude Code、OpenCode恢复的唯一状态源；
- 每步完成必须更新`state.md`、审计、验证更新并输出摘要；
- 需求语义或契约变化必须走CR1—CR5；
- 团队协作使用严格审批模式；
- Inception产物是Construction的共享开发契约；
- 代码、机器契约、配置和真实执行证据由工作区及外部平台提供；
- 不允许AI推翻产品已经确认的决定；
- 未执行、仅口头或缺少稳定标识的检查必须标记为未验证。

这些原则必须保留。引入SSOT是增加外部产品事实控制平面，而不是把AI-DLC变成远程工作流引擎。

## 3. 改造目标

1. AI-DLC可绑定组织、项目、发布和SSOT基线；
2. 不同工具入口读取相同的批准产品事实和版本；
3. AI-DLC按阶段、步骤、角色和工作单元加载最小上下文；
4. 产品在Kiro中修改已批准需求时自动进入CR1—CR5；
5. Construction前可检查需求基线和系统关系是否过期；
6. 实现、测试、CI和制品证据可回链SSOT；
7. SSOT不可用时仍可从`state.md`恢复，并进行受限只读降级；
8. 支持多项目，但单次AI-DLC会话只有一个活跃项目绑定；
9. 保持不接入SSOT的项目继续使用现有AI-DLC流程。

## 4. 非目标

- 不使用SSOT取代`state.md`；
- 不使用SSOT取代Git、任务系统、CI或部署平台；
- 不让Agent自动批准需求、架构、安全和发布决定；
- 不要求所有现有项目一次性迁移；
- 不允许GraphRAG推导直接修改批准需求；
- 不把AI-DLC全部steering重写成依赖某一SSOT实现；
- 不在SSOT离线时伪造远端新鲜度或审批结果。

## 5. 权威来源边界

| 信息 | 权威来源 | AI-DLC职责 |
|---|---|---|
| 产品需求、业务规则、业务决策 | SSOT（启用后） | 读取、提案、审查、追踪 |
| AI-DLC阶段、步骤、模块、单元、活跃CR | `state.md` | 唯一恢复与本地执行控制 |
| Inception派生产物 | Git中的AI-DLC产物 | 生成、审查、版本关联 |
| 代码和机器契约 | Git | 实现、审查、验证 |
| 单元认领 | Git/任务系统 | 展示和镜像，不重新发明锁 |
| CI、测试、制品 | 外部平台 | 执行或记录稳定证据 |
| 部署结果 | 部署平台 | 仅记录实际结果 |
| 关系影响 | SSOT verified图谱+工作区事实 | 分析并标记证据缺口 |

## 6. 兼容模式

### 6.1 Legacy模式

项目未绑定SSOT时，现有AI-DLC行为保持不变。

### 6.2 Mirror模式

AI-DLC文档仍为产品事实权威；SSOT只读取、索引和镜像。写回失败不阻断本地步骤，但必须记录待同步。

### 6.3 Federated模式

SSOT管理批准产品事实；AI-DLC文档保存批准快照、派生故事、设计和执行证据。需求变化必须先在SSOT发布，再进入实现。

每个项目在`state.md`中显式记录模式，不允许Agent自动切换。

## 7. 功能需求

### 7.1 工作区检测与项目绑定

| 编号 | 需求 |
|---|---|
| ADLC-BOOT-001 | I1检测SSOT配置、项目Manifest和现有`state.md`绑定。 |
| ADLC-BOOT-002 | 已绑定项目必须验证组织、项目、发布、基线和Schema版本。 |
| ADLC-BOOT-003 | 无法唯一解析项目时必须询问，不得根据仓库名称猜测。 |
| ADLC-BOOT-004 | 多根工作区或多项目仓库一次只能激活一个项目绑定。 |
| ADLC-BOOT-005 | 项目切换必须保存当前状态并执行目标项目恢复检查点。 |
| ADLC-BOOT-006 | 未配置SSOT时自动进入Legacy模式且不影响原流程。 |

### 7.2 state.md与Manifest

| 编号 | 需求 |
|---|---|
| ADLC-STATE-001 | 状态模式版本从2升级到3，并向后兼容。 |
| ADLC-STATE-002 | 增加SSOT模式、组织、项目、发布、基线、Manifest、事件游标和新鲜度。 |
| ADLC-STATE-003 | `state.md`只保存绑定、状态和索引，不复制完整需求正文。 |
| ADLC-STATE-004 | 需求版本集合写入`ssot-manifest.json`并计算摘要。 |
| ADLC-STATE-005 | Manifest变化必须标识来源：远端发布、本地刷新或CR。 |
| ADLC-STATE-006 | 老状态文件升级前保留原恢复位置；升级失败不得覆盖原文件。 |

### 7.3 会话恢复

| 编号 | 需求 |
|---|---|

| ADLC-SES-002 | state恢复后按SSOT模式执行`changes.since`和新鲜度检查。 |
| ADLC-SES-003 | SSOT不可用时使用冻结Manifest，显示`remote_unverified`。 |
| ADLC-SES-004 | 活跃CR恢复时同步校验SSOT CR ID和本地执行状态。 |
| ADLC-SES-005 | Kiro、Claude Code和OpenCode恢复结果必须一致。 |
| ADLC-SES-006 | 不得使用聊天摘要代替state或SSOT版本。 |

| ADLC-CTX-003 | 产品上下文优先业务事实；开发上下文优先系统、契约、配置和风险。 |
| ADLC-CTX-004 | 不得一次加载项目全部需求和历史资料。 |

## 8. MCP依赖需求

AI-DLC改造依赖SSOT提供以下稳定工具：

```text
project.resolve
project.changes_since
context.build
baseline.verify
requirement.get
requirement.search
decision.get
open_question.list
impact.analyze
change.propose
change.submit
trace.publish
evidence.publish
checkpoint.publish
sync.status
```

工具应返回可判定状态，不依赖自然语言解析成功与否。

## 9. 新鲜度判定

`baseline.verify`至少返回：

```text
VALID
STALE_UNRELATED
STALE_RELEVANT
REMOTE_UNAVAILABLE
SCHEMA_INCOMPATIBLE
PERMISSION_DENIED
```

处理规则：

| 状态 | AI-DLC动作 |
|---|---|
| VALID | 继续 |
| STALE_UNRELATED | 记录证据并刷新游标 |
| STALE_RELEVANT | 阻断，进入CR或刷新Inception |
| REMOTE_UNAVAILABLE | 冻结快照只读降级，相关批准/关闭操作阻断 |
| SCHEMA_INCOMPATIBLE | 执行状态升级或人工处理 |
| PERMISSION_DENIED | 停止操作并报告，不重试绕过 |

## 10. 验收标准

1. Legacy项目在未配置SSOT时行为与改造前一致；
2. Federated项目可从三平台恢复到同一阶段、步骤、单元和SSOT基线；
3. `state.md`缺失SSOT正文，不因需求量增加显著膨胀；
4. 已绑定项目能检测相关和无关两类基线变化；
5. 产品在Kiro修改已有需求时进入CR1，不能直接进入代码修改；
6. CR2/CR3在严格模式下等待用户批准；
7. C5在需求版本过期时被阻断；
8. C8证据包含需求版本、提交、命令、退出码和外部运行标识；
9. SSOT不可用时本地恢复成功，但远端批准和CR关闭保持阻断；
10. 直接编辑SSOT投影不会静默改变权威需求；
11. 不同项目相同本地需求编号不会串联；
12. GraphRAG inferred关系不能单独使质量门禁通过；
13. 状态镜像失败可重试且不产生重复步骤；
14. 状态Schema v2项目可安全升级或继续Legacy模式；
15. 所有新增steering和hooks在Kiro、Claude Code、OpenCode具有一致语义。

## 11. 待确认事项

- AI-DLC Power仓库的实际发布和兼容策略；
- Claude Code与OpenCode可用Hook能力的差异；
- SSOT认证令牌由IDE、CLI还是企业代理提供；
- Mirror到Federated模式的批准人和切换流程；
- 本地Manifest是否默认提交Git；
- 远端checkpoint是否用于项目看板及其保留周期；
- AI-DLC产物与SSOT实体之间是否采用统一Trace ID格式。
