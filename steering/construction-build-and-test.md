# 构建和测试

**目的**：构建所有单元并执行全面的测试策略

## 前置条件
- 所有单元的代码生成必须完成
- 所有代码产物必须已生成
- 项目已准备好进行构建和测试

---

## 步骤 1：分析测试需求

> **测试执行策略**：参见 `common-test-execution-strategy.md` 的分层模型和决策规则。

分析项目以确定适当的测试策略：

### 1.1 确定测试执行层级

1. 识别本次 Construction 涉及的模块清单
2. 分析模块间依赖关系
3. 根据决策规则确定执行层级（L3 或 L4）：
   - 单模块项目 → L4（等同于全量）
   - 多模块 + 变更涉及 1-2 个模块 → L3（变更模块 + 直接依赖方）
   - 多模块 + 变更涉及公共模块 → L4 分批执行
   - 多模块 + 变更涉及 3+ 个模块 → L4 分批执行
4. 如果选择 L3，记录跳过的模块及理由

### 1.2 确定测试类型

- **单元测试**：在代码生成期间已按单元生成
- **集成测试**：测试单元/服务间的交互
- **性能测试**：负载、压力和可扩展性测试（如有 NFR 指标）
- **端到端测试**：完整用户工作流
- **契约测试**：服务间 API 契约验证（微服务）
- **安全测试**：漏洞扫描、渗透测试

### 1.3 制定分批计划（如需 L4）

若需全量测试且模块数 > 3：
- 按依赖层次分批（核心/公共模块优先）
- 每批不超过 3 个模块
- 记录分批计划到测试摘要

> **加载模板**：此时加载 `construction-build-and-test-templates.md` 获取各文档的输出格式模板。

按以下顺序生成文档到 `docs/aidlc/construction/build-and-test/`：

| # | 文件 | 条件 | 内容 |
|---|------|------|------|
| 2 | `build-instructions.md` | 始终 | 后端+前端构建命令、环境配置、故障排除 |
| 3 | `unit-test-instructions.md` | 始终 | 执行命令、预期结果、修复流程 |
| 4 | `integration-test-instructions.md` | 有多单元交互 | 测试场景、环境设置、运行命令 |
| 5 | `performance-test-instructions.md` | 有 NFR 性能指标 | 性能需求、测试工具、分析方法 |
| 6 | 额外测试文档 | 按需 | 契约测试、安全测试、E2E 测试 |

## 验证纪律（强制）

- **每个"通过"声明必须附带实际命令输出作为证据**
- 禁止"应该通过"、"看起来没问题"等未经验证的声明
- 流程：运行命令 → 阅读输出 → 确认结果 → 然后才能声明
- 违反 = 未验证的声明是谎言
