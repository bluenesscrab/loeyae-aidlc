# UI Mock 推导原则（分层推导版）

本文件规定 AI 在生成 UI Mock 时必须遵循的**内容决策原则**。UI Mock 中不同层次的内容面向不同受众，需用不同的推导思路。

---

## 前置规则：局部改动必须还原现有页面（优先级最高）

**适用条件**：当 mock-box 标记为「局部改动」类型时，以下规则强制执行，优先于三层分离和其他所有规则。

### 强制流程

1. **读取现有代码**：找到改造基础对应的代码文件（路由文件/页面组件/模板），读取其完整结构
2. **还原现有页面**：在 mock-box 中**先画出现有页面的完整布局**——包括所有已有的字段、组件、按钮、列表结构、tab等
3. **标注增量改动**：在已还原的完整页面上，用橙色边框 + "新增" 标签标注本次增量修改的部分
4. **非增量部分保持原样**：已有的元素必须保留、位置不变、样式不改

### 判定标准

如果某个 mock-box 打开后，去掉所有橙色「新增」标记后的内容，应该和现有系统的页面**高度一致**。如果看起来是一个全新的页面，那就违反了此规则。

### 为什么这条规则优先级最高

UI Mock 的核心价值之一是让开发团队知道「在哪里改、改什么」。如果"局部改动"的 mock 画得和原页面完全不同，开发无法将 mock 中的元素对应到现有代码中的组件，mock 失去了作为改动指南的作用。

### 常见违规

- ❌ 标记为"局部改动"但整个页面都是重新画的，与现有系统页面布局完全不同
- ❌ 只画了改动的部分，省略了页面中未改动的区域
- ❌ 未读取代码就凭想象画了一个"应该长这样"的页面
- ✅ 先读代码还原完整页面 → 在真实布局上用橙色边框标注新增部分
- ✅ 如果读不到代码（文件不存在/无法访问），明确告知用户并降级为"新增页面"类型

### 执行时的现实约束

如果现有页面的完整代码太复杂（如100+字段的大表单），可以：
- 只还原与改动相关的区域（如订单列表的搜索栏+表格部分）
- 用注释标明省略的区域：`<!-- 此处省略：与本次改动无关的XXX区域 -->`
- 但**绝不能**用一个和原页面不同的布局来替代

## 核心架构：三层分离

UI Mock 的内容分为三个层次，各自有独立的推导方向：

| 层次 | 包含什么 | 面向谁 | 推导方向 |
|------|----------|--------|----------|
| **第一层：UI 内容本体** | 列标题、按钮文案、条件表 summary/表头、页面结构组织、标签文案 | 最终用户（运营人员/商家/C端用户） | 用户正向推导 |
| **第二层：条件表内容** | 条件表单元格中的状态描述和差异说明 | 开发人员 + 需求验证 | 表头用用户语言，单元格用系统精确描述 |
| **第三层：底部说明** | mock-box-desc 区域 | 需求验证 + 开发人员 | 推导说明 + 业务规则补充 |

**禁止混淆层次**：不要在 UI 本体中使用系统术语，也不要在条件表单元格中使用模糊的自然语言。

---

## 第一层：UI 内容本体（面向用户）

### 推导方向

从「用户在此视图中想完成什么任务、需要看到什么信息、需要做什么操作」正向推导。**禁止**从系统内部定义反向映射。

### 推导步骤

生成每个 UI 元素前按顺序执行：

1. **确定用户角色**：这个页面的用户是谁？（运营人员？商家？消费者？）
2. **确定核心任务**：用户打开这个页面想完成什么？
3. **确定信息需求**：用户需要看到什么才能完成任务/做出判断？
4. **确定操作需求**：用户在什么情境下需要执行什么动作？
5. **自检**：这个元素去掉后，用户的任务是否受阻？如果不受阻，它不该存在。

### 具体规则

#### 规则 1.1：页面头部必须声明用户和任务

```html
<!-- ✅ 正确 -->
<p>用户：平台运营人员 | 核心任务：创建和管理联票规则、监控联票订单核销情况</p>

<!-- ❌ 错误 -->
<p>涉及页面：联票规则列表、联票规则新增/编辑弹窗...共 6 个 mock-box</p>
```

#### 规则 1.2：列标题只表达用户认知

判定标准：去掉某个标注/后缀后，用户仍能正确理解和操作 → 它是内部逻辑泄漏，应去掉。

```
✅ "参与景区"        → 用户理解为"这个规则覆盖哪些景区"
❌ "参与景区（核销范围）" → 泄漏了系统内部用这个字段控制核销

✅ "本店核销" / "本店分账" → 商家关心的是"跟我有什么关系"
❌ "核销范围" / "分账配置"  → 系统功能模块名
```

#### 规则 1.3：条件表的 summary 和表头用用户语言

```html
<!-- ✅ 用户视角 -->
<summary>📋 操作可用性（取决于规则是否已有商家在用）</summary>
<th>我想做的事</th><th>规则没人用过</th><th>规则已有商家在用</th>

<!-- ❌ 系统视角 -->
<summary>📋 状态条件表（本页面涉及 3 种状态，基准状态：启用+未使用）</summary>
<th>区域/组件</th><th>启用 + 未使用（基准）</th><th>启用 + 使用中</th>
```

#### 规则 1.4：列表信息不完整时，所有状态都需要 [查看]

推导逻辑：[查看] 的存在条件是「信息不对称」，与业务状态无关。

判定：列表展示字段 < 实体完整字段 → 所有状态都需要 [查看]。

#### 规则 1.5：mock-box 按用户任务流组织

以用户故事为驱动单元组织 mock-box，而非按系统功能模块罗列。

```html
<!-- ✅ 任务驱动 -->
<!-- ========== 任务1：管理联票规则（US-01） ========== -->
<!-- 用户想做什么：查看已有规则、创建新规则、停用/启用规则 -->

<!-- ❌ 模块驱动 -->
<!-- ==================== 页面1：联票规则列表 ==================== -->
```

#### 规则 1.6：UI 元素最小化原则——不画需求未要求的、不画用户不需要看到的

**核心原则**：UI Mock 只画用户完成任务所必需的元素。如果一个元素去掉后不影响用户完成操作，就不应该存在。

**三种常见违规模式**：

**模式 A：强化约束可见性**

需求只说了一个约束，AI 自行增加额外 UI 元素来"展示"这个约束。

| 场景 | AI 错误做法 | 正确做法 |
|------|------------|---------|
| 比例合计必须=100% | 表格底部加「合计：100% ✓」实时显示 | 不显示，提交时校验反馈 |
| 某行不可删除 | 操作列显示「固定」文字 | 不显示操作按钮（无按钮=不可操作） |
| 字段只读 | disabled + 灰色 + 锁图标 + tooltip | disabled 状态即可 |

**模式 B：展示已排除/不可选的选项**

选择器中已选/不可选的项目，AI 倾向于保留在列表中并标记为 disabled。但如果用户不需要"确认已选了哪些"，直接过滤掉比 disabled 展示更简洁。

| 场景 | AI 错误做法 | 正确做法 |
|------|------------|---------|
| 添加景区（已添加的不能重复选） | 已添加的项显示在列表中、checked + disabled + 高亮背景 + 「已添加」标记 | 已添加的项不出现在待选列表中 |
| 选择用户（已分配的不能重复选） | 列表展示所有人，已分配的灰色不可点 | 只展示未分配的人 |
| 绑定设备（已绑定的） | 全量列表 + 已绑定标记 | 只展示未绑定的设备 |

**判定方法**：
1. 用户打开这个选择器的目的是什么？→ 选择**新的**要添加的项
2. 用户需要在选择器中看到已选的项吗？→ 不需要（父页面已经展示了已选列表）
3. 去掉已选项后，用户能否顺利完成选择操作？→ 能
4. 结论：已选项不应出现在选择器中

**模式 C：模仿组件库模式而非推导最优交互**

AI 倾向于套用见过的常见 UI 模式（如穿梭框 Transfer），而不是从场景推导。判断标准：如果一个更简单的交互能完成同样的任务，就用更简单的。

**底层逻辑**：
- 「约束存在」≠「约束需要被可视化」
- 「不可操作」最简表达 = 不出现操作控件
- 「不可选」最简表达 = 不出现在可选列表中
- 校验反馈默认位置 = 提交时
- 每多一个 UI 元素 = 多一份认知负担

## 第二层：条件表内容（面向开发 + 验证）

### 推导方向

条件表是 UI 本体和系统实现之间的桥梁。它的**表头**面向用户（让人读懂），它的**单元格内容**面向开发（让人实现）。

### 具体规则

#### 规则 2.1：单元格内容必须包含系统精确描述

单元格中需要体现：
- 精确的状态判断条件（开发能直接写 if 语句的）
- 具体的组件行为（显示/隐藏/disabled/替换）
- 按钮的完整列表（不用"✅ 可以"代替具体按钮名）

```html
<!-- ✅ 正确：精确的单元格内容 -->
<td>
  [查看] [停用]<br>
  <span>编辑和删除 disabled（判断条件：存在关联的在架商品 OR 已售订单且 status ≠ cancelled）</span>
</td>

<!-- ⚠️ 不够精确 -->
<td>❌ 不可以（避免影响已售出的联票）</td>
```

#### 规则 2.2：状态列标题用用户语言，但需要括号标注系统枚举

```html
<!-- ✅ 兼顾可读性和精确性 -->
<th>规则没人用过<br><code style="font-size:10px">inUse=false</code></th>
<th>规则已有商家在用<br><code style="font-size:10px">inUse=true</code></th>

<!-- 或者更简洁的方式 -->
<th>没人用过（inUse=false）</th>
<th>已有商家在用（inUse=true）</th>
```

#### 规则 2.3：多状态页面必须生成条件表

判定标准：同一 UI 位置 × ≥2 种状态导致 UI 差异 → 必须在 mock-box 顶部生成条件表。没有例外。

条件表内**禁止省略**的内容：
- 操作列按钮在每种状态下的完整组合
- 状态之间的判断边界条件
- 与其他状态相比「被隐藏/disabled」的元素

---

## 第三层：底部说明（面向验证 + 开发）

### 推导方向

底部说明是「非可视化信息」的补充区域，分两个独立区块：

### 区块 A：推导说明（新）

解释 UI 设计决策的推导逻辑。让审核者能判断「这样设计是否合理」。

必须包含：
- 列/字段的选择逻辑（为什么展示这些列而非其他）
- 关键设计决策（为什么用弹窗而非跳转、为什么省略某列）
- 用户心智模型假设（用户看到这个页面时怎么想的）

```html
<h4>推导说明</h4>
<ul>
  <li><b>列的选择逻辑</b>：商家选规则时最关心"跟我有什么关系"，所以用"本店核销""本店分账"而非规则的技术属性</li>
  <li><b>去掉"状态"列</b>：只展示启用的规则给商家选，状态列多余</li>
</ul>
```

### 区块 B：业务规则补充（保留旧版精华）

补充 mock 中无法可视化展示的业务规则、边界条件、数值约束。

必须包含：
- 具体的计算规则（如手续费分摊规则）
- 边界判断条件（如"不含已取消订单"）
- 校验逻辑和错误处理
- 状态变更后的连锁影响
- 改动标注（如果是对现有页面的局部改造）

```html
<h4>业务规则</h4>
<ul>
  <li><b>手续费分摊</b>：单一承担方=该方扣除全部手续费；多个承担方=均摊；除不尽保留两位小数向上取整</li>
  <li><b>"使用中"判定</b>：存在关联的在架商品 OR 已售订单（status ≠ cancelled）</li>
  <li><b>停用后效果</b>：商家端规则选择器不显示；已创建商品不受影响；草稿商品不允许上架</li>
  <li><b>比例校验</b>：合计≠100% → [确认]按钮 disabled + 红色提示</li>
</ul>
```

## 自检清单

生成 UI Mock 后，逐项确认：

### 第一层自检
- [ ] 页面头部声明了用户角色和核心任务
- [ ] 所有列标题/按钮文案无系统内部术语泄漏
- [ ] 条件表 summary 和表头使用用户语言
- [ ] 列表信息不完整时，所有状态都有 [查看]
- [ ] mock-box 按用户任务流组织
- [ ] 没有自行添加需求未要求的 UI 元素（实时计算、状态标记、额外图标等）
- [ ] 不可操作的行/字段通过「不展示操作控件」表达，而非显示「固定/锁定」标记
- [ ] 选择器/添加弹窗中，已选/已添加的项不出现在待选列表中（除非需求明确要求穿梭框模式）

### 第二层自检
- [ ] 条件表单元格包含精确的状态判断条件
- [ ] 条件表覆盖了操作列在每种状态下的完整按钮组合
- [ ] 状态列标题标注了系统枚举值
- [ ] 没有状态差异被遗漏在条件表之外

### 第三层自检
- [ ] 底部说明包含「推导说明」区块
- [ ] 底部说明包含「业务规则」区块
- [ ] 业务规则覆盖了具体计算/校验/边界/连锁影响
- [ ] 推导说明解释了关键设计决策的"为什么"

## 完整性自检（生成后必须执行）

以下检查在每个 mock-box 生成完毕后逐项执行：

### 检查1：操作路径闭环

每一个可点击的按钮/入口，追问：
1. 点击后展示什么？→ 必须有对应的 mock（弹窗/页面/tab）
2. 操作完成后结果在哪里看？→ 必须有结果展示页面
3. 操作失败时展示什么？→ 必须有失败状态描述

**常见遗漏**：
- ❌ 画了[选择]按钮，没画选择弹窗
- ❌ 画了[退款]操作，没画退款后的结果查看页
- ❌ 画了tab标签，只做了其中一个tab的内容

### 检查2：确认类操作的信息量

任何需要用户「确认」才执行的操作（核销确认、退款确认、删除确认），弹窗中必须展示足够的上下文信息让用户判断「我操作的是对的目标」：
- 谁的订单？（用户名/手机号）
- 什么内容？（商品名/金额/数量）
- 当前状态？（已核销几个/有效期/是否已退过款）

**判定标准**：如果操作不可逆（核销/退款/删除），信息量必须足以让用户在不翻看其他页面的情况下做出判断。

### 检查3：表单联动完整性

所有 radio/select/switch 组件，每一个选项对应的子组件/字段变化都必须画出：
- radio 有 N 个选项 → 画 N 种状态的子组件
- select 选择后触发联动 → 画联动前后两种状态
- 选择器弹窗 → 确认数据源过滤规则（已选项是否需要排除）
- 确认选择模式（单选/多选）

**禁止**：只画一种选项的子组件，其余用文字描述代替。
