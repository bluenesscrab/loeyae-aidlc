# 产品模块划分规则

**职责**：定义产品级业务模块及其与部署服务的映射。服务、模块和工作单元的术语以 `common-terminology.md` 为准。

## 划分原则

1. **按业务能力和语言边界划分**，不按 Controller、Service、DAO 等技术层划分。
2. **明确数据所有权**：业务实体和写入责任归属于一个服务边界；其他边界通过显式契约访问。
3. **区分模块与服务**：模块是逻辑边界，服务是独立部署和数据所有权边界。模块不是默认微服务。
4. **依赖尽量单向**：循环依赖优先通过职责调整、接口反转或事件解耦处理，不通过共享数据库绕过。
5. **共享最小化**：公共包默认只承载无业务语义的技术能力；业务 Entity、数据库模型和领域对象不因多个模块使用而提取共享。
6. **存量边界优先**：定制项目先尊重已有部署、数据和 Owner 边界；需要改变时进入 CR 并提供迁移依据。

## 粒度判断

模块过大信号：职责难以一句话说明、包含多个独立业务语言或跨越多个数据 Owner。模块过小信号：没有独立业务职责、只能作为另一模块的内部实现。

服务拆分或合并属于部署与数据边界决策，不得仅因模块大小自动执行。

## 共享能力归属

### 可进入共享技术模块/包

- 无业务语义的日志、追踪、序列化和通用错误基础设施；
- 经项目确认的安全、通信或测试技术适配；
- 不包含领域状态和业务规则的工具。

### 不得默认共享

- User、Tenant、Order 等业务 Entity 或数据库模型；
- 领域服务、业务校验和状态机；
- 仅因两个消费者使用就提取的 DTO；
- 通过共享数据访问绕过服务契约的代码。

认证、租户、文件、通知等具有业务语义的能力应有明确 Owner；其他服务通过机器契约消费，而不是复制内部模型。

## `modules.md` 产物

```markdown
# 产品模块与服务映射

## 模块清单
| 模块 ID | 名称 | 业务职责 | 服务 ID | 数据 Owner | Owner | 优先级 |
|---------|------|----------|---------|------------|-------|--------|

## 服务映射
| 服务 ID | 部署边界证据 | 所含模块 | 数据存储 | 状态 |
|---------|--------------|----------|----------|------|

## 边界规则
- {功能、数据和新增需求的归属原则}

## 依赖关系
{只展示模块和服务的已确认方向，运行时细节引用 system-baseline}

## 开发/迁移顺序
1. {顺序及契约、数据或部署依据}

## 变更记录

|------|------|------|----|
```

全新项目尚未确定部署边界时，`服务 ID` 标记“待设计”，不得把模块 ID 自动当作服务 ID。存量项目必须引用服务目录证据。

## 跨边界功能

1. 在主导业务模块记录完整用户目标和业务不变量。
2. 各参与服务只负责其数据和行为边界。
3. 交互通过 `product/contracts.md` 索引到权威机器契约。
4. 跨服务场景在 I13 派生集成/故障用例，并在 I14 拆为分别归属各服务的工作单元。
5. 验证范围基于运行时依赖闭包，不只在主导模块测试。

## 变更规则

新增、合并、拆分或重新映射模块时：
1. 进入 CR，更新 `modules.md`；
2. 更新服务目录、运行时依赖和契约消费者状态；
3. 评估数据迁移、配置、版本并存与部署顺序；
4. 更新受影响模块的 Inception 产物和 state.md。

已完成代码是否重构由 CR 风险与批准计划决定，不默认推迟，也不顺手扩大范围。

## 完成标准

- [ ] 模块按业务边界划分且职责单一
- [ ] 模块与服务未混用，服务映射有证据或明确待设计
- [ ] 数据 Owner 唯一且跨边界访问有契约
- [ ] 共享内容不包含默认共享业务 Entity/模型/规则
- [ ] 依赖循环已消除或有批准方案
- [ ] 跨边界流程已进入契约、技术用例和工作单元规划
