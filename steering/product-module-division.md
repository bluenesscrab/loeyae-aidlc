# 模块划分规则

## 概述

本文档定义产品级 Inception 中"模块划分"步骤的规则、问题模板和产出物格式。

---

## 划分原则

### 核心原则

1. **按业务域划分，不按技术层划分**
   - ✅ 用户管理、订单管理、支付
   - ❌ Controller 层、Service 层、DAO 层

2. **每个模块应该能独立交付价值**
   - 模块完成后，用户能感知到一个完整的功能集
   - 不应该出现"模块 A 做完了但什么都用不了"的情况

3. **模块间依赖尽量单向**
   - 避免循环依赖（A 依赖 B，B 又依赖 A）
   - 如果出现循环，考虑提取公共部分到基础模块

4. **基础模块承载所有共享资源**
   - 被 2 个以上模块引用的实体、服务、工具类 → 放入基础模块
   - 认证/授权基础设施 → 基础模块
   - 公共配置和常量 → 基础模块

### 粒度判断

**模块太大的信号**：
- 需求摘要超过 3 句话才能说清楚
- 涉及 5 个以上独立的用户角色
- 预估开发时间超过 2 周（单人）

**模块太小的信号**：
- 只有 1-2 个 API 端点
- 没有独立的数据模型
- 完全依附于另一个模块，无法独立交付

## 基础模块定义

### 什么放入基础模块

| 类型 | 示例 | 判断标准 |
|------|------|---------|
| 公共实体 | User、Tenant、Org | 被 2+ 模块引用 |
| 认证授权 | JWT、RBAC、权限校验 | 所有模块都需要 |
| 公共工具 | 分页、响应封装、异常体系 | 框架级能力 |
| 公共组件 | 文件上传、通知服务 | 被 2+ 模块调用 |
| 基础配置 | 多租户、数据权限 | 全局生效 |

### 什么不放入基础模块

- 只被一个模块使用的实体或服务
- 特定业务逻辑（即使很通用）
- 可以通过接口契约解耦的功能

## 跨模块功能处理规则

### 归属原则

跨模块功能归属于**发起方/主导模块**：

| 场景 | 主导模块 | 原因 |
|------|---------|------|
| 用户下单 | order-management | 订单是流程发起方 |
| 支付回调 | payment | 支付是回调接收方 |
| 用户注册后初始化数据 | user-management | 用户是流程发起方 |

### 处理方式

1. 主导模块的 Inception 中定义完整业务流程
2. 被调用模块只需在 contracts.md 中暴露接口
3. 主导模块的用户故事中标注"依赖模块 X 的接口 Y"
4. 集成测试在主导模块的 build-and-test 中覆盖

## 渐进式划分规则

### 初始划分

- 不要求一次性列出所有模块
- 至少包含：1 个基础模块 + 当前要开发的业务模块
- 未来可能的模块可以标记为"规划中"（不占用开发资源）

### 新增模块

新增模块时必须：
1. 在 modules.md 中追加（标记添加时间）
2. 更新 contracts.md（定义与现有模块的接口）
3. 更新 state.md 中的模块进度总览
4. 检查是否引入循环依赖

### 模块合并/拆分

如果开发过程中发现模块划分不合理：
1. 在产品级产出物中记录变更原因
2. 更新 modules.md 和 contracts.md
3. 受影响模块的 Inception 产出物需要相应调整
4. 已完成的 Construction 代码不需要立即重构（在下一个迭代处理）
