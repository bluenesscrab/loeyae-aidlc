# 测试执行策略

**目的**：定义增量测试执行策略，根据变更范围智能选择测试执行粒度，避免全量测试导致的 OOM 和效率问题。

**加载时机**：TDD 验证阶段、构建和测试阶段。

---

## 核心原则

1. **变更范围决定测试范围** — 修改了什么，就测什么及其依赖方
2. **分层递进** — 从最小范围逐步扩大，而非一步到全量
3. **资源感知** — 考虑项目规模和可用资源，避免 OOM
4. **工具无关** — 策略适用于任何构建工具（Maven、Gradle、pnpm、yarn 等）

---

## 测试执行分层模型

```
┌─────────────────────────────────────────────────┐
│ L1：焦点测试（当前修改的类/文件对应的测试）      │  ← TDD RED/GREEN
├─────────────────────────────────────────────────┤
│ L2：模块测试（当前修改所在模块的全部测试）       │  ← TDD 无回归验证
├─────────────────────────────────────────────────┤
│ L3：影响域测试（当前模块 + 直接依赖当前模块的模块）│  ← 单元完成后验证
├─────────────────────────────────────────────────┤
│ L4：全量测试（所有模块）                         │  ← 构建和测试阶段 / CI
└─────────────────────────────────────────────────┘
```

---

## 各场景适用层级

| 场景 | 适用层级 | 说明 |
|------|----------|------|
| TDD RED 验证 | L1 | 只运行当前编写的测试方法/文件 |
| TDD GREEN 验证 | L1 + L2 | 当前测试通过 + 模块内无回归 |
| TDD REFACTOR 验证 | L2 | 重构后模块内全部通过 |
| 快速模式无回归 | L2 | 当前模块测试全部通过即可 |
| 单元（Unit）完成后 | L3 | 验证跨模块影响 |
| 构建和测试阶段 | L3 或 L4 | 根据项目规模选择（见下方决策规则） |
| CI/CD 流水线 | L4 | 全量，但可分批并行 |

---

## 构建和测试阶段的决策规则

根据项目特征决定最终测试层级：

```
如果 项目是单模块：
    执行 L4（等同于 L2，全量 = 单模块全部测试）

如果 项目是多模块 且 本次变更仅涉及 1-2 个模块：
    执行 L3（变更模块 + 直接依赖方）

如果 项目是多模块 且 本次变更涉及公共模块（被多模块依赖）：
    执行 L4，但分批执行（见"分批策略"）

如果 项目是多模块 且 本次变更涉及 3+ 个模块：
    执行 L4，但分批执行
```

## 影响域分析方法

### 确定受影响模块

1. **识别变更文件所属模块** — 根据文件路径确定
2. **查找依赖关系** — 分析构建配置中的模块依赖声明
3. **确定直接依赖方** — 哪些模块声明了对当前模块的依赖

### 示例（多构建工具）

**Maven 多模块**：
```bash
# 只构建变更模块及其依赖方（-amd = also-make-dependents）
mvn test -pl changed-module -amd
```

**Gradle 多模块**：
```bash
# 只测试变更模块
./gradlew :changed-module:test

# 变更模块 + 依赖方
./gradlew :changed-module:test :dependent-module:test
```

**pnpm workspace（前端多包）**：
```bash
# 只测试变更包
pnpm --filter changed-package test -- --run

# 变更包 + 依赖方
pnpm --filter changed-package... test -- --run
```

**Turborepo**：
```bash
# 只运行受影响的包的测试
turbo run test --filter=changed-package...
```

## 分批策略（防 OOM）

当必须执行 L4 全量测试时，采用分批执行避免资源耗尽：

### 分批规则

1. **按模块分批** — 每批不超过 3 个模块（可根据实际内存调整）
2. **串行批次** — 批次之间串行，批次内可并行（如资源允许）
3. **优先级排序** — 公共/核心模块先测，业务模块后测
4. **快速失败** — 任一批次失败立即停止，先修复再继续

### 示例

```bash
# Maven 分批（每批 2 个模块）
mvn test -pl core-module,common-module
mvn test -pl service-a,service-b
mvn test -pl service-c,web-module

# pnpm 分批
pnpm --filter core --filter common test -- --run
pnpm --filter service-a --filter service-b test -- --run
```

### JVM 项目内存建议

```bash
# Maven：限制 fork 进程内存
export MAVEN_OPTS="-Xmx512m"
mvn test -pl module-name -DargLine="-Xmx256m"

# Gradle：限制测试 JVM 内存
./gradlew :module:test -Dorg.gradle.jvmargs="-Xmx512m"
```

## 与 TDD 流程的集成

### 标准 TDD 模式

```
RED：
  运行 L1（单个测试方法/文件） → 确认失败

GREEN：
  运行 L1（单个测试方法/文件） → 确认通过
  运行 L2（模块内全部测试） → 确认无回归

REFACTOR：
  运行 L2（模块内全部测试） → 确认仍然绿色
```

### 快速模式

```
写代码 → 写测试 → 运行 L1 确认通过 → 运行 L2 确认无回归
```

### 单元完成

```
所有 TDD 循环完成 → 运行 L3（影响域测试） → 通过后标记单元完成
```

## 与构建和测试阶段的集成

在 `construction-build-and-test.md` 步骤 3（单元测试执行）中：

1. 分析本次 Construction 阶段涉及的模块清单
2. 根据"决策规则"确定执行层级（L3 或 L4）
3. 如果是 L4 且模块数 > 3，采用分批策略
4. 记录每批执行结果
5. 全部通过后汇总到测试摘要

---

## 验证纪律

- **每个层级的"通过"声明必须附带实际命令输出**
- 明确记录测试执行范围（哪些模块、哪些层级）
- 如果选择了 L3 而非 L4，必须说明理由（变更范围分析结果）
- 全量测试的跳过需在测试摘要中标注：`全量测试：跳过（原因：变更仅涉及 X 模块，已通过 L3 验证）`
