# 质量门禁规则

## 强制要求：所有阶段必须通过质量门禁检查

**关键**：在AI-DLC工作流的每个阶段，必须执行相应的质量门禁检查。不满足要求的阶段无法继续。

## 质量门禁级别

### 级别C：强制级别
- **不满足要求则无法继续**：门禁检查失败将阻止工作流进入下一阶段
- **必须修复**：必须解决所有质量问题后才能继续
- **自动验证**：AI自动执行验证，无需人工干预

## 代码实现合规性验证方法

### 需求符合性验证

AI在执行代码生成质量门禁时，必须按以下方法逐项验证：

#### 1. 检查是否有私自扩展的功能
- 对比生成的代码功能点与需求文档/功能设计文档
- 代码中出现的API端点、Service方法、页面路由必须在需求中有对应
- 如发现需求中未提及的功能，标记为"私自扩展"，门禁失败

#### 2. 检查是否有简化的功能
- 对照需求中标记为"高复杂度"的功能，验证实现是否完整
- 检查复杂功能的关键组件是否齐全（如：分页、过滤、排序、导出等）
- 检查错误处理是否完整（需求要求错误处理的功能必须有try-catch）
- 检查输入验证是否完整（需求要求校验的字段必须有验证逻辑）

#### 3. 检查是否完整实现了所有需求
- 逐条对照需求列表，确认每个功能需求都有对应的代码实现
- 确认每个验收标准都能通过代码逻辑满足

### 代码完整性验证

#### 占位符检查
AI必须扫描生成的代码，确认不存在以下模式：
- `TODO`、`FIXME`、`HACK`、`XXX`
- `待实现`、`待完成`、`待补充`、`待优化`
- `// ...`、`/* ... */`（省略号占位）
- 空方法体（仅有注释无实现的方法）
- `throw new UnsupportedOperationException()`（未实现占位）

#### 功能完整性检查
- 所有声明的接口方法都有完整实现
- 所有路由对应的页面组件都已创建
- 所有引用的工具函数/组件都已定义
- 无编译错误或类型错误

## 质量门禁执行流程

### 1. 触发时机
- **阶段结束时**：检查该阶段产出物质量
- **阶段转换时**：确认满足进入下一阶段的条件

### 2. 执行步骤
1. **加载检查清单**：根据当前阶段，从上方对应章节加载检查项
2. **逐项验证**：对照产出物，逐项检查是否满足要求
3. **生成结果**：标记每项的通过/失败状态
4. **决策**：
   - 全部通过 → 允许进入下一阶段
   - 任何一项失败 → 阻止继续，要求修复

### 3. 失败处理
1. **立即停止**：工作流立即停止，无法进入下一阶段
2. **问题记录**：在audit.md中记录具体的质量问题
3. **修复指导**：向用户说明失败原因，提供具体修复建议
4. **重新检查**：修复后重新执行门禁检查，直到全部通过

### 4. 通过标准
- 所有检查项标记为通过
- 无影响功能或安全的重大缺陷
- 产出物与需求/设计文档一致

## 质量门禁审计

### 审计要求
- **必须**：在audit.md中记录每次门禁检查
- **必须**：记录检查时间、检查项、检查结果
- **必须**：记录失败原因和修复建议
- **必须**：记录最终通过时间

### 审计格式
```markdown
## 质量门禁检查
**阶段**: [阶段名称]
**检查时间**: [ISO时间戳]
**检查项数量**: [总数]
**通过数量**: [通过数]
**失败数量**: [失败数]
**失败项**: [列出失败项及原因]
**处理结果**: [通过/阻止]
**修复建议**: [具体建议]
```
