# 质量门禁

**目的**：用与当前步骤、技术栈和已确认需求匹配的证据阻止缺陷流入下一阶段。门禁不适用项必须记录依据，不能伪造通过。

## 通用执行规则

1. 进入门禁时只加载当前阶段检查项。
2. 每项记录 `通过 / 失败 / 不适用 / 未验证` 及证据位置。
3. 任一必需项失败或未验证时阻断；修复后重跑受影响检查。
4. 不得为满足门禁擅自引入工具、改变需求、关闭检查或删除失败测试。
5. 技术栈专属项仅在对应技术栈适用；项目现有规范优先于通用建议。
6. 结果写入 state.md，详细证据写入当前阶段审计或报告。

## Inception 门禁

### 工作区检测

- [ ] state.md 已创建或恢复，项目/协作/架构/审批模式和复杂度已记录
- [ ] 现有代码、技术栈、构建测试入口和 AI-DLC 产物状态已识别
- [ ] 当前意图和下一路由有依据

### 需求与故事

- [ ] 需求范围、边界、约束和验收标准明确且可验证
- [ ] 不确定事项已询问，未用默认假设填补
- [ ] 需求审查通过；存量项目与逆向产物一致
- [ ] 已生成需求文档时，用户故事完整覆盖需求并通过反向审查
- [ ] UI Mock（如执行）与需求、故事一致

深度专属方法（优先级模型、角色画像、数据模型等）仅按 `inception-requirements-analysis.md` 的已选深度检查。

### 规划与设计

- [ ] 工作流计划覆盖全部已批准需求和必要验证
- [ ] 应用设计（如触发）定义边界、接口、数据和关键业务规则
- [ ] 测试用例派生（如触发）覆盖全部 Gherkin 场景且锚点可执行
- [ ] 工作单元（如触发）职责单一、依赖无环或有明确解法、故事无遗漏
- [ ] Construction 入场条件已满足或有经审批的跳过记录

## Construction 门禁

### 单元实现

- [ ] 每项实现可追溯到需求、故事或已批准 CR
- [ ] TDD 存在实际 RED 和 GREEN 证据
- [ ] 无未批准功能、无需求简化、无 TODO/FIXME/空实现
- [ ] 规格合规审查先于代码质量审查，失败项已修复并重审
- [ ] 当前单元影响域测试实际通过
- [ ] 框架规范仅在适用技术栈按对应 steering/MCP 逐项验证

### 通用代码质量与安全

- [ ] 输入、权限、敏感数据、错误处理与日志满足已确认风险和项目规范
- [ ] 接口、数据迁移和调用方保持一致
- [ ] 新增代码沿用项目命名、结构、类型和测试惯例
- [ ] 未修改无关代码，建议性优化已记录而未扩大范围

具体安全检查加载 `common-tech-security.md`；前端、测试、数据库和 Loeyae Boot 检查分别按对应规则加载，不对其他技术栈套用。

### 实际构建和测试

- [ ] 所有必要组件实际构建成功并记录命令、工作目录和退出码
- [ ] 适用 lint、类型、静态和安全检查实际通过
- [ ] 按 L3/L4 策略实际执行测试且 0 失败
- [ ] 条件集成、契约、E2E、性能或迁移测试已执行，或记录不适用依据
- [ ] `build-and-test-summary.md` 与实施报告包含真实、脱敏、可复现证据

任何环境阻塞均标记“未验证”，Construction 不得完全通过。

## Operations 门禁（部署准备）

只检查用户已选择的部署目标：

- [ ] 生成文件与确认的容器、CI/CD、编排和环境目标一致，无多余平台配置
- [ ] 无硬编码密钥、Token、密码、私有绝对路径或未经确认的生产自动发布
- [ ] 所有占位符已替换或登记为部署参数
- [ ] 健康检查、资源、迁移、smoke test 和回滚与需求匹配
- [ ] Docker/Kubernetes/Compose/Jenkins/Nginx 等适用配置已执行可用的语法、渲染、dry-run 或静态验证
- [ ] `deployment-guide.md` 可从 Construction 已验证制品开始执行

工具不可用时标记“未验证”并给出精确命令，不得宣称 Operations 完全通过。

## 结果格式

```markdown
## 质量门禁结果
| 检查项 | 状态 | 证据/依据 |
|--------|------|-----------|
| {检查项} | 通过/失败/不适用/未验证 | {命令、文件或理由} |

结论：通过 / 阻断
```

state.md 至少记录阶段、时间、结论、证据文件和阻断原因。失败后只重跑受影响门禁及其依赖检查。
