# 质量门禁规则

## 强制要求：所有阶段必须通过质量门禁检查

**关键**：在AI-DLC工作流的每个阶段，必须执行相应的质量门禁检查。不满足要求的阶段无法继续。

## 质量门禁级别

### 级别C：强制级别
- **不满足要求则无法继续**：门禁检查失败将阻止工作流进入下一阶段
- **必须修复**：必须解决所有质量问题后才能继续
- **自动验证**：系统自动执行验证，无需人工干预

## 质量门禁执行流程

### 1. 门禁检查触发时机
- **阶段开始时**：检查前置条件
- **阶段结束时**：检查产出物质量
- **阶段转换时**：检查是否满足进入下一阶段的条件

### 2. 门禁检查失败处理
1. **立即停止**：工作流立即停止，无法进入下一阶段
2. **问题记录**：在audit.md中详细记录质量问题
3. **修复指导**：提供具体的修复建议
4. **重新检查**：修复后重新执行门禁检查

### 3. 门禁检查通过标准
- **所有检查项通过**：所有必须检查项都标记为完成
- **无重大缺陷**：无影响功能或安全的重大缺陷
- **质量达标**：满足预定义的质量标准

## 质量门禁集成到工作流

### 核心工作流修改要求
1. **必须**：在每个阶段结束时调用质量门禁检查
2. **必须**：门禁检查通过后才能显示"完成"消息
3. **必须**：门禁检查失败时提供明确的错误信息
4. **必须**：在state.md中记录门禁检查状态

### 质量门禁检查函数
```markdown
## 质量门禁检查流程

1. **加载检查清单**：根据当前阶段加载对应的质量门禁检查项
2. **执行检查**：逐项检查产出物是否符合要求
3. **生成报告**：生成详细的质量检查报告
4. **决策**：根据检查结果决定是否允许继续

- **全部通过**：允许进入下一阶段
- **任何失败**：阻止继续，要求修复
```

## 质量门禁检查工具

### 自动化检查项
- **代码规范检查**：使用linter工具（ESLint、Checkstyle等）
- **安全扫描**：使用安全扫描工具（SonarQube、OWASP ZAP等）
- **测试覆盖率**：使用测试覆盖率工具（JaCoCo、Istanbul等）
- **构建验证**：使用构建工具验证（Maven、Gradle、pnpm等）

### 手动检查项
- **需求完整性**：人工评审需求文档
- **设计合理性**：人工评审架构设计
- **用户体验**：人工评审用户界面设计

## 质量门禁配置

### 可配置参数
- **严格程度**：C（强制级别）
- **检查项**：可根据项目需求调整
- **通过标准**：可根据项目需求调整
- **检查工具**：可根据技术栈调整

### 配置示例
```yaml
quality_gates:
  strictness: "C"  # 强制级别
  phases:
    inception:
      requirements_analysis: true
      user_stories: true
      application_design: true
    construction:
      code_generation: true
      security: true
      testing: true
  tools:
    code_quality: ["eslint", "checkstyle"]
    security: ["sonarqube", "owasp_zap"]
    testing: ["jacoco", "jest"]
```

## 质量门禁最佳实践

### 实施建议
1. **渐进式实施**：从关键检查项开始，逐步增加
2. **团队培训**：确保团队理解质量门禁的重要性
3. **持续改进**：定期评审和优化质量门禁
4. **工具集成**：尽可能自动化质量检查

### 常见问题处理
1. **误报处理**：建立误报反馈机制
2. **例外处理**：建立例外审批流程
3. **性能影响**：监控质量检查对开发效率的影响
4. **工具维护**：定期更新检查工具和规则
