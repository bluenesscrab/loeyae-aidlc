# 运行时依赖分析

**职责**：建立可部署单元、数据存储、外部系统之间的运行时依赖事实，用于逆向工程、变更影响评估、工作单元规划和测试范围计算。构建依赖不等同于运行时依赖。

## 执行条件

满足任一条件时执行：
- 存在两个及以上可独立部署单元；
- 存在跨进程同步调用、异步消息、共享配置、共享数据存储或外部系统接入；
- CR 涉及接口、事件、配置、数据所有权或部署拓扑变化。

单进程且无外部运行时依赖时跳过，并在 `state.md` 记录依据。

## 输入与事实来源

按可信度从高到低读取：
1. 机器可读契约、路由、消息或部署配置；
2. 代码中的客户端、生产者、消费者和数据访问关系；
3. 运行配置与项目已有服务目录；
4. 已确认的设计文档。

冲突时不得自行选择；记录冲突、证据位置和待确认项。技术栈专属识别由检测到的适配规则完成，本文件只定义通用模型。

## 运行时关系模型

每条关系至少记录：

| 字段 | 要求 |
|------|------|
| `source` / `target` | 稳定标识；外部系统必须显式标记 |
| `type` | `sync-api`、`async-message`、`shared-config`、`data-access`、`file-transfer` 或 `other` |
| `contract_ref` | 机器契约、配置或代码证据路径；未知时标记未验证 |
| `direction` | 调用、发布/订阅或读写方向 |
| `criticality` | 关键/重要/一般，必须有业务依据 |
| `failure_behavior` | 超时、重试、降级、幂等或人工补偿的现状 |
| `owners` | 提供方与消费方负责人；未知时列入待确认项 |

禁止仅凭构建模块依赖推断调用方向，也禁止把共享代码包当作可部署服务。

## 产物

在 `<system-baseline-root>/` 创建或更新：

- `service-catalog.md`：可部署单元、职责、代码位置、数据所有权、Owner 和部署边界；
- `runtime-dependencies.md`：运行时关系清单、消费者反向索引、关键链路及证据；
- `external-systems.md`：外部系统、协议、数据方向、环境依赖、失败边界和负责人。

`<system-baseline-root>` 的实际位置由 `common-directory-structure.md` 定义。产物只记录业务项目事实，不复制通用规则。

## 影响域计算

对变更目标执行以下闭包计算：
1. 识别直接修改的服务、契约、配置、数据和外部接入；
2. 沿消费者反向索引找出直接消费者；
3. 对破坏性契约、共享配置、跨服务数据写入继续向下游扩展；
4. 区分“必须修改”“必须验证”“仅观察”三类范围；
5. 将结果写入 CR、执行计划或构建测试矩阵。

影响域必须同时包含构建依赖和运行时依赖；两者冲突时取并集并记录原因。

## 新鲜度与恢复

- 产物记录分析时间、代码版本标识和证据摘要。
- 恢复会话或进入 CR 时，对比相关代码/契约/配置的版本或修改时间。
- 仅局部变化时增量刷新相关节点与反向索引；无法判断是否过期时状态为“待复核”，不得作为完整影响评估依据。
- `state.md` 只记录基线路径、新鲜度和本次受影响节点，不复制完整依赖图。

## 完成标准

- [ ] 所有可部署单元与外部系统有稳定标识和证据位置
- [ ] 同步、异步、配置和数据关系已按适用范围识别
- [ ] 每条关键关系存在消费者反向索引
- [ ] 未知项和冲突未被假设填补
- [ ] 影响域已映射到设计、工作单元或验证范围
- [ ] 基线路径、新鲜度和未验证项已写入 `state.md`
