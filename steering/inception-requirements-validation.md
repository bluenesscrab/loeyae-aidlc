# 5 维度需求验证（标准 + 全面深度）

**加载条件**：标准深度和全面深度时加载本文件。最小深度跳过。

**核心原则**：需求必须经过多维度验证才能进入下一阶段。验证失败则回退修复。

---

## 验证维度

| 维度 | 关注点 | 关键问题 |
|------|--------|----------|
| **真实性** | 真实需求 | 这是真正的用户/业务需求吗？ |
| **完整性** | 覆盖度 | 所有方面都完整指定了吗？ |
| **一致性** | 无矛盾 | 需求之间没有冲突吗？ |
| **可行性** | 可实现 | 技术和经济上可行吗？ |
| **可验证性** | 可测试 | 能客观验证是否满足吗？ |

---

## 标准深度验证（简化版）

重点关注**完整性**和**可验证性**两个维度：

**完整性检查**：
- [ ] 所有功能需求都有明确描述
- [ ] 非功能需求已识别（性能、安全、可用性）
- [ ] 错误场景已考虑
- [ ] 数据流转已描述

**可验证性检查**：
- [ ] 每个需求都有验收标准
- [ ] 验收标准是可量化的（无"快速"、"友好"等模糊词）
- [ ] 可以为每个验收标准设计测试用例

**模糊词扫描**（强制）：

| 类别 | 模糊词 | 量化替代 |
|------|--------|----------|
| 性能 | 快、响应迅速、高效 | "< 2s 响应时间"、"95th 百分位 < 500ms" |
| 易用性 | 友好、简单、直观 | "任务完成率 > 90%"、"< 3 次点击" |
| 质量 | 可靠、稳定、安全 | "99.9% 可用性"、"OWASP Top 10 合规" |
| 规模 | 大量、很多、可扩展 | "10,000 并发用户"、"1TB 存储" |

**执行**：
1. 扫描需求文档中的模糊词
2. 为每个模糊词提供量化替代建议
3. 向用户确认量化标准

## 迭代回退（验证失败时）

**触发条件**：任何维度评分 <80%。

**回退路径**：

| 失败维度 | 回退目标 | 修复方式 |
|----------|----------|----------|
| 真实性 | 多角色澄清 | 向用户确认需求来源和真实性 |
| 完整性（缺失） | 多角色澄清 | 补充缺失的需求规格 |
| 完整性（不清晰） | 多角色澄清 | 澄清模糊的需求描述 |
| 一致性 | 多角色澄清 | 解决矛盾，请用户决策 |
| 可行性 | 架构视角澄清 | 讨论技术约束，调整范围 |
| 可验证性 | 测试视角澄清 | 定义量化验收标准 |

**回退执行**：
1. 向用户展示验证失败的维度和具体问题
2. 说明需要回退到哪个步骤
3. 执行针对性的补充澄清（只问与失败维度相关的问题）
4. 更新需求文档
5. 重新执行验证（仅验证之前失败的维度）

**最大回退次数**：同一维度最多回退 2 次。如果 2 次回退后仍未通过：
- 向用户说明情况
- 记录为已知风险
- 由用户决定是否继续（带风险继续 / 暂停项目）
