# 说服式防护与验证纪律

## 概述

规则告诉你"不许做什么"。说服式防护告诉你"为什么不能做"。

当 Agent 面临时间压力、上下文疲劳或"看起来很简单"的诱惑时，硬规则容易被合理化绕过。说服式防护通过内化原因来阻止跳步——不是因为规则禁止，而是因为你理解跳步的代价。

**核心原则**：违反规则的字面意思就是违反规则的精神。

---

## 铁律

```
没有证据的完成声明是谎言，不是效率。
没有失败测试的代码是猜测，不是实现。
没有根因分析的修复是赌博，不是调试。
跳过的步骤不会消失——它们变成 bug 回来找你。
```

---

## 红旗信号 — 立即停止

如果你发现自己在想以下任何一条，**立即停止并重新评估**：

### 跳步合理化
- "这次跳过没关系"
- "太简单了不需要这个步骤"
- "用户在催，先跳过"
- "我很确定这样就行"
- "先做完再补"

### 完成幻觉
- "应该没问题了"
- "看起来是对的"
- "我很有信心"
- "逻辑上应该通过"
- 任何包含"应该"、"大概"、"可能"的完成声明

### 调试捷径
- "试试改这个看看"
- "快速修一下"
- "先试几个方案"
- "感觉是这里的问题"

**所有这些都意味着：你正在走捷径。停下来，回到流程。**

## 合理化反驳表

| 你的借口 | 现实 |
|----------|------|
| "太简单了不需要测试" | 简单代码也会出 bug。写测试只需 30 秒。 |
| "用户在催，没时间走流程" | 系统化流程比乱试更快。赶工 = 返工。 |
| "这次跳过，下次补" | "下次"永远不会来。技术债从这里开始。 |
| "我已经手动验证过了" | 手动验证是临时的。没有记录，不能重跑，下次改动就失效。 |
| "先写代码再补测试" | 后补的测试立即通过，证明不了任何东西。你永远不知道它是否真的在测试正确的行为。 |
| "改动太小不值得审查" | 小改动引入的 bug 最难发现。审查只需 2 分钟。 |
| "我很确定根因是这个" | 确定 ≠ 证据。跑一下验证只需几秒。 |
| "部分检查就够了" | 部分检查证明不了完整性。跑完整验证。 |
| "Agent 报告成功了" | Agent 报告 ≠ 独立验证。自己确认。 |
| "删掉已写的代码太浪费了" | 沉没成本谬误。保留未经测试验证的代码才是真正的浪费。 |
| "保留作为参考，然后写测试" | 你会"适配"它。那就是后补测试。删除意味着删除。 |
| "TDD 太教条了，我在务实" | TDD 就是务实：比调试更快发现 bug，防止回归，文档化行为。"务实"的捷径 = 在生产环境调试 = 更慢。 |
| "再试一次修复" (已失败 2+ 次) | 3+ 次失败 = 架构问题。停止修补，质疑设计。 |

## 验证纪律

### 完成声明的铁律

```
在声明任何完成状态之前：

1. 识别：什么命令能证明这个声明？
2. 执行：运行完整命令（新鲜的、完整的）
3. 阅读：完整输出，检查退出码，计数失败
4. 验证：输出是否确认了声明？
   - 否：陈述实际状态 + 证据
   - 是：陈述声明 + 证据
5. 然后才能：做出声明

跳过任何步骤 = 说谎，不是验证
```

### 常见声明的验证要求

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| "测试通过" | 测试命令输出：0 失败 | 之前的运行、"应该通过" |
| "构建成功" | 构建命令：exit 0 | lint 通过、日志看起来正常 |
| "Bug 已修复" | 原始症状测试：通过 | 代码改了、假设修好了 |
| "需求已满足" | 逐项检查清单 | 测试通过 |
| "代码质量合格" | 审查通过 + lint 通过 | 自我感觉良好 |

### 禁止的表达

以下表达在未经验证前**绝对禁止使用**：

- ✅ → ❌ 不能在未运行验证前使用
- "完成" → ❌ 不能在未确认所有检查项前使用
- "通过" → ❌ 不能在未看到实际输出前使用
- "没问题" → ❌ 不能在未提供证据前使用
- "Great!"、"Perfect!"、"Done!" → ❌ 不能在验证前表达满意

## 流程遵从的心理防线

### 为什么每个步骤都重要

**需求分析不能跳过**：
- 跳过需求分析 = 解决错误的问题
- 解决错误的问题 = 所有后续工作归零
- 代价：不是"浪费 30 分钟分析"，而是"浪费 3 天实现错误的东西"

**设计不能跳过**：
- 跳过设计 = 边写边想
- 边写边想 = 反复重构
- 代价：不是"浪费 1 小时设计"，而是"浪费 2 天重构 + 引入 bug"

**TDD 不能跳过**：
- 跳过 TDD = 不知道代码是否正确
- 不知道是否正确 = 在生产环境发现 bug
- 代价：不是"浪费 10 分钟写测试"，而是"浪费 4 小时调试 + 用户受影响"

**代码审查不能跳过**：
- 跳过审查 = 盲点不被发现
- 盲点不被发现 = 技术债累积
- 代价：不是"浪费 5 分钟审查"，而是"3 个月后花 2 周还债"

**质量门禁不能跳过**：
- 跳过门禁 = 问题传递到下一阶段
- 问题传递 = 修复成本指数增长
- 代价：需求阶段修复 1 小时，代码阶段修复 1 天，生产阶段修复 1 周

### 时间压力下的正确响应

当感到时间压力时：

```
❌ 错误响应：跳过步骤以"节省时间"
✅ 正确响应：
  1. 告知用户当前进度和预估时间
  2. 询问是否可以缩小范围（而非跳步）
  3. 如果用户坚持跳步，明确告知风险后执行
  4. 记录跳步决策和风险到审计日志
```

---

## 与质量门禁的关系

说服式防护**替换**了质量门禁中的"硬规则"部分：

- ~~"如果任何检查项失败，工作流立即停止"~~ → "如果任何检查项失败，理解为什么失败比强制停止更重要——但你仍然不能继续，直到问题解决"
- ~~"必须修复后才能继续"~~ → "继续意味着把问题传递给下一阶段，修复成本会指数增长。现在修复是最便宜的时机。"

质量门禁的**检查清单本身保留不变**——改变的是执行方式：从"因为规则说不行"变为"因为你理解为什么不行"。

---

## 应用范围

本文件的规则适用于**所有阶段**：

- **Inception**：不跳过需求确认、不假设用户意图、不省略用户故事
- **Construction**：不跳过 TDD、不跳过审查、不跳过验证
- **调试**：不猜测修复、不跳过根因分析
- **任何时候**：不做未经验证的完成声明

---

## 最终规则

```
流程存在是因为它有效。
跳步的诱惑来自对代价的低估。
每次你想跳步时，问自己：
"如果这个决定是错的，代价是什么？"

答案几乎总是：比走完流程贵得多。
```
