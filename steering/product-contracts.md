# 模块间接口契约规范

## 概述

本文档定义产品级 Inception 中"模块间接口契约"步骤的规范，包括契约的定义方式、粒度要求和变更管理。

---

## 契约类型

### 同步接口（Service 调用）

模块间通过 Service 接口直接调用：

```markdown
### [提供方模块] → [消费方模块]

**接口**：`[ServiceName].[methodName]([params]): [returnType]`
**说明**：[一句话描述用途]
**调用时机**：[什么场景下调用]
```

### 异步事件（Event 通知）

模块间通过事件解耦：

```markdown
### [发布方模块] 发布 → [订阅方模块] 订阅

**事件**：`[EventName]`
**触发条件**：[什么时候发布]
**携带数据**：[关键字段，不需要完整定义]
**订阅方行为**：[一句话描述收到后做什么]
```

### 共享数据模型（基础模块提供）

基础模块提供的公共实体和工具：

```markdown
### [基础模块] 提供

**实体**：`[EntityName]` — [一句话说明]
**服务**：`[ServiceName]` — [一句话说明]
**工具**：`[UtilName]` — [一句话说明]
```

## 契约粒度要求

### 必须定义

- 方法/事件名称
- 关键参数（类型 + 一句话说明）
- 返回值类型
- 调用时机/触发条件

### 不需要定义

- 完整的请求/响应字段列表（模块级 Inception 细化）
- 异常处理细节（模块级 Construction 处理）
- 性能要求（模块级 NFR 处理）
- 具体实现方式

### 粒度示例

**✅ 合适的粒度**：
```
UserQueryService.getUser(userId: Long): UserDTO
说明：根据用户 ID 查询用户基本信息（姓名、角色、组织）
```

**❌ 太粗**：
```
UserService：提供用户相关功能
```

**❌ 太细**：
```
UserQueryService.getUser(userId: Long): UserDTO
  - UserDTO.id: Long 用户ID
  - UserDTO.name: String 用户姓名，最大50字符
  - UserDTO.roles: List<RoleDTO> 角色列表
    - RoleDTO.id: Long 角色ID
    - RoleDTO.code: String 角色编码
    ...
```

## 契约规则
- 模块间只通过定义的接口/事件通信
- 禁止直接访问其他模块的数据库表
- 禁止直接引用其他模块的内部类
- 接口变更需要记录到变更日志

## 变更日志
| 时间 | 变更模块 | 变更内容 | 影响模块 | 状态 |
|------|---------|---------|---------|------|
| [ISO时间] | [模块] | [变更描述] | [受影响模块列表] | [已同步/待同步] |
```

## 变更管理

### 变更触发场景

1. **基础模块接口变更**：影响所有依赖模块
2. **业务模块新增接口**：影响消费方模块
3. **业务模块修改接口**：影响消费方模块
4. **新增模块**：需要定义与现有模块的接口

### 变更记录规则

每次接口变更时，在 contracts.md 的"变更日志"中追加记录：

```markdown
| 2026-05-15T10:00:00Z | user-management | UserQueryService.getUser 新增 orgId 参数 | order-management, payment | 待同步 |
```

### 变更同步检查

各模块恢复时，AI 自动执行：
1. 读取 contracts.md 的变更日志
2. 筛选"状态 = 待同步"且"影响模块"包含当前模块的记录
3. 如有未同步变更，提示用户：

```markdown
**⚠️ 检测到接口契约变更：**

| 时间 | 变更 | 影响 |
|------|------|------|
| [时间] | [变更描述] | [对当前模块的影响] |

**建议操作**：
A) 立即处理 — 调整当前模块的设计/代码以适配变更
B) 稍后处理 — 记录待办，继续当前工作
C) 查看详情 — 了解变更的完整上下文

[回答]:
```

4. 用户确认处理后，将该记录状态更新为"已同步"

### 变更影响评估

| 变更类型 | 影响级别 | 处理方式 |
|---------|---------|---------|
| 新增接口 | 低 | 消费方按需使用，不影响现有代码 |
| 新增参数（可选） | 低 | 消费方无需修改 |
| 新增参数（必填） | 高 | 消费方必须适配 |
| 修改返回值 | 高 | 消费方必须适配 |
| 删除接口 | 高 | 消费方必须移除调用 |
| 新增事件 | 低 | 订阅方按需订阅 |
| 修改事件数据 | 中 | 订阅方需要适配 |

## 依赖关系规则

### 允许的依赖方向

```text
基础模块 ← 所有业务模块（单向）
业务模块 A ← 业务模块 B（单向，通过接口）
```

### 禁止的依赖

- ❌ 循环依赖：A → B → A
- ❌ 基础模块依赖业务模块
- ❌ 直接数据库访问（跨模块）
- ❌ 直接引用其他模块的内部实现类

### 循环依赖的解决方案

如果发现潜在循环依赖：

1. **提取公共部分**：将双方都需要的功能提取到基础模块
2. **事件解耦**：将同步调用改为异步事件
3. **接口反转**：让被依赖方定义接口，依赖方实现

---

## 与模块级 Inception 的衔接

### 模块级 Inception 如何使用契约

1. **需求分析**：引用契约中定义的接口作为外部依赖
2. **应用设计**：实现契约中承诺的接口；调用契约中定义的外部接口
3. **单元生成**：接口实现和接口调用可能分属不同单元

### 模块级 Inception 发现契约不足

如果模块级 Inception 过程中发现需要新的跨模块接口：

1. 暂停模块级 Inception
2. 回到产品级，更新 contracts.md
3. 记录变更日志
4. 返回模块级 Inception 继续

### Mock 接口策略

当基础模块的 Construction 尚未完成时，业务模块可以：
- 在代码生成阶段使用接口的 Mock 实现
- Mock 实现放在测试目录（不进入生产代码）
- 基础模块 Construction 完成后，在集成测试阶段验证真实调用
