# 变更请求流程（Change Request）

**目的**：对已完成或已建立基线的功能进行可追溯变更，确保需求、系统基线、实现和验证证据一致。

## 触发条件

满足任一条件即进入 CR：
1. 用户修改、调整或重构已有功能；
2. 需求语义、接口/事件契约、共享配置、数据所有权或部署顺序变化；
3. Construction 发现实现超出或改变产品故事。

实现少于产品故事属于缺陷，回 Construction 补齐；实现多于或不同于产品故事必须走 CR，禁止维护独立“开发版需求”。

## CR1：变更范围定位

1. 读取 `state.md`，确认阶段、活跃模块/服务/单元和系统基线新鲜度。
2. 定位需求、故事、技术用例、设计、工作单元、代码、配置和验证证据。
3. 分布式项目加载 `common-runtime-dependency-analysis.md`，从目标节点获取消费者反向索引。
4. 基线过期或影响节点未知时先增量刷新；不得基于旧依赖图继续评估。

输出目标、当前行为、期望行为、直接修改范围和待确认项。

## CR2：影响评估

沿以下链路评估，按适用范围加载契约、配置和一致性治理规则：

```text
需求/CR → 故事与技术用例 → 设计 → 服务/工作单元 → 契约/配置/数据 → 运行时消费者 → 代码与部署 → 验证证据
```

### 评估维度

| 维度 | 必须识别 |
|------|----------|
| 服务与模块 | 直接修改和仅需验证的边界 |
| 运行时依赖 | 同步、异步、网关、文件和外部系统消费者 |
| 契约 | 权威来源、兼容性和逐消费者状态 |
| 配置 | 作用域、消费者、生效方式、版本组合和回滚 |
| 数据 | Owner、Schema/迁移、跨边界写入和对账 |
| 一致性 | 幂等、重试、补偿、乱序、部分成功和恢复 |
| 页面 | 需求、故事、UI Mock 与实现基线 |
| 交付 | 更新/部署顺序、版本并存、回滚和外部证据 |

### L1-L5 风险分级

| 级别 | 判定 | 最低处理 |
|------|------|----------|
| L1 局部 | 单文件/单组件，不改变外部行为 | 定向实现与验证 |
| L2 单服务 | 单服务行为或内部数据变化，无跨边界兼容影响 | 精简 CR，服务级回归 |
| L3 跨边界兼容 | 多服务/外部系统受影响，但契约与配置保持兼容 | 完整影响域、消费者验证 |
| L4 跨边界破坏 | 破坏性契约、共享配置、数据迁移或一致性边界变化 | 完整流程、迁移与回滚审批 |
| L5 系统关键 | 关键链路、多外部系统、难回滚或影响未知 | 严格审批、系统级验证和 Go/No-Go 证据 |

取所有维度最高级别；缺少消费者、Owner、契约或配置事实时至少 L4，直到不确定性消除。L3-L5 不得走快速通道。

CR2 必须向 Boss 展示：风险级别、必须修改/必须验证/仅观察范围、消费者清单、证据缺口、迁移/回滚风险，并等待确认。

## CR3：变更计划

按依赖与部署顺序列出：
1. 基准产物和机器事实来源修改；
2. 各服务/消费者工作单元及 `service_id`；
3. 契约、配置、迁移和一致性适配动作；
4. 产品用例与技术用例更新；
5. 构建、契约、集成、故障、配置、迁移和回滚验证；
6. 外部 CI/平台证据的稳定标识要求。

在 `state.md` 的“活跃变更请求”记录 CR ID、L1-L5、基线版本/新鲜度、受影响节点、执行顺序、验证矩阵、阻断项和回滚条件。只有口头结论的外部检查标记“未验证”。

## CR4：执行变更与基准回写

- **L1-L2**：限定修改范围，进入受影响 Construction 步骤并执行服务级影响域验证。
- **L3**：按消费者矩阵执行，兼容性与所有受影响消费者验证通过后才能完成。
- **L4-L5**：从最早受影响 Inception 步骤重做，执行迁移、版本并存、部署顺序和回滚验证；L5 使用严格审批。

每完成一项立即更新 state.md。不得通过关闭检查、删除失败用例或把未验证项标记不适用来降低风险。

### 基准回写（始终执行）

按本次实际影响更新：
- 产品需求、Gherkin 与技术用例来源；
- 机器契约和 `product/contracts.md` 索引；
- 服务目录、运行时依赖、配置清单和一致性场景；
- 应用设计、工作单元、代码和验证矩阵；
- 受影响页面的需求、故事、UI Mock 与实现。

机器事实来源优先，AI-DLC 文档只保存索引和决策。禁止用独立增量文档替代权威基线回写。

## CR5：一致性验证

至少验证：
- 需求/故事/技术用例 ↔ 设计 ↔ 实现；
- 契约权威来源 ↔ 提供方 ↔ 每个消费者；
- 配置清单 ↔ 每个消费者 ↔ 环境/版本组合；
- 数据与一致性场景 ↔ 迁移/故障/恢复证据；
- 运行时影响域 ↔ C8 实际执行范围；
- 页面需求 ↔ 故事 ↔ UI Mock ↔ 实现（适用时）。

任一必需项失败或未验证时保持 CR 活跃并回到 CR4。全部通过后，记录真实证据、更新系统基线新鲜度，清除“活跃变更请求”。

## 完成消息

只报告实际修改、L1-L5 风险、验证范围、通过证据和仍未验证项。下一步提示词：

`使用 AI-DLC，继续上次的工作`
