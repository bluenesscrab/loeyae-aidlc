# 变更请求流程（CHANGE REQUEST）

**目的**：对已完成的功能进行安全、可追溯的修改 — 最小化返工，保证产物一致性

**触发条件**：意图路由判定为"变更已有功能"

---

## CR1：变更范围定位

1. 读取 `state.md`，确认项目当前状态
2. 根据用户描述的目标功能，定位对应的产出物：
   - 需求文档中的哪些条目
   - 用户故事中的哪些故事
   - 单元生成中的哪些单元
   - 已实现的哪些代码文件
3. 输出定位结果摘要

---

## CR2：影响评估

根据变更范围，沿依赖链评估影响：

```
需求文档 → 用户故事 → 工作流规划 → 应用设计 → 单元划分 → 功能设计 → 代码
```

**评估维度**：

| 维度 | 评估内容 |
|------|----------|
| 范围 | 涉及几个产出物层级？ |
| 复杂度 | 是字段级微调还是逻辑重构？ |
| 风险 | 是否影响其他功能的接口？ |
| 工作量 | 预估返工量（最小/中等/大） |

**影响级别判定**：

| 级别 | 特征 | 处理策略 |
|------|------|----------|
| 🟢 低影响 | 仅修改 1-2 个产出物层级，不影响接口 | 直接修改产物 + 更新代码 |
| 🟡 中影响 | 修改 3+ 产出物层级，但不改变架构 | 从最早受影响的产物开始，逐层更新 |
| 🔴 高影响 | 改变接口契约或架构决策 | 建议从对应步骤重新开始（按 `common-workflow-changes.md` 类型 4 处理） |

**展示评估结果**（🔴 强制审批）：

```markdown
Boss，变更影响评估结果：

**目标功能**：[功能名称]
**变更描述**：[用户描述的改动]
**影响级别**：[🟢低/🟡中/🔴高]

**受影响的产出物**：
1. [产物名] — [需要做什么修改]
2. [产物名] — [需要做什么修改]
...

**预估工作量**：[最小/中等/大]
**风险点**：[如有]

确认按此计划执行变更？

A) 确认，开始执行
B) 调整变更范围（[说明要调整什么]）
C) 放弃变更

[回答]:
```

---

## CR3：变更计划

用户确认后，生成变更执行计划：

1. 列出需要修改的产出物清单（按依赖顺序排列）
2. 对每个产出物标注具体修改内容
3. 标注哪些步骤需要重新执行（如 TDD、代码审查）
4. 更新 `state.md`：记录变更请求信息

**state.md 变更记录格式**：
```markdown
## 活跃变更请求
- **变更目标**：[功能名称]
- **变更描述**：[描述]
- **影响级别**：[级别]
- **变更计划**：
  - [ ] [产物1] — [修改内容]
  - [ ] [产物2] — [修改内容]
  - [ ] 代码更新
  - [ ] 一致性验证
- **发起时间**：[ISO 时间戳]
```

---

## CR4：执行变更

按变更计划的依赖顺序逐项执行：

**低影响变更**（🟢）：
- 直接修改产出物文件
- 更新代码（进入 Construction 的代码生成流程，但范围限定为变更部分）
- 每完成一项在 state.md 中勾选

**中影响变更**（🟡）：
- 从最早受影响的 Inception 产出物开始修改
- 逐层向下更新依赖产出物
- 进入受影响单元的 Construction 流程（功能设计 → 代码生成 → 审查）
- 每完成一项在 state.md 中勾选

**高影响变更**（🔴）：
- 归档当前产物
- 从对应步骤重新开始（按 `common-workflow-changes.md` 的类型 4 处理）
- 所有依赖步骤重做

---

## CR5：一致性验证

变更完成后，执行交叉验证确保一致性：
- 修改过的需求 ↔ 用户故事一致
- 用户故事 ↔ 代码实现一致
- 接口 ↔ 调用方一致（如有）
- 更新 state.md：清除"活跃变更请求"，记录变更完成

**完成消息**：
```markdown
✅ 变更完成

**变更摘要**：[功能名] — [改动描述]
**修改的产出物**：[列表]
**修改的代码文件**：[列表]
**验证结果**：一致性检查通过

下一步提示词：
`使用 AI-DLC，继续上次的工作`
```
