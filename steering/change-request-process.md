# 变更请求流程（CHANGE REQUEST）

**目的**：对已完成的功能进行安全、可追溯的修改 — 最小化返工，保证产物一致性

**触发条件**（任一即触发）：
1. 意图路由判定为"变更已有功能"
2. **实现期偏离触发**（Construction 阶段）：开发在 TDD/审查/对账过程中发现实现与产品用户故事（Gherkin）不一致

### 实现期偏离的三种情况

| 情况 | 表现 | 处理 |
|---|---|---|
| A. 实现 < 产品故事 | 开发漏了或简化了产品故事的某个 Given/Then | 不是演化，是 bug。回 Construction 补实现，不走 CR |
| B. 实现 > 产品故事 | 开发多做了产品故事未要求的功能 | ⚠️ 触发 CR。开发不得默默保留：产品确认要→CR4.1 回写产品故事；不要→删除多余实现 |
| C. 实现 ≠ 产品故事 | 开发改了产品意图（如接口语义变了、Then 的业务结果变了） | 🔴 触发 CR（按高影响处理）。开发必须说明：原产品故事说 X，实现成了 Y，原因 Z。产品决策：接受 Y→回写；改回 X→返工 |

### 实现期偏离的核心纪律

**禁止产出独立的"开发版用户故事/需求"**。任何与产品故事的偏离，唯一合法出口是走 CR 回写产品版——产品版始终是唯一真相。

回写时（CR4.1）的额外要求：
- 回写后的产品故事**必须符合 Gherkin 格式**（见 `inception-user-stories.md` 验收标准格式要求），不得回写成自由文字
- 回写后，受影响 UC-D 的"产品故事溯源"段必须同步更新（原样复制新 Gherkin）
- 回写记录追加到 UC-D 文件的"派生日志"段，注明 CR 编号、变更原因、产品批准

---

## CR1：变更范围定位

1. 读取 `state.md`，确认项目当前状态
2. 根据用户描述的目标功能，定位对应的产出物：
   - 需求文档中的哪些条目
   - 用户故事中的哪些故事
   - 单元生成中的哪些单元
   - 已实现的哪些代码文件
3. 输出定位结果摘要

---

## CR2：影响评估

根据变更范围，沿依赖链评估影响：

```
需求文档 → 用户故事 → 工作流规划 → 应用设计 → 单元划分 → 功能设计 → 代码
```

**评估维度**：

| 维度 | 评估内容 |
|------|----------|
| 范围 | 涉及几个产出物层级？ |
| 复杂度 | 是字段级微调还是逻辑重构？ |
| 风险 | 是否影响其他功能的接口？ |
| 页面影响 | 变更是否涉及已有页面？哪些页面的基准产出物需要同步更新？ |
| 工作量 | 预估返工量（最小/中等/大） |

**影响级别判定**：

| 级别 | 特征 | 处理策略 |
|------|------|----------|
| 🟢 低影响 | 仅修改 1-2 个产出物层级，不影响接口 | 直接修改产物 + 更新代码 |
| 🟡 中影响 | 修改 3+ 产出物层级，但不改变架构 | 从最早受影响的产物开始，逐层更新 |
| 🔴 高影响 | 改变接口契约或架构决策 | 建议从对应步骤重新开始（按 `common-workflow-changes.md` 类型 4 处理） |

**展示评估结果**（🔴 强制审批）：

```markdown
Boss，变更影响评估结果：

**目标功能**：[功能名称]
**变更描述**：[用户描述的改动]
**影响级别**：[🟢低/🟡中/🔴高]

**受影响的产出物**：
1. [产物名] — [需要做什么修改]
2. [产物名] — [需要做什么修改]
...

**预估工作量**：[最小/中等/大]
**风险点**：[如有]

确认按此计划执行变更？

A) 确认，开始执行
B) 调整变更范围（[说明要调整什么]）
C) 放弃变更

[回答]:
```

---

## CR3：变更计划

用户确认后，生成变更执行计划：

1. 列出需要修改的产出物清单（按依赖顺序排列）
2. 对每个产出物标注具体修改内容
3. 标注哪些步骤需要重新执行（如 TDD、代码审查）
4. 更新 `state.md`：记录变更请求信息

**state.md 变更记录格式**：
```markdown
## 活跃变更请求
- **变更目标**：[功能名称]
- **变更描述**：[描述]
- **影响级别**：[级别]
- **受影响页面**：[页面列表]
- **变更计划**：
  - [ ] [产物1] — [修改内容]
  - [ ] [产物2] — [修改内容]
  - [ ] 代码更新
  - [ ] 基准产出物回写
  - [ ] 一致性验证
- **发起时间**：[ISO 时间戳]
```

---

## CR4：执行变更

按变更计划的依赖顺序逐项执行：

**低影响变更**（🟢）：
- 直接修改产出物文件
- 更新代码（进入 Construction 的代码生成流程，但范围限定为变更部分）
- 每完成一项在 state.md 中勾选

**中影响变更**（🟡）：
- 从最早受影响的 Inception 产出物开始修改
- 逐层向下更新依赖产出物
- 进入受影响单元的 Construction 流程（功能设计 → 代码生成 → 审查）
- 每完成一项在 state.md 中勾选

**高影响变更**（🔴）：
- 归档当前产物
- 从对应步骤重新开始（按 `common-workflow-changes.md` 的类型 4 处理）
- 所有依赖步骤重做

---

## CR4.1：基准产出物回写（强制）

**目的**：确保变更后，受影响页面的基准产出物（需求、故事、UI Mock）是唯一且最新的权威来源。防止后续修改时参照旧版产出物导致功能丢失。

**触发条件**：CR4 完成后，**始终执行**。

**执行步骤**：

1. **识别受影响页面**：列出本次变更涉及的所有前端页面（不论是新增入口、移除功能还是调整布局）

2. **逐页面回写基准产出物**：

   对每个受影响页面，执行以下更新：

   | 产出物 | 回写内容 | 回写方式 |
   |--------|---------|---------|
   | 需求文档 | 页面功能描述必须反映变更后的完整状态 | 直接修改原文件中该页面的相关段落 |
   | 用户故事 | 涉及该页面的故事必须反映变更后的完整交互 | 修改或追加故事条目 |
   | UI Mock HTML | 该页面的 Mock 必须展示变更后的完整界面 | 直接修改对应端的 HTML 文件 |

3. **删除/标记旧产出物中的冲突内容**：
   - 如果功能从模块 A 迁移到模块 B：在模块 A 的产出物中标记「已迁移至模块 B，本处不再维护」
   - 如果页面新增了功能入口：确保该页面的 UI Mock 中包含新入口
   - 如果页面移除了功能：确保该页面的需求和 UI Mock 中已移除对应内容

4. **回写完成标记**：在 state.md 的变更计划中勾选「基准产出物回写 ✅」

**禁止行为**：
- ❌ 变更产出物作为独立增量存在，不回写到页面基准产出物
- ❌ 同一页面在不同产出物文件中存在矛盾描述
- ❌ UI Mock 中某页面的内容与最新需求文档不一致

---

## CR5：一致性验证

变更完成后，执行交叉验证确保一致性：
- 修改过的需求 ↔ 用户故事一致
- 用户故事 ↔ 代码实现一致
- 接口 ↔ 调用方一致（如有）
- **页面级一致性**（强制）：对每个受影响页面，验证以下三者描述一致：
  - 该页面的需求文档段落
  - 该页面相关的用户故事
  - 该页面的 UI Mock HTML
  - 该页面的实际代码实现
- 更新 state.md：清除"活跃变更请求"，记录变更完成

**页面级验证未通过时**：
1. 列出不一致项（如"UI Mock 中 AI 中心页面缺少 AI 绘图入口，但需求文档和代码中已包含"）
2. 回退到 CR4.1 修正基准产出物
3. 重新验证直到通过

**完成消息**：
```markdown
✅ 变更完成

**变更摘要**：[功能名] — [改动描述]
**修改的产出物**：[列表]
**修改的代码文件**：[列表]
**验证结果**：一致性检查通过

下一步提示词：
`使用 AI-DLC，继续上次的工作`
```
