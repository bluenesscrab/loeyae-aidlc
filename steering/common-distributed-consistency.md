# 分布式一致性分析

**职责**：在一个业务结果跨越多个事务、服务、数据存储或外部系统时，明确一致性边界、失败状态、恢复策略与可执行验证。本文不规定具体中间件或实现模式。

## 执行条件

满足任一条件时执行：
- 一次业务操作跨两个及以上数据所有权边界；
- 同步调用与异步事件共同决定业务结果；
- 存在重试、补偿、对账、去重或最终一致性要求；
- 外部系统响应可能延迟、重复、乱序或不可查询；
- CR 改变事务边界、事件顺序、幂等键或状态机。

单一事务边界内的本地原子操作可跳过，并记录依据。

## 必须明确的事实

每条一致性场景在 `<system-baseline-root>/consistency-scenarios.md` 记录：
- 场景 ID、业务目标和源需求/故事/CR；
- 参与方、数据 Owner 与本地事务边界；
- 事实提交点和不可逆操作；
- 允许的不一致窗口及用户可见状态；
- 幂等键、去重范围和重复处理结果；
- 超时、重试上限、退避和终止条件；
- 补偿、人工介入、对账与审计证据；
- 乱序、重复、丢失、部分成功和恢复后的期望状态。

不知道的恢复语义必须询问业务或系统 Owner，禁止默认采用“无限重试”或“最终会成功”。

## 设计约束

1. 先确定业务不变量，再选择实现方式。
2. 每个跨边界写操作只能由其数据 Owner 执行；禁止为方便直接写入其他服务的数据存储。
3. 重试必须与幂等共同设计；仅增加重试次数不构成恢复方案。
4. 补偿是显式业务动作，必须定义可补偿范围和不可补偿后的处理。
5. 对外部系统不能假设支持事务、幂等或状态查询；只能依据已确认契约。
6. 监控、审计和人工处置入口属于完成方案的一部分，但真实平台能力必须有项目证据。

## 失败用例矩阵

按适用范围至少覆盖：

| 故障 | 必须验证 |
|------|----------|
| 请求超时但对方可能成功 | 重试不会重复产生业务结果，且可查询/对账 |
| 消息重复或重复回调 | 幂等键和去重边界有效 |
| 消息乱序 | 状态不会回退或违反业务不变量 |
| 局部提交后下游失败 | 补偿、重试或人工处置进入明确状态 |
| 外部系统长时间不可用 | 降级、积压上限、恢复和告警路径明确 |
| 应用/消费者重启 | 未完成工作可恢复且审计连续 |
| 部署期间版本并存 | 新旧契约与状态转换兼容 |

## 流程衔接

- I12：定义一致性场景和业务不变量；
- I13：将失败矩阵派生为技术用例；
- I14：把参与方工作映射到各自工作单元，不把一个跨服务流程误当作单一服务；
- CR2/CR3：评估状态、数据、契约与部署顺序影响；
- C8：执行故障、恢复和对账验证，或引用可信外部证据。

## 完成标准

- [ ] 所有跨边界写流程有明确数据 Owner 和本地事务边界
- [ ] 业务不变量、不一致窗口和用户可见状态已确认
- [ ] 幂等、重试、补偿、对账与人工处置责任明确
- [ ] 适用故障用例有实际验证证据或标记未验证
- [ ] 部署与回滚期间的版本并存风险已评估
- [ ] `state.md` 记录受影响场景、证据和阻断项
