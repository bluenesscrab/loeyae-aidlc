# 代码审查

## 概述

派发独立审查者检查代码，在问题级联之前捕获它们。审查者获得精心构造的上下文进行评估——永远不是你的会话历史。

**核心原则**：早审查，常审查。

---

## 审查时机

### 强制审查

| 时机 | 触发条件 |
|------|----------|
| 每个单元完成后 | 子 Agent 模式下的两阶段审查 |
| 所有单元完成后 | 最终全局审查 |
| 合并前 | 确保整体一致性 |

### 建议审查（可选但有价值）

| 时机 | 原因 |
|------|------|
| 卡住时 | 新鲜视角可能发现盲点 |
| 重构前 | 建立基线 |
| 修复复杂 bug 后 | 确认修复没有引入新问题 |

---

## 两阶段审查流程

### 阶段 1：规格合规审查

**目的**：验证实现是否完全匹配规格要求。不多不少。

**审查清单**：

```markdown
## 规格合规检查

### 完整性（缺失检查）
- [ ] 需求文档中的每个功能点都已实现
- [ ] 用户故事中的每个验收标准都已满足
- [ ] 接口契约中定义的每个端点/方法都已实现
- [ ] 错误处理覆盖了规格中定义的所有异常场景

### 准确性（偏差检查）
- [ ] 实现的行为与规格描述一致
- [ ] 数据模型与功能设计一致
- [ ] API 响应格式与契约一致
- [ ] 业务规则的实现与需求描述一致

### 无多余（过度实现检查）
- [ ] 没有添加规格未要求的功能
- [ ] 没有添加规格未要求的参数/字段
- [ ] 没有添加规格未要求的端点
- [ ] 没有"顺便"做的改进

### 接口契约合规
- [ ] 模块间调用符合 contracts.md 定义
- [ ] 请求/响应格式与契约一致
- [ ] 错误码与契约定义一致
```

**审查结果格式**：

```markdown
## 规格合规审查结果

**状态**：✅ 合规 | ❌ 不合规

### 如果不合规：

**缺失**（规格要求但未实现）：
1. [具体缺失项 + 规格引用]

**多余**（规格未要求但添加了）：
1. [具体多余项]

**偏差**（实现与规格意图不符）：
1. [具体偏差 + 期望行为 vs 实际行为]

### 修复要求：
- [具体修复指令]
```

### 阶段 2：代码质量审查

**目的**：确保代码质量、可维护性和安全性。

**前置条件**：规格合规审查必须先通过。

**审查维度**：

#### 关键问题（必须修复，阻塞继续）

| 类别 | 检查项 |
|------|--------|
| 逻辑错误 | 条件判断错误、循环边界错误、空指针风险 |
| 安全漏洞 | SQL 注入、XSS、未授权访问、敏感数据泄露 |
| 数据风险 | 数据丢失可能、并发问题、事务不完整 |
| 异常处理 | 未捕获异常、异常吞没、错误的异常类型 |

#### 重要问题（应该修复）

| 类别 | 检查项 |
|------|--------|
| 命名 | 变量/方法/类命名不清晰或不符合规范 |
| 结构 | 方法过长、类职责不单一、层次混乱 |
| 重复 | 代码重复、可提取的公共逻辑 |
| 测试 | 测试覆盖不足、测试质量差、缺少边界测试 |
| 规范 | 违反团队编码规范（Loeyae Boot / Vue 3） |

#### 建议（可选修复）

| 类别 | 检查项 |
|------|--------|
| 性能 | 可优化的查询、不必要的循环、缓存机会 |
| 抽象 | 更好的设计模式、更清晰的抽象层次 |
| 文档 | 复杂逻辑缺少注释、公共 API 缺少文档 |

**审查结果格式**：

```markdown
## 代码质量审查结果

**状态**：✅ 通过 | ⚠️ 需要修复

### 优点
- [做得好的地方，具体引用代码]

### 问题

**[关键]** — 必须修复
1. 文件：`path/to/file.java` L42
   问题：[描述]
   建议：[修复方案]

**[重要]** — 应该修复
1. 文件：`path/to/file.java` L78
   问题：[描述]
   建议：[修复方案]

**[建议]** — 可选
1. [描述 + 建议]

### 评估
- 关键问题：X 个（必须全部修复）
- 重要问题：X 个（建议全部修复）
- 建议：X 个（自行决定）

**结论**：可以继续 | 需要修复后重新审查
```

## 审查循环

```
审查发现问题
  ↓
实现者修复
  ↓
重新审查（同一审查者视角）
  ↓
通过？→ 继续
未通过？→ 回到修复
```

**规则**：
- 审查发现问题 = 必须修复 = 必须重新审查
- 不允许"修复了应该没问题了"跳过重新审查
- 重新审查只检查修复部分 + 修复是否引入新问题

## 最终全局审查

所有单元完成后，执行一次全局审查：

```markdown
## 最终全局审查清单

### 跨单元一致性
- [ ] 单元间接口调用正确
- [ ] 共享数据模型一致
- [ ] 错误处理策略统一
- [ ] 命名风格统一

### 集成完整性
- [ ] 所有单元能正确协作
- [ ] 依赖关系正确
- [ ] 配置完整（无遗漏的配置项）

### 非功能需求
- [ ] 性能要求满足（如有）
- [ ] 安全要求满足
- [ ] 日志和监控就位

### 测试完整性
- [ ] 单元测试覆盖率达标
- [ ] 集成测试覆盖关键路径
- [ ] 边界情况已测试

### 部署就绪
- [ ] 无硬编码配置
- [ ] 数据库迁移脚本就绪（如需）
- [ ] 文档更新（API 文档、README）
```

---

## 处理审查反馈

### 必须修复（关键 + 重要）

```
1. 修复问题
2. 运行测试确认无回归
3. 重新提交审查
```

### 不同意审查意见

如果认为审查意见不正确：

```
1. 提供技术理由说明为什么当前实现是正确的
2. 引用代码/测试证明
3. 如果审查者仍然坚持，升级给用户决定
```

**绝不**：
- 忽略关键问题
- 带着未修复的重要问题继续
- 不提供理由就拒绝反馈

## 与 AI-DLC 工作流的集成

### 在 Construction 阶段的位置

```
代码生成 — TDD 执行阶段
  ↓ 每个单元完成
代码审查（本文件）
  ↓ 审查通过
标记单元完成，进入下一个单元
  ↓ 所有单元完成
最终全局审查
  ↓ 全局审查通过
构建和测试
```

审查结果记录到对应的审计文件：

```markdown
**时间戳**: [ISO 时间戳]

**规格合规结果**: ✅ 合规
**代码质量结果**: ⚠️ 需要修复
**问题**: [关键] 0 个, [重要] 2 个, [建议] 1 个
**修复状态**: 已修复并重新审查通过
```

## 红旗信号

**绝不：**
- 因为"很简单"就跳过审查
- 忽略关键问题
- 带着未修复的重要问题继续
- 与有效的技术反馈争论（不提供证据）
- 在规格合规通过之前做质量审查
- 修复后不重新审查

---

## 最终规则

```
代码审查不是可选的。
它是质量的最后一道防线。
跳过审查 = 把 bug 交给用户。
```
