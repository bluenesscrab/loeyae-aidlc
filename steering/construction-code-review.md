# 代码审查

## 概述

派发独立审查者检查代码，在问题级联之前捕获它们。审查者获得精心构造的上下文进行评估——永远不是你的会话历史。

**核心原则**：早审查，常审查。

---

## 审查时机

### 强制审查

| 时机 | 触发条件 |
|------|----------|
| 每个单元完成后 | 双轴独立审查（Standards + Spec） |
| 所有单元完成后 | 最终全局审查 |
| 合并前 | 确保整体一致性 |

### 建议审查（可选但有价值）

| 时机 | 原因 |
|------|------|
| 卡住时 | 新鲜视角可能发现盲点 |
| 重构前 | 建立基线 |
| 修复复杂 bug 后 | 确认修复没有引入新问题 |

## 双轴独立审查流程

### Standards Axis：代码质量审查

**目的**：验证代码是否符合项目规范和质量标准。

**审查维度**：

#### 关键问题（必须修复，阻塞继续）

| 类别 | 检查项 |
|------|--------|
| 逻辑错误 | 条件判断错误、循环边界错误、空指针风险 |
| 安全漏洞 | SQL 注入、XSS、未授权访问、敏感数据泄露 |
| 数据风险 | 数据丢失可能、并发问题、事务不完整 |
| 异常处理 | 未捕获异常、异常吞没、错误的异常类型 |

#### 重要问题（应该修复）

**Fowler 代码 Smells 基线**（11种）：

| Smell | 定义 | 修复建议 |
|-------|------|----------|
| **Mysterious Name** | 命名不揭示意图 | 重命名；若无诚实名称则设计有问题 |
| **Duplicated Code** | 相同逻辑在多处出现 | 提取公共方法/类 |
| **Feature Envy** | 方法更多访问其他对象数据而非自身 | 将方法移到它羡慕的数据所在类 |
| **Data Clumps** | 相同字段/参数总是成组出现 | 封装为独立类型 |
| **Primitive Obsession** | 基本类型代表领域概念 | 创建领域小类型 |
| **Repeated Switches** | 相同 switch/if-cascade 多处重复 | 用多态或共享 Map 替代 |
| **Shotgun Surgery** | 一个变更导致多处分散修改 | 将变更聚合到一个模块 |
| **Divergent Change** | 一个模块因多个无关原因被修改 | 拆分模块，单一变更原因 |
| **Speculative Generality** | 为规格未要求的未来需求添加抽象 | 删除，回归直到真实需求出现 |
| **Message Chains** | 长 `a.b().c().d()` 调用链 | 在第一个对象后隐藏导航方法 |
| **Middle Man** | 类仅做委托转发 | 删除中间层，直接调用目标 |

**项目规范覆盖规则**：
- 以上 Smells 为基线判断
- 项目编码规范（MCP Skills + CLAUDE.md）优先
- 项目规范明确允许的情况，不标记为 Smell

**其他重要问题**：

| 类别 | 检查项 |
|------|--------|
| 命名 | 违反团队命名规范（Loeyae Boot / Vue 3） |
| 结构 | 方法过长（> 50行）、类职责不单一、层次混乱 |
| 测试 | 测试覆盖不足、测试质量差、缺少边界测试 |
| 规范 | 违反团队编码规范 |

#### 建议（可选修复）

| 类别 | 检查项 |
|------|--------|
| 性能 | 可优化的查询、不必要的循环、缓存机会 |
| 抽象 | 更好的设计模式、更清晰的抽象层次 |
| 文档 | 复杂逻辑缺少注释、公共 API 缺少文档 |

**Standards Axis 审查结果格式**：

```markdown
## Standards Axis 审查结果

**状态**：✅ 通过 | ⚠️ 需要修复

### 优点
- [做得好的地方，具体引用代码]

### 问题

**[关键]** — 必须修复
1. 文件：`path/to/file.java` L42
   问题：[描述]
   建议：[修复方案]

**[重要 — Smell: Feature Envy]**
1. 文件：`path/to/file.java` L78
   问题：[描述]
   建议：[修复方案]

**[重要 — 其他]**
1. 文件：`path/to/file.java` L90
   问题：[描述]
   建议：[修复方案]

**[建议]** — 可选
1. [描述 + 建议]

### 评估
- 关键问题：X 个
- 重要问题：X 个（Smells: Y 个, 其他: Z 个）
- 建议：X 个
```

### Spec Axis：规格合规审查

**审查清单**：

```markdown
## 规格合规检查

### 完整性（缺失检查）
- [ ] 需求文档中的每个功能点都已实现
- [ ] 用户故事中的每个验收标准都已满足
- [ ] 接口契约中定义的每个端点/方法都已实现
- [ ] 错误处理覆盖了规格中定义的所有异常场景

### 准确性（偏差检查）
- [ ] 实现的行为与规格描述一致
- [ ] 数据模型与功能设计一致
- [ ] API 响应格式与契约一致
- [ ] 业务规则的实现与需求描述一致

### 无多余（过度实现检查）
- [ ] 没有添加规格未要求的功能
- [ ] 没有添加规格未要求的参数/字段
- [ ] 没有添加规格未要求的端点
- [ ] 没有"顺便"做的改进

### 接口契约合规
- [ ] 模块间调用符合 contracts.md 定义
- [ ] 请求/响应格式与契约一致
- [ ] 错误码与契约定义一致

### UI Mock 合规（条件：该单元涉及前端页面且存在 UI Mock）
- [ ] 页面结构与 UI Mock HTML 原型布局一致
- [ ] 表单字段（名称、类型、必填标记）与 UI Mock 定义一致
- [ ] 操作按钮和交互行为覆盖 UI Mock 中定义的所有用户操作
- [ ] 列表/表格的列定义与 UI Mock 展示一致
- [ ] 业务状态展示（标签、颜色、图标）与 UI Mock 一致
- [ ] 条件渲染逻辑（角色权限、状态驱动的显隐）与 UI Mock 标注一致
- [ ] 响应式/多端适配与 UI Mock 标注的断点行为一致（如有标注）

### 数据流完整性（强制）
- [ ] 每个数据读取操作都有对应的真实写入来源（非 Mock/硬编码）
- [ ] 依赖其他模块/单元的数据，该模块/单元已实现或在本次范围内
- [ ] 外部接口数据源已确认可用（非假设存在）
- [ ] 初始/种子数据的准备方案已落实（SQL 脚本、数据迁移、或录入入口）
- [ ] 列表/查询页面对应的数据写入入口已实现（不存在"能查不能增"的空页面）
- [ ] 数据源验证产物（`data-source-validation.md`）中无 🚨 无来源项
```

**审查结果格式**：

```markdown
## 规格合规审查结果

**状态**：✅ 合规 | ❌ 不合规

### 如果不合规：

**缺失**（规格要求但未实现）：
1. [具体缺失项 + 规格引用]

**多余**（规格未要求但添加了）：
1. [具体多余项]

**偏差**（实现与规格意图不符）：
1. [具体偏差 + 期望行为 vs 实际行为]

**UI Mock 不一致**（条件：涉及前端页面且存在 UI Mock）：
1. [具体不一致项 + UI Mock 文件引用 + 期望 vs 实际]

**数据流缺陷**（数据空中楼阁）：
1. [具体数据读取操作 + 缺失的写入来源 + 影响范围]

### 修复要求：
- [具体修复指令]
```

## 双轴审查模式说明

两个审查轴（Standards + Spec）**独立执行、独立报告**：

```
┌──────────────────────┐
│ Standards Axis       │ → Standards 报告
├──────────────────────┤
│ Spec Axis            │ → Spec 报告
└──────────────────────┘
         ↓
    合并报告（不重排序，两轴各自保留）
         ↓
    是否有关键问题？
    是 → 修复循环
    否 → 审查通过
```

**为什么两轴独立报告？**
- Standards 通过但 Spec 失败 → 需求未满足
- Spec 通过但 Standards 失败 → 代码质量差
- 合并排序可能掩盖某轴问题

**审查记录格式**：

```markdown
## 代码审查 — 单元 X
**时间戳**: [ISO 时间戳]
**审查模式**: 双轴独立审查
**Standards 结果**: ✅ 通过 | ⚠️ 需要修复
**Spec 结果**: ✅ 合规 | ❌ 不合规
**Standards 问题**: 关键 X 个, 重要 Y 个（Smells Z 个）, 建议 W 个
**Spec 问题**: 缺失 X 个, 偏差 Y 个, 多余 Z 个
**修复状态**: [已修复并重新审查通过 | 待修复]
```

---

## 审查循环

```
审查发现问题
  ↓
实现者修复
  ↓
重新审查（同一审查者视角）
  ↓
通过？→ 继续
未通过？→ 回到修复
```

**规则**：
- 审查发现问题 = 必须修复 = 必须重新审查
- 不允许"修复了应该没问题了"跳过重新审查
- 重新审查只检查修复部分 + 修复是否引入新问题

## 最终全局审查

所有单元完成后，执行一次全局审查：

```markdown
## 最终全局审查清单

### 跨单元一致性
- [ ] 单元间接口调用正确
- [ ] 共享数据模型一致
- [ ] 错误处理策略统一
- [ ] 命名风格统一

### 集成完整性
- [ ] 所有单元能正确协作
- [ ] 依赖关系正确
- [ ] 配置完整（无遗漏的配置项）

### 非功能需求
- [ ] 性能要求满足（如有）
- [ ] 安全要求满足
- [ ] 日志和监控就位

### 测试完整性
- [ ] 单元测试覆盖率达标
- [ ] 集成测试覆盖关键路径
- [ ] 边界情况已测试

### 部署就绪
- [ ] 无硬编码配置
- [ ] 数据库迁移脚本就绪（如需）
- [ ] 文档更新（API 文档、README）
```

## 处理审查反馈

### 必须修复（关键 + 重要）

```
1. 修复问题
2. 运行测试确认无回归
3. 重新提交审查
```

### 不同意审查意见

如果认为审查意见不正确：

```
1. 提供技术理由说明为什么当前实现是正确的
2. 引用代码/测试证明
3. 如果审查者仍然坚持，升级给用户决定
```

**绝不**：
- 忽略关键问题
- 带着未修复的重要问题继续
- 不提供理由就拒绝反馈

## 红旗信号

**绝不：**
- 因为"很简单"就跳过审查
- 忽略关键问题
- 带着未修复的重要问题继续
- 与有效的技术反馈争论（不提供证据）
- 在规格合规通过之前做质量审查（两轴可并行，但都必须通过）
- 修复后不重新审查

---

## 最终规则

```
代码审查不是可选的。
它是质量的最后一道防线。
跳过审查 = 把 bug 交给用户。
```
