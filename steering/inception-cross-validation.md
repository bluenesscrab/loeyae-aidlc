# Inception 产物交叉验证（强制审查）

**目的**：确保 Inception 阶段各产物之间保持一致性，问题发现越早成本越低。

**执行时机**：每个产物完成后立即执行对应的审查项，不等待所有产物完成。

---

## 审查项总览

| 审查项 | 执行时机 | 条件 | 步骤编号 |
|--------|---------|------|----------|
| a) 需求 ↔ 逆向工程 | 需求分析完成后 | 有逆向工程产物 | 3.1 |
| b) 需求 ↔ 澄清记录 | 需求分析完成后 | 始终 | 3.1 |
| c) 用户故事 ↔ 需求 | 用户故事完成后 | 始终 | 4.1 |
| d) 需求完整性评估 | 用户故事完成后 | 始终 | 4.1 |
| e) UI Mock ↔ 用户故事+需求 | UI Mock 完成后 | 执行了 UI Mock | 4.6 |

---

## 审查项 a：需求文档 ↔ 逆向工程

**执行条件**：存在逆向工程产物（`docs/aidlc/inception/reverse-engineering/`）

**输入文件**：
- `docs/aidlc/inception/requirements/requirements.md`
- `docs/aidlc/inception/reverse-engineering/architecture.md`
- `docs/aidlc/inception/reverse-engineering/component-inventory.md`

**审查清单**：

| # | 检查点 | 通过标准 |
|---|--------|---------|
| 1 | 需求涉及的现有组件是否在逆向工程中被识别 | 所有涉及组件均已列出 |
| 2 | 需求是否与现有系统架构矛盾 | 无架构冲突，或已说明改造方案 |
| 3 | 需求引用的现有功能是否与逆向工程描述一致 | 功能描述无偏差 |
| 4 | 需求中的"新增"与"改造"分类是否与现有代码对应 | 分类准确 |
| 5 | 需求是否遗漏了逆向工程中发现的相关依赖 | 依赖关系完整 |

**未通过处理**：
1. 列出所有不一致项
2. 向 Boss 报告发现的问题
3. 修正需求文档中的不一致
4. 重新执行审查项 a 直到通过

---

## 审查项 b：需求文档 ↔ 澄清记录

**执行条件**：始终执行

**输入文件**：
- `docs/aidlc/inception/requirements/requirements.md`
- `docs/aidlc/inception/requirements/requirement-verification-questions.md`
- `docs/aidlc/audit.md`（澄清问答部分）

**审查清单**：

| # | 检查点 | 通过标准 |
|---|--------|---------|
| 1 | 用户在澄清中提供的每个答案是否已体现在需求文档中 | 所有答案均已反映 |
| 2 | 用户明确否定的内容是否已从需求中移除 | 无被否定内容残留 |
| 3 | 用户补充的新信息是否已纳入需求 | 补充信息完整纳入 |
| 4 | 澄清中确认的优先级是否与需求文档优先级一致 | 优先级一致 |
| 5 | 澄清中确认的范围边界是否在需求中明确标注 | 范围边界清晰 |

**未通过处理**：
1. 列出所有遗漏/不一致项
2. 向 Boss 报告发现的问题
3. 修正需求文档
4. 重新执行审查项 b 直到通过

---

## 审查项 c：用户故事 ↔ 需求文档

**执行条件**：始终执行（用户故事完成后）

**输入文件**：
- `docs/aidlc/inception/user-stories/stories.md`
- `docs/aidlc/inception/requirements/requirements.md`

**审查清单**：

| # | 检查点 | 通过标准 |
|---|--------|---------|
| 1 | 需求文档中每个功能需求（FR）是否都有对应的用户故事覆盖 | 100% FR 被覆盖 |
| 2 | 用户故事的验收标准是否与需求中的 GWT 一致 | 验收标准对齐 |
| 3 | 用户故事涉及的角色是否与需求中的目标用户一致 | 角色映射正确 |
| 4 | 用户故事是否引入了需求文档中不存在的功能 | 无超范围故事（或已标注为增强建议） |
| 5 | 需求中的优先级（Must/Should/Could）是否在故事中体现 | 优先级传递正确 |

**未通过处理**：
1. 列出覆盖缺口和不一致项
2. 向 Boss 报告并建议：补充用户故事 或 修正需求文档
3. 获得 Boss 确认后执行修正
4. 重新执行审查项 c 直到通过

---

## 审查项 d：基于用户故事评估需求完整性

**执行条件**：始终执行（与审查项 c 同时，用户故事完成后）

**目的**：从用户故事角度**反向**验证需求文档的完整性。当用户故事描述了用户期望的行为，但需求文档中找不到对应支撑时，说明需求可能有遗漏。

**输入文件**：
- `docs/aidlc/inception/user-stories/stories.md`
- `docs/aidlc/inception/user-stories/personas.md`
- `docs/aidlc/inception/requirements/requirements.md`

**审查清单**：

| # | 检查点 | 通过标准 |
|---|--------|---------|
| 1 | 每个用户画像的核心场景是否都有需求支撑 | 所有场景有需求覆盖 |
| 2 | 用户故事中隐含的前置条件是否在需求中定义 | 前置条件明确 |
| 3 | 用户故事中的异常路径是否在需求中有对应处理 | 异常路径覆盖 |
| 4 | 跨故事的数据流转是否在需求中有完整定义 | 数据流完整 |
| 5 | 用户故事暴露的新需求是否已标注 | 新发现需求已记录 |

**未通过处理**：
1. 列出需求文档中可能遗漏的内容
2. 向 Boss 报告发现并建议补充
3. 获得 Boss 确认后补充需求文档
4. 重新执行审查项 d 直到通过

2. 向 Boss 报告发现的问题
3. 修正 UI Mock 中的不一致
4. 重新执行审查项 e 直到通过

## 审查结果输出格式

每次审查完成后，输出以下格式的审查报告：

```markdown
## 🔍 交叉验证报告 — 审查项 {a/b/c/d/e}

**审查时间**：{ISO时间}
**审查对象**：{被审查的产物}

### 审查结果

| # | 检查点 | 结果 | 说明 |
|---|--------|------|------|
| 1 | {检查点描述} | ✅/❌ | {具体说明} |
| ... | ... | ... | ... |

### 总结
- **通过项**：{N} / {Total}
- **未通过项**：{列出未通过项及原因}
- **整体结论**：✅ 通过 / ❌ 未通过，需修正

### 修正建议（如未通过）
1. {具体修正建议}
2. ...
```

---

## 与步骤完成协议的关系

- 审查步骤本身也是一个步骤，完成后必须执行步骤完成协议（更新 state.md、微型摘要、审计记录）
- 审查步骤在 state.md 中记录为 `{步骤编号} 交叉验证 ✅/❌`
- 如果审查未通过并修正后重新通过，在审计中记录"首次未通过 → 修正后通过"

> ```
