# 错误处理和恢复流程

## 通用错误处理原则

### 当错误发生时
1. **识别错误**：清楚说明出了什么问题
2. **评估影响**：判断错误是阻塞性的还是可以绕过的
3. **沟通**：告知用户错误情况和可选方案
4. **提供解决方案**：提供清晰的步骤来解决或绕过错误
5. **记录**：在审计文件中记录错误和解决方案

### 错误严重级别

| 级别 | 含义 | 行动 |
|------|------|------|
| **严重** | 工作流无法继续（缺失必需文件、无效输入、系统错误） | 立即停止，请求用户介入 |
| **高** | 阶段无法完成（回答不完整、前序依赖缺失、矛盾回复） | 暂停当前步骤，定位问题 |
| **中** | 可通过变通方案继续（可选产物缺失、非关键验证失败） | 记录问题，继续执行 |
| **低** | 不阻塞进度（格式不一致、可选信息缺失） | 记录后继续 |

## 阶段特定的错误处理速查

> **原则**：不需要预加载所有阶段的错误场景。遇到错误时，按通用原则处理 + 查阅下表定位。

| 阶段 | 常见错误 | 处理策略 |
|------|---------|---------|
| 工作区检测 | state.md 损坏、无法读取工作区 | 备份 → 询问用户重建或恢复 |
| 需求分析 | 矛盾需求、回答不完整 | 创建后续问题澄清，不得跳过 |
| 用户故事 | 无法映射需求到故事 | 返回需求阶段补充，或标记不完整 |
| 应用设计 | 架构决策矛盾、边界不清 | 添加后续问题，默认单体架构 |
| 功能设计 | 循环依赖、计划步骤缺失 | 重构边界打破循环，补充计划 |
| NFR 实施 | 技术栈不兼容、需人工操作 | 标记人工任务，等待确认 |
| 代码生成 | 设计信息不足、语法错误、测试失败 | 补充信息 → 修复 → 重新验证 |
| Operations | 构建工具不确定、部署目标不清 | 请用户指定，提供常见选项 |

**通用处理流程**：
1. 识别错误所属阶段
2. 判断严重级别
3. 按级别处理（严重=停止、高=暂停、中=变通、低=记录）
4. 在审计文件中记录错误和解决方案
5. 恢复后继续

## 恢复流程

### 阶段部分完成

1. 加载阶段计划文件
2. 识别最后完成的步骤（最后一个 [x]）
3. 验证之前步骤确实完成
4. 从下一个未完成步骤恢复

### 状态文件损坏

1. 创建备份：`state.md.backup`
2. 询问用户实际处于哪个阶段
3. 根据现有产物重建 state.md
4. 从当前阶段恢复

### 产物缺失

1. 识别缺失产物
2. 判断是否可重新生成
3. 可以 → 返回该阶段重新生成
4. 不可以 → 请用户手动提供
5. 记录缺口

### 用户要求重新开始

1. 确认用户理解数据将丢失
2. 归档现有产物
3. 重置 state.md 对应阶段
4. 从头执行

### 用户要求跳过

1. 确认用户理解跳过影响
2. 在审计中记录跳过原因
3. state.md 标记"SKIPPED"
4. 继续下一阶段

## 会话恢复错误

| 场景 | 处理 |
|------|------|
| state.md 显示完成但产物不存在 | 重置阶段状态，重新执行 |
| 产物存在但 state.md 显示未完成 | 验证产物有效 → 更新 state.md |
| 多阶段标记为"当前" | 审查产物确定实际进度 → 修正 |
| 加载产物包含矛盾 | 呈现给用户选择 → 调和后继续 |
| 前序产物无法加载 | 列出缺失项 → 重新生成或请用户提供 |

**恢复最佳实践**：
- 始终验证 state.md 与实际产物是否匹配
- 增量加载，逐阶段验证
- 关键产物缺失立即停止
- 记录所有恢复操作到审计文件

## 升级处理

### 立即请求用户帮助
- 用户输入矛盾或模糊
- 缺少必需信息且无法推断
- AI 无法解决的技术约束
- 需要业务判断的决策

### 尝试解决后请求
- 同一步骤反复出错（3+ 次）
- 不常见的项目结构
- 外部系统集成问题

### 建议重新开始
- 多个阶段存在关联错误
- 状态文件严重损坏
- 架构决策需要撤销

## 日志格式

```markdown
## 错误 - [阶段名称]
**时间戳**：[ISO 时间戳]
**级别**：[严重/高/中/低]
**描述**：[出了什么问题]
**原因**：[为什么发生]
**解决方案**：[如何解决]
```

---

## 预防最佳实践

1. 尽早验证输入和依赖
2. 频繁设置检查点（步骤完成协议）
3. 不假设——立即澄清歧义
4. 在审计文件中记录所有决策
