## 团队强制规则

1. 使用简体中文交互
2. 禁止 TODO、FIXME 或占位符 — 所有功能必须实现到位
3. 按需加载文档，避免不必要的 token 消耗
4. 无法实现时明确告知用户原因，提出替代方案
5. 说服式防护替代硬规则：理解为什么不能跳步，而非仅因规则禁止
6. 支持团队协作 — 识别协作模式，按角色加载最小上下文

## 复杂度评估与快速通道

| 级别 | 条件 | 执行路径 |
|------|------|----------|
| 简单 | ≤3文件 + 无新数据模型 + 无跨服务 + 意图清晰 + 无安全/性能风险 | 快速通道：确认 → TDD → 验证，跳过 Inception 文档，审计仅一行；若低估复杂度则升级 |
| 中等 | 4-10文件 + 有业务逻辑但范围明确 + 单服务内 | 精简流程：需求分析 → 工作流规划 → [功能设计] → 代码生成 → 构建测试，跳过用户故事/NFR/基础设施设计 |
| 复杂 | >10文件 或 跨服务 或 新架构组件 或 高风险 | 完整流程：三步阶段全部按需执行 |

评估时机：工作区检测后首次评估，实现中可升级（不可降级），用户可覆盖。

## 审批模式

| 模式 | 说明 |
|------|------|
| 严格 | 所有审批点都等待用户确认（团队协作、高风险、新手） |
| 标准（默认） | 仅强制审批点(🔴)等待，默认审批点(🟡)自动通过 |
| 自主 | 仅需求确认和架构决策等待 |

审批点分级：🔴 强制审批（任何模式必等：需求确认、架构决策、产品级 Inception、安全相关变更、代码生成结果、构建测试确认）；🟡 默认审批（严格模式等待，标准/自主自动通过：代码生成计划、功能设计、NFR 设计、逆向工程）；🟢 信息通知（不等待）。

## 三阶段概览

**INCEPTION** — 规划、需求收集和架构决策，聚焦做什么（WHAT）和为什么做（WHY）。
核心步骤：工作区检测 → 产品级 Inception(条件,多模块) → 模块选择(条件,多模块) → 逆向工程(条件) → 需求分析 → 用户故事(条件) → UI Mock(条件) → 工作流规划 → 应用设计(条件) → 单元生成(条件)。

**CONSTRUCTION** — 详细设计、NFR 实现和代码生成，聚焦怎么做（HOW）。
核心步骤：Per-Unit 循环（功能设计 → NFR需求/设计 → 基础设施设计 → 代码生成[TDD: RED→GREEN→REFACTOR → 两阶段审查] → 系统化调试[条件]）→ 最终全局审查 → 构建和测试。

**OPERATIONS** — 生成 CI/CD 配置和部署文档，聚焦如何运行和维护（HOW to operate）。
核心步骤：部署需求分析 → 部署规划问答 → CI/CD 配置文件生成 → 部署文档生成。

## 加载策略

按需加载，禁止预加载所有文件。启动时自动加载本文件。恢复项目时按需加载 session-continuity 或 team-collaboration。进入某阶段时加载对应 steering 文件。执行特定任务时加载对应规则文件（如代码生成时加载 TDD 规则、内容验证规则等）。禁止一次性加载所有通用规则文件或预加载后续阶段文件。

## 核心原则

- 自适应执行：仅执行有价值的阶段
- 复杂度驱动：简单任务走快速通道，复杂任务走完整流程 — 流程复杂度匹配任务复杂度
- 手术式变更：每一行改动追溯到用户请求，不"顺手改进"无关代码
- 审批分级：强制审批(🔴)任何模式必等，默认审批(🟡)按模式决定，信息通知(🟢)不等待
- 透明规划：开始前始终展示执行计划
- 用户控制：用户可覆盖复杂度评估和审批模式
- 说服式防护：理解为什么不能跳步，跳过步骤的代价远高于执行成本
- TDD 纪律：没有失败测试就没有生产代码，违反则删除代码重新开始
- 系统化调试：先找根因再修复，3+ 次修复失败 = 架构问题，停止修补
- 两阶段审查：每个单元完成后必须经过规格合规 + 代码质量两阶段审查
- 质量门禁：所有阶段必须通过强制质量门禁检查才能继续
- 内容验证：按 content-validation.md 规则在文件创建前验证内容
- 禁止涌现行为：Construction 阶段使用各自规则文件中定义的标准化完成消息
- Session 交接：每个阶段完成消息内置新 Session 继续选项，交接提示词写入 state.md
- 团队协作：支持接力(多人按步骤)和认领(自主认领单元)模式，单人时自动退化
