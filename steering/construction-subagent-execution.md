# 平台自适应子 Agent 执行

## 概述

通过派发独立子 Agent 执行每个单元的代码生成，保持主 Agent 上下文清洁，实现高质量快速迭代。

**核心原则**：新鲜子 Agent + 两阶段审查（规格合规 → 代码质量）= 高质量、快速迭代

**平台自适应**：检测当前平台是否支持子 Agent 派发。支持则使用子 Agent 模式，不支持则退化为单 Agent 模式但保留两阶段审查逻辑。

---

## 平台检测

### 支持子 Agent 的平台

| 平台 | 子 Agent 机制 | 检测方式 |
|------|-------------|----------|
| Kiro | `invoke_sub_agent` 工具 | 检查可用工具列表 |
| Claude Code | `Task` 工具 | 检查可用工具列表 |
| Cursor | Composer Agent | 检查可用工具列表 |
| OpenCode | 子进程派发 | 检查可用工具列表 |

### 模式选择逻辑

```
如果 平台支持子 Agent 派发：
  → 使用「子 Agent 模式」
否则：
  → 使用「单 Agent 两阶段审查模式」
```

在工作区检测阶段记录到 state.md：
```markdown
执行模式: 子Agent模式 | 单Agent两阶段审查模式
```

## 子 Agent 模式

### 执行流程

```
┌─────────────────────────────────────────────────────────────┐
│ 主 Agent（协调者）                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 读取代码生成计划，提取所有任务                             │
│  2. 对每个任务：                                             │
│     ┌──────────────────────────────────────────────┐        │
│     │ 派发实现者子 Agent                            │        │
│     │  - 提供：任务完整文本 + 上下文 + TDD 要求      │        │
│     │  - 实现者：TDD 循环 → 自审 → 提交             │        │
│     └──────────────────────────────────────────────┘        │
│              ↓                                              │
│     ┌──────────────────────────────────────────────┐        │
│     │ 派发规格审查子 Agent                          │        │
│     │  - 检查：代码是否匹配规格？多了什么？少了什么？ │        │
│     │  - 不合格 → 实现者修复 → 重新审查              │        │
│     └──────────────────────────────────────────────┘        │
│              ↓                                              │
│     ┌──────────────────────────────────────────────┐        │
│     │ 派发质量审查子 Agent                          │        │
│     │  - 检查：代码质量、命名、结构、测试覆盖        │        │
│     │  - 不合格 → 实现者修复 → 重新审查              │        │
│     └──────────────────────────────────────────────┘        │
│              ↓                                              │
│     标记任务完成                                             │
│                                                             │
│  3. 所有任务完成后：派发最终审查子 Agent                      │
│  4. 进入代码审查阶段                                         │
└─────────────────────────────────────────────────────────────┘
```

### 实现者子 Agent 指令模板

```markdown
## 你的角色

你是一个实现者。你的任务是按照 TDD 流程实现以下功能。

## 任务

{TASK_FULL_TEXT}

## 上下文

{RELEVANT_CONTEXT}
- 项目结构：{PROJECT_STRUCTURE}
- 相关文件：{RELATED_FILES}
- 接口契约：{CONTRACTS}

## TDD 要求

你必须严格遵循 RED-GREEN-REFACTOR 循环：
1. 先写一个失败测试
2. 运行测试，确认失败
3. 写最少代码让测试通过
4. 运行测试，确认通过
5. 重构（可选）
6. 重复直到任务完成

## 编码规范

{CODING_STANDARDS}

## 完成标准

- 所有测试通过
- 代码符合编码规范
- 无 TODO/FIXME
- 自审完成（检查命名、结构、边界情况）

## 报告格式

完成后报告：
- 状态：DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED
- 实现摘要
- 测试覆盖
- 自审发现
- 关注点（如有）
```

### 规格审查子 Agent 指令模板

```markdown
## 你的角色

你是规格合规审查员。你的任务是验证实现是否完全匹配规格要求。

## 原始规格

{TASK_SPEC}

## 审查范围

检查以下 git 变更：
- 基准：{BASE_SHA}
- 当前：{HEAD_SHA}

## 审查清单

1. **完整性**：规格中的每个要求都已实现？
2. **准确性**：实现是否正确反映了规格意图？
3. **无多余**：是否添加了规格未要求的功能？
4. **接口契约**：是否符合模块间接口契约？

## 报告格式

```
规格合规审查结果：✅ 合规 | ❌ 不合规

如果不合规：
- 缺失：[规格要求但未实现的]
- 多余：[规格未要求但添加的]
- 偏差：[实现与规格意图不符的]

建议修复：
- [具体修复建议]
```
```

### 质量审查子 Agent 指令模板

```markdown
## 你的角色

你是代码质量审查员。你的任务是确保代码质量达标。

## 审查范围

检查以下 git 变更：
- 基准：{BASE_SHA}
- 当前：{HEAD_SHA}

## 审查维度

### 关键（必须修复）
- 逻辑错误
- 安全漏洞
- 数据丢失风险
- 未处理的异常

### 重要（应该修复）
- 命名不清晰
- 代码重复
- 缺少边界检查
- 测试覆盖不足
- 违反编码规范

### 建议（可选修复）
- 性能优化机会
- 更好的抽象
- 文档补充

## 报告格式

```
代码质量审查结果：✅ 通过 | ⚠️ 有问题

优点：
- [做得好的地方]

问题：
- [关键] ...
- [重要] ...
- [建议] ...

评估：可以继续 | 需要修复
```
```

### 处理实现者状态

| 状态 | 含义 | 处理方式 |
|------|------|----------|
| DONE | 任务完成 | 进入规格审查 |
| DONE_WITH_CONCERNS | 完成但有疑虑 | 评估疑虑，决定是否处理后再审查 |
| NEEDS_CONTEXT | 需要更多信息 | 提供缺失上下文，重新派发 |
| BLOCKED | 无法完成 | 评估阻塞原因（见下方） |

**BLOCKED 处理策略：**
1. 上下文问题 → 提供更多上下文，重新派发
2. 任务太复杂 → 拆分为更小的子任务
3. 计划本身有问题 → 暂停，向用户说明并建议修改计划
4. 架构问题 → 暂停，触发系统化调试流程

## 单 Agent 两阶段审查模式

当平台不支持子 Agent 时，主 Agent 自己执行但模拟两阶段审查。

### 执行流程

```
对每个单元：
  1. 执行 TDD 循环（RED-GREEN-REFACTOR）
  2. 完成后，切换到「规格审查员」视角：
     - 重新阅读原始规格
     - 逐项检查实现是否匹配
     - 记录审查结果
  3. 如有问题，修复后重新审查
  4. 切换到「质量审查员」视角：
     - 检查代码质量
     - 记录审查结果
  5. 如有问题，修复后重新审查
  6. 两阶段都通过后，标记任务完成
```

### 视角切换标记

在审计日志中明确标记视角切换：

```markdown
## 规格合规审查（单Agent模式）
**时间戳**: [ISO 时间戳]
**视角**: 规格审查员
**审查对象**: 单元 X 的实现
**审查结果**: ✅ 合规 / ❌ 不合规
**详情**: ...

---

## 代码质量审查（单Agent模式）
**时间戳**: [ISO 时间戳]
**视角**: 质量审查员
**审查对象**: 单元 X 的实现
**审查结果**: ✅ 通过 / ⚠️ 有问题
**详情**: ...
```

### 单 Agent 模式的额外纪律

因为没有独立审查者的"新鲜视角"，单 Agent 模式需要额外纪律：

1. **强制重读规格**：审查前必须重新读取原始规格文件，不依赖记忆
2. **逐项检查**：不允许"整体看起来没问题"，必须逐项对照
3. **记录证据**：每个审查结论必须附带具体代码引用
4. **自我质疑**：主动寻找自己可能遗漏的问题

---

## 连续执行原则

**不要在任务之间暂停询问用户。** 连续执行计划中的所有任务。

唯一停止的理由：
- BLOCKED 状态且无法自行解决
- 发现计划本身有歧义，无法继续
- 所有任务完成

**禁止：**
- "要继续吗？"
- "任务 1 完成，要开始任务 2 吗？"
- 进度汇报（除非用户主动询问）

用户让你执行计划，就执行它。

## 与 AI-DLC 工作流的集成

### 在 Construction 阶段的位置

```
Construction 阶段：
  Per-Unit 循环：
    1. 功能设计（条件）
    2. NFR 需求/设计（条件）
    3. 基础设施设计（条件）
    4. 代码生成 — 规划阶段（必执行）
       ↓ 用户批准计划
    5. 代码生成 — TDD 执行阶段（必执行）  ← 本文件控制
       - 子 Agent 模式：派发实现者 + 两阶段审查
       - 单 Agent 模式：TDD 循环 + 两阶段自审
    6. 代码审查（必执行）                  ← construction-code-review.md
```

### 上下文传递

主 Agent 向实现者子 Agent 传递的上下文：

| 内容 | 来源 |
|------|------|
| 任务完整文本 | 代码生成计划 |
| 接口契约 | `docs/aidlc/product/contracts.md` 或应用设计 |
| 编码规范 | MCP Skill（如适用）或 steering 文件 |
| 相关现有代码 | 工作区扫描结果 |
| TDD 执行序列 | 代码生成计划中的 TDD 规划 |

**禁止传递**：
- 主 Agent 的会话历史
- 其他单元的实现细节（除非有依赖）
- 完整的 Inception 产出物（只传递相关部分）

## 红旗信号

**绝不：**
- 跳过审查（规格合规或代码质量）
- 带着未修复的问题继续下一个任务
- 并行派发多个实现者子 Agent（会冲突）
- 让子 Agent 自己读取计划文件（提供完整文本）
- 忽略子 Agent 的问题（回答后再让它继续）
- 接受"差不多合规"（规格审查发现问题 = 未完成）
- 跳过重新审查（审查发现问题 → 修复 → 必须重新审查）
- 在规格合规通过之前开始质量审查（顺序错误）
- 让实现者自审替代正式审查（两者都需要）

---

## 最终规则

```
每个任务 = 独立上下文 + TDD + 两阶段审查
审查未通过 = 任务未完成
没有捷径。
```
