# 需求分析（自适应）

**角色**：产品负责人

**自适应阶段**：始终执行。详细程度根据问题复杂度自适应调整。

**参见 [depth-levels.md](common-depth-levels.md) 了解自适应深度说明**

## 前置条件
- 工作区检测必须完成
- 逆向工程必须完成（如为存量项目）

## 执行步骤

### 步骤 1：加载逆向工程上下文（如可用）

**如果是存量项目**：
- 加载 `docs/aidlc/inception/reverse-engineering/architecture.md`
- 加载 `docs/aidlc/inception/reverse-engineering/component-inventory.md`
- 加载 `docs/aidlc/inception/reverse-engineering/technology-stack.md`
- 加载 `docs/aidlc/inception/reverse-engineering/frontend-architecture.md`（如存在）
- 利用这些信息在分析请求时理解现有系统

### 步骤 2：分析用户请求（意图分析）

#### 2.1 请求清晰度
- **清晰**：具体、明确、可操作
- **模糊**：笼统、含糊、需要澄清
- **不完整**：缺少关键信息

#### 2.2 请求类型
- **新功能**：添加新功能
- **Bug 修复**：修复现有问题
- **重构**：改善代码结构
- **升级**：更新依赖或框架
- **迁移**：迁移到不同技术
- **增强**：改进现有功能
- **新项目**：从零开始

#### 2.3 初始范围估计
- **单文件**：修改一个文件
- **单组件**：修改一个组件/包
- **多组件**：跨多个组件修改
- **系统级**：影响整个系统的修改
- **跨系统**：影响多个系统的修改

#### 2.4 初始复杂度估计
- **微小**：简单、直接的修改
- **简单**：实现路径清晰
- **中等**：有一定复杂度，需多方面考虑
- **复杂**：显著复杂度，需大量考虑

### 步骤 3：确定需求深度

**根据请求分析确定深度：**

**最小深度** — 适用于：
- 请求清晰且简单
- 不需要详细需求
- 仅记录基本理解
- **跳过多角色澄清**，直接生成需求文档

**标准深度** — 适用于：
- 请求需要澄清
- 需要功能和非功能需求
- 正常复杂度
- **执行产品视角 + 架构视角澄清**（2轮）

**全面深度** — 适用于：
- 涉及多方干系人的复杂项目
- 高风险或关键系统
- 需要详细需求和可追溯性
- **执行全部4个角色视角澄清**（4轮）

### 步骤 4：评估当前需求

分析用户提供的所有内容：
   - 意图声明或描述（已记录在 audit.md 中）
   - 现有需求文档（如提及则搜索工作区）
   - 粘贴的内容或文件引用
   - 将任何非 markdown 文档转换为 markdown 格式

### 步骤 5：多角色澄清（自适应轮次）

**核心原则**：从不同角色视角逐轮澄清需求，确保需求完整、无歧义。

**自适应规则**：
- 最小深度 → 跳过本步骤，直接进入步骤6
- 标准深度 → 执行轮次1（产品视角）+ 轮次2（架构视角）
- 全面深度 → 执行全部4个轮次

**每轮执行规则**：
1. 以当前角色视角分析需求，识别该角色关注的缺失/歧义点
2. 生成该角色视角的澄清问题（遵循问题格式指南）
3. 等待用户回答
4. 分析回答，如有后续歧义则追问（同一角色内最多追问1次）
5. 该角色视角澄清完成后，进入下一角色轮次

**问题文件**：所有澄清问题统一写入 `docs/aidlc/inception/requirements/requirement-verification-questions.md`，按角色轮次分节。

---

#### 轮次 1：产品视角

**角色**：产品经理
**关注点**：业务价值、用户场景、优先级、验收标准

**必须评估的领域**：
- **业务目标**：这个需求解决什么业务问题？带来什么价值？
- **目标用户**：谁是主要用户？有哪些用户角色？
- **用户场景**：核心使用场景是什么？用户旅程如何？
- **优先级**：功能优先级如何排序？MVP 包含哪些？
- **验收标准**：如何判断功能"完成"？成功标准是什么？
- **业务约束**：时间限制、预算限制、合规要求？

**前端维度**（如项目包含前端）：
- 需要哪些页面？页面间导航关系？
- 核心交互方式（表单、列表、弹窗、拖拽等）？
- 是否有 Figma 设计稿或设计规范？

**输出**：产品视角澄清问题 + 用户回答

---

#### 轮次 2：架构视角

**角色**：架构师
**关注点**：技术可行性、系统边界、集成点、性能约束

**必须评估的领域**：
- **系统边界**：新功能在系统中的位置？与哪些模块交互？
- **集成点**：需要对接哪些外部系统/API？数据如何流转？
- **技术约束**：是否有技术栈限制？兼容性要求？
- **性能需求**：并发量、响应时间、数据量级？
- **可扩展性**：未来可能的扩展方向？需要预留哪些扩展点？
- **安全需求**：认证授权要求？数据敏感级别？

**前端维度**（如项目包含前端）：
- 状态管理方案？哪些数据需要全局状态？
- 前后端接口契约？数据格式？
- 浏览器兼容性要求？移动端适配？

**输出**：架构视角澄清问题 + 用户回答

---

#### 轮次 3：开发视角

**角色**：开发者
**关注点**：实现细节、数据模型、接口定义、边界条件

**必须评估的领域**：
- **数据模型**：需要哪些实体？字段定义？关系？
- **接口定义**：API 端点设计？请求/响应格式？
- **业务规则**：具体的计算逻辑？状态流转规则？
- **边界条件**：输入限制？数据范围？并发处理？
- **错误处理**：异常场景如何处理？错误提示？
- **数据迁移**：是否需要数据迁移？历史数据处理？

**前端维度**（如项目包含前端）：
- 组件拆分方案？复用哪些现有组件？
- 表单校验规则？字段级别的验证逻辑？
- 列表分页/排序/筛选需求？

**输出**：开发视角澄清问题 + 用户回答

---

#### 轮次 4：测试视角

**角色**：测试人员
**关注点**：测试场景、边界用例、异常流程、验收标准可测试性

**必须评估的领域**：
- **正常流程**：主要的正向测试场景？
- **异常流程**：可能的错误场景？系统如何响应？
- **边界用例**：极端输入？空值？超长数据？
- **并发场景**：多用户同时操作？数据竞争？
- **权限场景**：不同角色的操作权限？越权访问？
- **验收标准可测试性**：每个验收标准是否可以编写自动化测试？

**前端维度**（如项目包含前端）：
- 交互异常场景（网络断开、超时、重复提交）？
- 响应式测试场景（不同屏幕尺寸）？
- 可访问性测试要求？

**输出**：测试视角澄清问题 + 用户回答

---

#### 轮次 5：全局一致性澄清（条件触发）

**角色**：无特定角色（全局视角）
**触发条件**：仅在前4轮澄清中发现以下情况时执行：
- 跨角色矛盾（如产品需求与技术约束冲突）
- 前后回答不一致
- 多角色视角揭示的遗漏（某个需求在多个角色视角中都被提及但定义不同）
- 优先级冲突（业务优先级与技术风险优先级矛盾）

**如未发现上述情况，跳过本轮次。**

**必须评估的领域**：
- **矛盾识别**：列出各角色视角之间的矛盾点，请用户决策
- **一致性确认**：确认关键定义在各角色视角中的理解一致
- **遗漏补充**：多角色交叉分析后发现的新遗漏点
- **优先级仲裁**：当业务需求与技术约束冲突时，请用户明确取舍

**输出**：全局一致性问题 + 用户决策

---

### 步骤 6：生成需求文档
   - 创建 `docs/aidlc/inception/requirements/requirements.md`
   - 顶部包含意图分析摘要：
     - 用户请求
     - 请求类型
     - 范围估计
     - 复杂度估计
   - 包含功能和非功能需求
   - 纳入所有角色轮次的澄清结果
   - 按角色视角标注需求来源（便于追溯）
   - 提供关键需求的简要摘要

### 步骤 7：更新状态跟踪

更新 `docs/aidlc/state.md`：

```markdown
## 阶段进度
### 🔵 INCEPTION 阶段
- [x] 工作区检测
- [x] 逆向工程（如适用）
- [x] 需求分析
```

### 步骤 8：记录并继续
   - 在 `docs/aidlc/audit.md` 中记录确认提示及时间戳
   - 按以下结构展示完成消息：
     1. **完成公告**（强制）：始终以此开头：

```markdown
# 🔍 需求分析完成
```

     2. **AI 摘要**（可选）：提供结构化的需求要点摘要
        - 格式："需求分析已识别 [项目类型/复杂度]："
        - 列出关键功能需求（要点列表）
        - 列出关键非功能需求（要点列表）
        - 如相关，提及架构考虑或技术决策
        - 标注澄清深度（最小/标准/全面）和执行的角色轮次
        - 不要包含工作流指令（"请审查"、"让我知道"、"进入下一阶段"、"在我们继续之前"）
        - 保持事实性和内容聚焦
     3. **格式化工作流消息**（强制）：始终以此格式结尾：

```markdown
> **📋 <u>**需要审查：**</u>**
> 请检查需求文档：`docs/aidlc/inception/requirements/requirements.md`



> **🚀 <u>**下一步？**</u>**
>
> **你可以：**
>
> 🔧 **请求修改** - 根据审查结果要求修改需求
> [如果用户故事将被跳过，添加此选项：]
> 📝 **添加用户故事** - 选择包含**用户故事**步骤（当前因项目简单而跳过）
> ✅ **确认并继续** - 确认需求，进入**[用户故事/工作流规划]**

---
```

**注意**：仅当用户故事步骤将被跳过时才包含"添加用户故事"选项。将 [用户故事/工作流规划] 替换为实际的下一步骤名称。

   - 在用户明确确认前不得继续
   - 记录确认回复及时间戳
   - 在 state.md 中更新需求分析阶段完成状态
