# 复杂度评估与快速通道

工作区检测完成后立即评估；实施或 CR 发现新影响时只允许升级，不得为赶进度降级。

## 复杂度分级

| 级别 | 条件 | 执行路径 |
|------|------|----------|
| 简单 | ≤3 个文件、意图清晰、单组件局部变化，且不触发快速通道排除项 | 快速通道 |
| 中等 | 4-10 个文件、单服务内业务变化、影响域明确、可独立回滚 | 精简流程 |
| 复杂 | >10 个文件，或跨服务/外部系统/新架构组件/高风险/影响未知 | 完整流程 |

## 快速通道排除项

满足任一条件时不得进入或继续快速通道：
- 新数据模型、Schema 或数据迁移；
- 跨进程调用、事件或运行时消费者变化；
- 接口/事件契约语义或兼容性变化；
- 多消费者、动态、路由、鉴权、限流、数据源或外部端点配置变化；
- 跨数据所有权写入、幂等、补偿、对账或一致性边界变化；
- 外部系统责任边界或故障行为变化；
- 新安全机制、明确性能风险或部署拓扑变化；
- 系统基线过期、消费者/Owner 未知或影响域无法闭合。

用户要求“快速”不覆盖这些排除项；应缩小交付范围，而不是跳过系统级分析和验证。

## 快速通道

```text
工作区检测 → 一句话确认 → 测试与实现 → 影响域验证 → 完成
```

- 跳过用户故事、应用设计、单元生成及条件设计文档。
- 保留需求确认、TDD、真实编译/测试、适用技术规范和证据记录。
- 只在 `audit-summary.md` 记录关键节点，不创建空产物。
- 发现排除项后立即升级并回到最早受影响步骤。

## 精简流程

```text
工作区检测 → 需求/故事与审查 → 工作流规划 → [应用/功能设计] → TDD 与审查 → 构建测试
```

保留需求、故事、工作流规划、必要设计、TDD、两阶段审查和实际验证。NFR 与基础设施仅在触发条件满足时执行。

### 不可跳过条件

| 步骤 | 条件 |
|------|------|
| I12 应用设计 | 新接口、跨模块/服务、多端、复杂规则、契约/配置/一致性变化 |
| I13 技术用例 | 存在已批准的契约、配置、迁移、一致性或外部故障风险 |
| I14 单元生成 | 需多个工作单元或跨服务协调 |
| TDD 与两阶段审查 | 始终 |
| 实际构建和影响域测试 | 始终 |

## 完整流程

适用于新系统、跨服务定制、破坏性契约/配置/数据变化及 L4-L5 CR。所有条件步骤按证据执行，不创建不适用产物。

## 评估时机

1. I1 后首次评估并写入 state.md。
2. CR2 基于运行时影响域重评。
3. 实施中发现新消费者、配置、数据或外部依赖时立即升级。
4. 用户可以要求更严格路径；不能要求跳过必需门禁。

## Construction 条件阈值

| 步骤 | 执行阈值 |
|------|----------|
| 功能设计 | 新数据模型、业务规则 ≥3 条或状态机变化 |
| NFR 需求 | 明确性能指标、新安全机制或可用性/恢复目标 |
| NFR 设计 | NFR 已识别特殊模式 |
| 基础设施设计 | 新基础设施组件或部署拓扑变化 |

完整流程仍按对应 steering 的实际条件判断。
