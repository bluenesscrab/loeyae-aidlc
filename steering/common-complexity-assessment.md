# 复杂度评估与快速通道

**在工作区检测完成后，立即评估任务复杂度，决定执行路径。**

---

## 复杂度分级

| 级别 | 条件（全部满足） | 执行路径 |
|------|-----------------|----------|
| **简单** | ≤3 个文件变更 + 无新数据模型 + 无跨服务调用 + 用户意图完全清晰 + 无安全/性能风险 | 快速通道 |
| **中等** | 4-10 个文件 + 有业务逻辑但范围明确 + 单服务内变更 | 精简流程 |
| **复杂** | >10 个文件 或 跨服务 或 新架构组件 或 高风险 | 完整流程 |

---

## 快速通道（简单任务）

**适用场景**：加字段、修 bug、加简单接口、调整配置等。

**流程**：
```
工作区检测 → 确认理解（1句话） → 写测试 + 实现 → 验证通过 → 完成
```

**规则**：
- 跳过：用户故事、应用设计、单元生成、功能设计、NFR 全套
- 保留：需求确认（口头，1 句话）、TDD（可用快速模式）、编译验证、MCP Skill 加载（如适用）
- 审计：仅记录到 audit-summary.md 一行，不创建分段文件
- 产出物：不生成 Inception 阶段文档，直接进入代码
- **退出条件**：如果实现过程中发现复杂度被低估（如需要新数据模型、涉及多服务），立即退出快速通道，回到标准流程

---

## 精简流程（中等任务）

**适用场景**：新增业务功能、修改现有流程、中等规模重构。

**流程**：
```
工作区检测 → 需求分析（标准深度） → 工作流规划 → [功能设计] → 代码生成 → 构建测试
```

**规则**：
- 跳过：用户故事（除非多角色）、NFR 需求/设计（除非有明确性能要求）、基础设施设计（除非有部署变更）
- 保留：需求分析、工作流规划、功能设计（如有新业务逻辑）、代码生成全流程
- 审计：标准分段格式
- 审批：按审批模式执行

---

## 完整流程（复杂任务）

**适用场景**：新系统、跨服务重构、高风险变更。

**流程**：与当前流程完全一致，所有条件步骤按需执行。

---

## 复杂度评估时机

1. **工作区检测完成后**：首次评估，记录到 state.md
2. **实现过程中**：如果发现复杂度被低估，可以升级（但不能降级）
3. **用户可覆盖**：用户可以明确指定"用快速通道"或"走完整流程"

---

## Construction 条件步骤的复杂度阈值

**快速通道任务**：跳过所有条件设计步骤，直接进入代码生成。

**精简流程任务**：按以下阈值判断是否执行条件步骤：

| 步骤 | 执行阈值 | 说明 |
|------|----------|------|
| 功能设计 | 有新数据模型 或 业务规则 ≥3 条 | 简单 CRUD 不需要 |
| NFR 需求 | 有**明确的**性能指标（如 QPS、响应时间）或**新增**安全机制 | "需要安全考虑"不算，所有代码都需要安全考虑 |
| NFR 设计 | NFR 需求步骤已执行且识别出需要特殊模式 | 跟随 NFR 需求 |
| 基础设施设计 | 有新的基础设施组件（缓存、MQ、新数据库）或部署架构变更 | 使用现有基础设施不算 |

**完整流程任务**：按原有条件判断执行。
