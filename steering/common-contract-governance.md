# 契约治理

**职责**：管理跨边界接口、事件和数据交换契约的事实来源、兼容性、消费者状态与验证证据。模块或服务如何划分由其他步骤决定。

## 执行条件

存在任一跨进程调用、异步事件、外部系统交换或前后端接口时执行。纯进程内调用且不形成共享边界时跳过；其方法签名可留在应用设计产物中。

## 契约边界

| 边界 | 权威事实 | AI-DLC 索引职责 |
|------|----------|-----------------|
| 进程内模块 | 项目代码与应用设计 | 可在设计中记录完整签名 |
| 跨进程同步接口 | 项目采用的机器可读接口规范或代码生成源 | 记录位置、提供方、消费者、版本和状态 |
| 异步事件 | 项目采用的事件 Schema/消息规范 | 记录主题/事件标识、发布方、订阅方和兼容策略 |
| 外部系统交换 | 对方规范、适配层契约或经确认的交换定义 | 记录责任边界、版本、映射和验证方式 |

存在机器可读事实来源时，`contracts.md` 不复制完整字段定义，只保存索引、决策、兼容性结论和消费者状态。不存在机器来源时，必须明确标记“文档契约”，列出补齐或确认方式，不得伪装为已自动校验。

## 最小契约记录

每项契约至少包含：
- 稳定契约 ID、类型和权威来源路径；
- 提供方、已知消费者及 Owner；
- 版本或演进策略；
- 鉴权、错误语义和幂等要求（适用时）；
- 当前兼容性结论及证据；
- 每个消费者的 `待评估 / 待适配 / 已适配 / 已验证 / 不适用` 状态。

共享包默认只承载无业务语义的技术能力。业务实体、数据库模型和领域对象不得因“复用方便”成为跨服务共享契约；跨边界使用显式 DTO、事件或 Schema。

## 变更分类

| 类别 | 判定 | 最低处理 |
|------|------|----------|
| 兼容新增 | 现有消费者无需修改即可继续工作 | 更新契约与新增行为验证 |
| 条件兼容 | 是否兼容取决于消费者或环境 | 逐消费者确认并验证 |
| 破坏性 | 删除、重命名、收紧约束、改变语义或顺序要求 | CR 高风险处理，制定迁移与回滚方案 |
| 未知 | 缺少消费者、版本或证据 | 阻断完成声明，先补事实 |

“可选字段”“新增事件字段”等不能自动判定兼容，必须依据实际序列化、校验与消费者行为。

## 执行流程

1. 定位契约权威来源及其提供方。
2. 从运行时依赖基线获取消费者反向索引。
3. 比较变更前后结构、语义、约束和错误行为。
4. 为每个消费者记录影响、适配动作与验证证据。
5. 将迁移顺序、双版本窗口或回滚条件写入 CR/执行计划。
6. 更新 `contracts.md` 索引和变更日志，不覆盖机器事实来源。

## 验证证据

可接受证据包括项目已有兼容性检查、消费者驱动契约测试、集成测试、Schema 校验、CI 运行标识或可复现命令输出。只有文档审阅而无执行证据时，状态最多为“已评估”，不得标记“已验证”。

## 完成标准

- [ ] 每项跨边界契约有明确权威来源或“文档契约”标记
- [ ] 提供方与所有已知消费者已建立反向索引
- [ ] 兼容性按结构和语义评估，不以版本号代替
- [ ] 每个受影响消费者有独立状态和证据
- [ ] 破坏性变化有迁移、部署顺序与回滚条件
- [ ] `state.md` 记录本次受影响契约、阻断项和证据引用
