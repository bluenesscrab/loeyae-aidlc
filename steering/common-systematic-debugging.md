# 系统化调试

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖底层问题。

**核心原则**：在尝试修复之前，必须找到根因。症状修复是失败。

**违反这个流程的字面意思就是违反调试的精神。**

---

## 铁律

```
没有根因调查，就没有修复。
```

如果你没有完成第一阶段，你不能提出修复方案。

---

## 适用时机

用于**任何**技术问题：
- 测试失败
- 生产 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用**：
- 时间压力下（紧急情况让猜测变得诱人）
- "一个快速修复"看起来很明显
- 你已经尝试了多次修复
- 之前的修复没有生效
- 你不完全理解问题

**不要跳过，即使**：
- 问题看起来很简单（简单 bug 也有根因）
- 你很赶时间（系统化比乱试更快）
- 用户想要立即修复（系统化方法比反复折腾更快）

---

## 四阶段流程

你**必须**完成每个阶段才能进入下一个。

### 第一阶段：根因调查

**在尝试任何修复之前：**

#### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 它们通常包含确切的解决方案
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误码

#### 2. 稳定重现

- 能可靠地触发吗？
- 确切的步骤是什么？
- 每次都发生吗？
- 如果不能重现 → 收集更多数据，不要猜测

#### 3. 检查最近变更

- 什么变更可能导致了这个问题？
- Git diff，最近的提交
- 新依赖，配置变更
- 环境差异

#### 4. 多组件系统中收集证据

**当系统有多个组件时**（API → Service → Database，前端 → 后端 → 数据库）：

**在提出修复之前，添加诊断检测：**

```
对每个组件边界：
  - 记录进入组件的数据
  - 记录离开组件的数据
  - 验证环境/配置传播
  - 检查每层的状态

运行一次收集证据，显示在哪里断裂
然后分析证据，识别失败的组件
然后调查那个特定组件
```

**示例（多层系统）：**
```java
// 层 1：Controller
log.info("=== Controller 收到请求: {}", request);

// 层 2：Service
log.info("=== Service 处理参数: {}", params);
log.info("=== Service 返回结果: {}", result);

// 层 3：Repository
log.info("=== SQL 执行: {}", sql);
log.info("=== 查询结果行数: {}", rows.size());
```

**这揭示了**：哪一层失败（Controller → Service ✓，Service → Repository ✗）

#### 5. 追踪数据流

**当错误在调用栈深处时：**

- 坏值从哪里来？
- 什么调用了这个方法并传入了坏值？
- 继续向上追踪直到找到源头
- 在源头修复，不是在症状处

---

### 第二阶段：模式分析

**在修复之前找到模式：**

#### 1. 找到工作的示例
- 在同一代码库中找到类似的工作代码
- 什么类似的东西是正常工作的？

#### 2. 对比参考
- 如果在实现某个模式，完整阅读参考实现
- 不要略读——逐行阅读
- 在应用之前完全理解模式

#### 3. 识别差异
- 工作的和坏的之间有什么不同？
- 列出每个差异，无论多小
- 不要假设"那个不可能有影响"

#### 4. 理解依赖
- 这需要什么其他组件？
- 什么设置、配置、环境？
- 它做了什么假设？

---

### 第三阶段：假设与测试

**科学方法：**

#### 1. 形成单一假设
- 清晰陈述："我认为 X 是根因，因为 Y"
- 写下来
- 具体，不模糊

#### 2. 最小测试
- 做最小的可能变更来测试假设
- 一次一个变量
- 不要同时修复多个东西

#### 3. 验证后再继续
- 有效？→ 第四阶段
- 无效？→ 形成新假设
- **不要**在上面叠加更多修复

#### 4. 当你不知道时
- 说"我不理解 X"
- 不要假装知道
- 寻求帮助
- 做更多研究

---

### 第四阶段：实施

**修复根因，不是症状：**

#### 1. 创建失败测试用例
- 最简单的重现
- 自动化测试（如果可能）
- 必须在修复之前有
- 使用 TDD 流程（参见 `construction-tdd.md`）

#### 2. 实施单一修复
- 解决已识别的根因
- 一次一个变更
- 没有"顺便"的改进
- 没有捆绑的重构

#### 3. 验证修复
- 测试现在通过了？
- 没有其他测试被破坏？
- 问题真的解决了？

#### 4. 如果修复无效
- **停止**
- 计数：你已经尝试了多少次修复？
- 如果 < 3：回到第一阶段，用新信息重新分析
- **如果 ≥ 3：停止并质疑架构（第 5 步）**
- **不要**在没有架构讨论的情况下尝试第 4 次修复

#### 5. 如果 3+ 次修复失败：质疑架构

**表明架构问题的模式：**
- 每次修复揭示新的共享状态/耦合/不同位置的问题
- 修复需要"大规模重构"才能实施
- 每次修复在其他地方产生新症状

**停止并质疑基本面：**
- 这个模式从根本上是合理的吗？
- 我们是否"因为惯性而坚持"？
- 应该重构架构还是继续修补症状？

**在尝试更多修复之前与用户讨论。**

这不是失败的假设——这是错误的架构。

---

## 红旗信号 — 停止并遵循流程

如果你发现自己在想：
- "先快速修一下，以后再调查"
- "试试改 X 看看行不行"
- "加多个改动，跑测试"
- "跳过测试，我手动验证"
- "大概是 X，让我修一下"
- "我不完全理解但这可能有效"
- "模式说 X 但我要不同地适配"
- "主要问题是：[列出修复方案但没有调查]"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（已经尝试了 2+ 次）**
- **每次修复揭示不同位置的新问题**

**所有这些都意味着：停止。回到第一阶段。**

**如果 3+ 次修复失败：** 质疑架构（见第四阶段第 5 步）

---

## 常见合理化与反驳

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急情况，没时间走流程" | 系统化调试比猜测-检查-折腾更快。 |
| "先试这个，然后再调查" | 第一次修复设定了模式。从一开始就做对。 |
| "修复后再写测试" | 未测试的修复不持久。先测试证明它。 |
| "同时修多个节省时间" | 无法隔离什么有效。会引入新 bug。 |
| "参考太长，我适配模式" | 部分理解保证 bug。完整阅读。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修。 |

## 与 AI-DLC 工作流的集成

### 触发条件

系统化调试在以下情况自动触发：

1. **TDD 循环中**：测试以意外方式失败
2. **构建失败**：编译或构建错误
3. **集成问题**：单元间交互出错
4. **审查发现的 bug**：代码审查中发现的逻辑错误

### 调试记录

在审计日志中记录调试过程：

```markdown
## 系统化调试
**时间戳**: [ISO 时间戳]
**触发**: [什么问题触发了调试]
**第一阶段 — 根因调查**:
  - 错误信息: [完整错误]
  - 重现步骤: [步骤]
  - 最近变更: [相关变更]
  - 数据流追踪: [追踪结果]
**第二阶段 — 模式分析**:
  - 工作示例: [参考]
  - 差异: [发现的差异]
**第三阶段 — 假设**:
  - 假设: "[具体假设]"
  - 测试结果: [验证/否定]
**第四阶段 — 实施**:
  - 失败测试: [测试名]
  - 修复: [修复描述]
  - 验证: [测试通过确认]
```

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| 1. 根因 | 读错误、重现、检查变更、收集证据 | 理解什么和为什么 |
| 2. 模式 | 找工作示例、对比 | 识别差异 |
| 3. 假设 | 形成理论、最小测试 | 确认或新假设 |
| 4. 实施 | 创建测试、修复、验证 | Bug 解决，测试通过 |

---

## 最终规则

```
猜测不是调试。
系统化方法：15-30 分钟修复。
随机修复方法：2-3 小时折腾。
首次修复成功率：95% vs 40%。
引入新 bug：接近零 vs 常见。

选择很明显。
```
